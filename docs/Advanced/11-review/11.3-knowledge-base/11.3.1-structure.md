# 11.3.1 文檔結構：項目文檔的組織方式

## 一句話破題

好的文檔結構讓人**一眼就知道去哪裏找想要的信息**，降低認知負擔。

## 核心價值

合理的文檔結構能讓你：
- 快速定位所需信息
- 新人更容易上手
- 文檔更新更有條理

## 推薦的目錄結構

```
docs/
├── README.md           # 文檔導航入口
├── getting-started/    # 快速開始
│   ├── installation.md
│   ├── configuration.md
│   └── first-project.md
├── guides/             # 使用指南
│   ├── authentication.md
│   ├── database.md
│   └── deployment.md
├── api/                # API 參考
│   ├── overview.md
│   ├── users.md
│   └── products.md
├── architecture/       # 架構設計
│   ├── overview.md
│   ├── data-flow.md
│   └── decisions/      # 架構決策記錄
├── contributing/       # 貢獻指南
│   ├── code-style.md
│   └── pull-requests.md
└── changelog/          # 變更記錄
    └── CHANGELOG.md
```

## 文檔類型說明

| 類型 | 目的 | 受衆 |
|------|------|------|
| **快速開始** | 讓新人5分鐘跑起來 | 新用戶 |
| **使用指南** | 解釋如何完成特定任務 | 開發者 |
| **API 參考** | 詳細的接口說明 | 開發者 |
| **架構設計** | 解釋系統如何工作 | 架構師、維護者 |
| **貢獻指南** | 如何參與項目 | 貢獻者 |

## 架構決策記錄

使用 ADR 記錄重要的技術決策：

```markdown
<!-- docs/architecture/decisions/001-use-prisma.md -->
# ADR 001: 使用 Prisma 作爲 ORM

## 狀態
已採納

## 背景
項目需要一個類型安全的數據庫訪問層

## 決策
採用 Prisma 作爲 ORM

## 理由
- 類型安全，與 TypeScript 集成好
- 自動生成遷移腳本
- 社區活躍，文檔完善

## 後果
- 需要學習 Prisma 特定語法
- 複雜查詢可能需要原生 SQL
```

## 避坑指南

::: danger 新手最容易犯的錯
1. 所有文檔堆在一個文件裏
2. 沒有導航入口，找不到文檔在哪
3. 文檔命名不規範（如 `doc1.md`、`新建文檔.md`）
4. 文檔之間沒有相互鏈接
:::
