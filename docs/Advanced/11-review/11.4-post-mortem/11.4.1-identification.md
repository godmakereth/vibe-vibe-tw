# 11.4.1 問題識別：故障現象與影響範圍

## 一句話破題

覆盤的第一步是**準確描述發生了什麼**——不是"系統掛了"，而是"用戶無法登錄，持續30分鐘，影響1000用戶"。

## 核心價值

準確識別問題能讓你：
- 聚焦真正的問題，而不是表面現象
- 量化影響，確定優先級
- 爲後續分析提供清晰的起點

## 問題描述的 5W1H

| 要素 | 問題 | 示例 |
|------|------|------|
| **What** | 發生了什麼？ | 用戶無法登錄 |
| **When** | 什麼時候發生的？ | 2024-01-15 10:00-10:30 |
| **Where** | 在哪裏發生的？ | 生產環境、登錄頁面 |
| **Who** | 誰受到了影響？ | 所有嘗試登錄的用戶 |
| **How** | 如何表現的？ | 登錄按鈕點擊後一直loading |
| **How much** | 影響多大？ | 約1000次登錄嘗試失敗 |

## 影響範圍評估

### 用戶影響

```typescript
// 從日誌或監控中提取影響數據
const impactMetrics = {
  affectedUsers: 1000,           // 受影響用戶數
  failedRequests: 5000,          // 失敗請求數
  errorRate: 0.15,               // 錯誤率 15%
  duration: '30 minutes',        // 持續時間
  revenueImpact: '$10,000',      // 收入影響（如適用）
}
```

### 嚴重程度分級

| 級別 | 定義 | 響應時間 |
|------|------|----------|
| **P0** | 核心功能完全不可用 | 立即響應 |
| **P1** | 核心功能部分不可用 | 1小時內 |
| **P2** | 非核心功能受影響 | 4小時內 |
| **P3** | 體驗問題，不影響功能 | 1天內 |

## 問題識別檢查清單

```markdown
□ 問題描述是否清晰、具體？
□ 是否有準確的時間範圍？
□ 是否量化了影響範圍？
□ 是否確定了嚴重程度？
□ 是否記錄了完整的時間線？
□ 是否保留了相關日誌和截圖？
```

## 常見錯誤

| 錯誤描述 | 正確描述 |
|----------|----------|
| "系統掛了" | "登錄服務返回 500 錯誤" |
| "很多用戶受影響" | "約 1000 名用戶無法登錄" |
| "持續了一段時間" | "持續 30 分鐘（10:00-10:30）" |

## 避坑指南

::: danger 新手最容易犯的錯
1. 描述太模糊，無法定位問題
2. 只關注技術現象，忽略業務影響
3. 沒有保留現場日誌和截圖
4. 時間線記錄不完整
:::
