# 8.5 開工前對齊了嗎——工作流程

再好的代碼也救不了"做錯了方向"——對齊、評審、驗收是確保產出價值的關鍵流程。

## 爲什麼需要流程

| 問題 | 後果 | 解決方案 |
|------|------|----------|
| 需求理解偏差 | 做完發現不是要的 | 需求對齊 |
| 技術方案有坑 | 開發到一半發現走不通 | 技術評審 |
| 完成標準模糊 | 做完不知道算不算完 | 驗收標準 |
| 上線無規範 | 上線後出問題不知道怎麼回滾 | 部署流程 |

## 完整開發流程

```mermaid
flowchart LR
    A[需求對齊] --> B[技術評審]
    B --> C[開發實現]
    C --> D[代碼審查]
    D --> E[功能驗收]
    E --> F[部署上線]
    F --> G[監控反饋]
```

## 本節結構

1. **需求對齊**：PRD 評審與確認，確保做對的事
2. **技術評審**：方案設計與風險評估，確保技術可行
3. **驗收標準**：功能測試與性能指標，確保質量達標
4. **部署流程**：1Panel 部署與上線規範

## 流程簡化原則

對於小團隊或個人項目，可以簡化流程：

| 完整流程 | 簡化版本 |
|----------|----------|
| PRD 文檔 | 簡單 Feature List |
| 技術評審會 | 自己寫個 ADR |
| 測試團隊驗收 | 自動化測試 + 自測 |
| 運維部署 | CI/CD 自動化 |

**核心原則**：
- 有記錄 > 無記錄
- 自動化 > 手動操作
- 儘早發現問題 > 上線後修復

## AI 協作指南

AI 可以輔助流程中的多個環節：

- **需求對齊**：幫助拆解需求、生成用戶故事
- **技術評審**：分析技術方案的優劣
- **驗收標準**：生成測試用例、檢查邊界條件
- **部署流程**：生成部署腳本、檢查配置
