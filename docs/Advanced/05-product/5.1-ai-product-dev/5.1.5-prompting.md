# 5.1.5 怎麼和 AI 好好說話——Prompt 工程核心原則

### 一句話破題

好的 Prompt 三要素：**清晰**（AI 能理解）、**具體**（減少歧義）、**結構化**（便於執行）。

### 核心原則

#### 原則一：清晰表達意圖

```
❌ 模糊："幫我優化這個代碼"
（優化什麼？性能？可讀性？安全性？）

✅ 清晰："這個函數運行太慢了，請優化性能，
目前處理 1000 條數據需要 5 秒，希望降到 1 秒以內"
```

#### 原則二：提供充足上下文

```
❌ 缺乏上下文："寫一個用戶表"

✅ 上下文完整：
"用 Prisma 寫一個用戶表：
- 使用 PostgreSQL 數據庫
- 需要支持郵箱登錄
- 用戶有基本信息（暱稱、頭像）
- 需要記錄創建和更新時間"
```

#### 原則三：明確約束條件

```
❌ 無約束："做一個登錄頁面"

✅ 有約束：
"做一個登錄頁面：
- 使用 Next.js App Router
- 樣式用 Tailwind CSS
- 需要表單驗證（郵箱格式、密碼長度 8-20）
- 調用 /api/auth/login 接口"
```

### Prompt 結構模板

一個完整的開發類 Prompt 通常包含：

```markdown
## 任務
[一句話說明要做什麼]

## 背景
[相關的上下文信息]

## 技術約束
- [使用什麼技術棧]
- [需要遵循什麼規範]

## 輸入/輸出
- 輸入：[接收什麼數據]
- 輸出：[返回什麼結果]

## 示例
[如果有的話，提供參考示例]

## 注意事項
[容易出錯的地方]
```

### 實戰示例

**需求**：創建一個文章管理的 API

```markdown
## 任務
創建文章 CRUD 的 API 路由

## 背景
這是一個博客系統的後端接口，前端使用 React，
需要支持文章的創建、讀取、更新和刪除

## 技術約束
- Next.js 16 App Router
- 使用 Prisma ORM
- 數據庫是 PostgreSQL
- 返回 JSON 格式

## 接口定義
- GET /api/posts - 獲取文章列表（支持分頁）
- GET /api/posts/[id] - 獲取單篇文章
- POST /api/posts - 創建文章
- PUT /api/posts/[id] - 更新文章
- DELETE /api/posts/[id] - 刪除文章

## 數據結構
文章包含：id, title, content, authorId, 
createdAt, updatedAt

## 注意事項
- 創建和更新需要驗證必填字段
- 刪除要返回 204 狀態碼
- 列表接口默認每頁 10 條
```

### 常見誤區

**誤區 1：一次說太多**

```
❌ "幫我做一個完整的博客系統，包括文章管理、用戶系統、
評論功能、搜索、RSS..."

✅ 分步驟：先做文章 CRUD，驗證通過後再加其他功能
```

**誤區 2：期望 AI 讀心**

```
❌ "按照通常的做法來就行"
（AI 不知道你認爲的"通常"是什麼）

✅ 明確指定：使用 RESTful 風格，遵循項目現有的代碼規範
```

**誤區 3：不給反饋**

```
❌ 收到代碼後直接使用，發現問題才說"你給的不對"

✅ 收到代碼後檢查，具體說明哪裏不符合預期：
"創建文章時，如果標題爲空應該返回 400 錯誤，
但現在返回的是 500"
```

### 提升 Prompt 質量的技巧

1. **先寫後審**：寫完 Prompt 先讀一遍，看自己是否能理解
2. **假設 AI 是新人**：把 AI 當作剛入職的同事，需要詳細說明背景
3. **提供示例**：能給代碼示例就給，AI 從示例中學習最快
4. **迭代優化**：第一次不完美很正常，根據結果調整 Prompt
