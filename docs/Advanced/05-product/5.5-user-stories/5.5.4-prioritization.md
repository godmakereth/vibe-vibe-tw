# 5.5.4 到底先做哪個——優先級矩陣

### 一句話破題

用**重要性**和**緊急性**兩個維度，科學地決定功能的開發順序。

### 艾森豪威爾矩陣

```
              緊急
               ↑
       ┌───────┼───────┐
       │  P1   │  P0   │
       │ 儘快做 │ 立即做 │
       ├───────┼───────┤
       │  P3   │  P2   │
       │ 以後說 │ 計劃做 │
       └───────┼───────┘
    不重要 ←───┼───→ 重要
               │
             不緊急
```

| 象限 | 特徵 | 行動 |
|------|------|------|
| P0 | 重要且緊急 | 立即做，投入核心資源 |
| P1 | 緊急不重要 | 儘快做，可以簡化實現 |
| P2 | 重要不緊急 | 計劃做，安排到迭代中 |
| P3 | 不重要不緊急 | 以後說，或者不做 |

### 如何判斷重要性

問自己：**對用戶核心價值有多大影響？**

```
高重要性：
- 沒有這個功能，產品無法使用
- 影響核心用戶體驗
- 是競爭差異化的關鍵

低重要性：
- 有也行，沒有也不影響使用
- 只有少數用戶會用
- 錦上添花的功能
```

### 如何判斷緊急性

問自己：**時間敏感度有多高？**

```
高緊急性：
- 有明確的截止日期
- 不做會阻塞其他工作
- 用戶在等這個功能

低緊急性：
- 沒有時間要求
- 可以隨時開始做
- 用戶暫時不需要
```

### 實際案例：博客系統功能排序

```markdown
## 功能優先級排序

### P0 - 立即做（MVP 必須）
| 功能 | 重要性 | 緊急性 | 理由 |
|------|--------|--------|------|
| 文章 CRUD | 高 | 高 | 核心功能 |
| 文章列表 | 高 | 高 | 核心功能 |
| 文章詳情 | 高 | 高 | 核心功能 |

### P1 - 儘快做（MVP 後第一優先）
| 功能 | 重要性 | 緊急性 | 理由 |
|------|--------|--------|------|
| 分類標籤 | 中 | 中 | 內容組織 |
| Markdown 編輯器 | 中 | 高 | 用戶期待 |

### P2 - 計劃做（下個版本）
| 功能 | 重要性 | 緊急性 | 理由 |
|------|--------|--------|------|
| 搜索 | 高 | 低 | 文章多了才需要 |
| SEO | 高 | 低 | 可後續優化 |

### P3 - 以後說（暫不規劃）
| 功能 | 重要性 | 緊急性 | 理由 |
|------|--------|--------|------|
| 評論 | 低 | 低 | 可用第三方 |
| 多語言 | 低 | 低 | 初期不需要 |
```

### MoSCoW 方法

另一種常用的優先級劃分方法：

| 級別 | 含義 | 說明 |
|------|------|------|
| **M**ust have | 必須有 | 沒有就無法交付 |
| **S**hould have | 應該有 | 重要但非關鍵 |
| **C**ould have | 可以有 | 有時間就做 |
| **W**on't have | 不做 | 本次不考慮 |

### RICE 評分法

更量化的優先級評估方法：

```
RICE 分數 = (Reach × Impact × Confidence) / Effort

- Reach：影響多少用戶
- Impact：對每個用戶的影響程度（0.25/0.5/1/2/3）
- Confidence：對評估的信心程度（100%/80%/50%）
- Effort：開發所需人天
```

**示例**：

| 功能 | Reach | Impact | Confidence | Effort | RICE |
|------|-------|--------|------------|--------|------|
| 搜索 | 100 | 2 | 80% | 3 | 53 |
| 評論 | 50 | 1 | 50% | 5 | 5 |
| SEO | 1000 | 1 | 80% | 2 | 400 |

### 讓 AI 幫你排優先級

```
你：以下功能請幫我排優先級：
- 文章 CRUD
- 評論系統
- 搜索功能
- RSS 訂閱
- 多語言支持

排序依據：博客系統 MVP，獨立開發，兩週時間

AI：## 優先級排序

### P0（必須做）
1. **文章 CRUD** - 核心功能，沒有就不是博客

### P1（應該做）
2. **搜索功能** - 文章多了必須有，但 MVP 可簡化

### P2（可以延後）
3. **評論系統** - 可用 Giscus 替代
4. **RSS 訂閱** - 增強功能

### P3（暫不考慮）
5. **多語言支持** - 初期不需要
```

### 優先級調整原則

優先級不是固定的，需要動態調整：

1. **依賴關係**：被依賴的功能優先級提高
2. **用戶反饋**：用戶強烈需求的優先級提高
3. **資源變化**：資源充足時可提前做 P2
4. **風險變化**：高風險功能可能需要提前驗證

### 實用建議

1. **先粗後細**：先分 P0/P1/P2/P3，再細排順序
2. **團隊對齊**：確保大家對優先級有共識
3. **可視化**：用看板工具展示優先級
4. **定期回顧**：每個迭代開始時重新評估
