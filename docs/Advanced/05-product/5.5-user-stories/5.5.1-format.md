# 5.5.1 像用戶一樣描述需求——用戶故事格式

### 一句話破題

用戶故事的標準格式：**作爲 [角色]，我希望 [做什麼]，以便 [獲得什麼價值]**。

### 用戶故事的三要素

```
作爲 [Who] - 誰會使用這個功能
我希望 [What] - 想要做什麼
以便 [Why] - 爲什麼想要這樣做
```

### 好的用戶故事 vs 差的用戶故事

| 差的描述 | 好的用戶故事 |
|----------|--------------|
| "做登錄功能" | 作爲用戶，我希望用郵箱密碼登錄，以便訪問我的個人數據 |
| "文章列表" | 作爲讀者，我希望瀏覽文章列表，以便發現感興趣的內容 |
| "添加搜索" | 作爲讀者，我希望搜索文章，以便快速找到特定主題 |

### 爲什麼要寫"以便"

"以便"部分是最容易被忽略，卻最重要的：

```
❌ 作爲用戶，我希望能登錄

✅ 作爲用戶，我希望能登錄，以便：
   - 保存我的閱讀記錄
   - 發表評論
   - 收藏文章
```

知道了"爲什麼"，才能更好地設計"怎麼做"。

### 不同角色的用戶故事

一個產品可能有多種用戶角色：

```markdown
## 博客系統用戶故事

### 作爲訪客
- 我希望瀏覽文章列表，以便發現感興趣的內容
- 我希望閱讀文章詳情，以便學習新知識
- 我希望搜索文章，以便快速定位內容

### 作爲註冊用戶
- 我希望登錄賬號，以便使用個人功能
- 我希望評論文章，以便與作者互動
- 我希望收藏文章，以便日後重讀

### 作爲博主
- 我希望撰寫文章，以便分享我的知識
- 我希望管理文章，以便維護內容質量
- 我希望查看統計，以便了解讀者偏好
```

### 用戶故事的驗收標準

每個用戶故事都應該有驗收標準：

```markdown
## 用戶故事：文章發佈

作爲博主，我希望發佈文章，以便分享我的技術經驗

### 驗收標準
- [ ] 可以輸入標題（1-100 字符）
- [ ] 可以輸入正文（支持 Markdown）
- [ ] 可以選擇分類（至少選一個）
- [ ] 可以添加標籤（最多 5 個）
- [ ] 發佈成功後跳轉到文章詳情頁
- [ ] 發佈成功後在列表頁可見

### 邊界情況
- 標題爲空：顯示錯誤提示
- 正文爲空：顯示錯誤提示
- 網絡失敗：保存爲草稿
```

### INVEST 原則

好的用戶故事應該符合 INVEST 原則：

| 原則 | 含義 | 示例 |
|------|------|------|
| **I**ndependent | 獨立，不依賴其他故事 | 登錄和註冊可以分開做 |
| **N**egotiable | 可協商，細節可討論 | 密碼規則可以再定 |
| **V**aluable | 有價值，對用戶有意義 | 不做"爲了技術而技術"的功能 |
| **E**stimable | 可估算，能評估工作量 | 大故事要拆小 |
| **S**mall | 小，幾天內能完成 | 太大就拆分 |
| **T**estable | 可測試，有驗收標準 | 知道什麼算"完成" |

### 用戶故事拆分

太大的故事需要拆分：

```
❌ 作爲用戶，我希望完整的認證系統

✅ 拆分爲：
1. 作爲用戶，我希望註冊賬號
2. 作爲用戶，我希望登錄賬號
3. 作爲用戶，我希望重置密碼
4. 作爲用戶，我希望修改個人信息
```

### 讓 AI 幫你寫用戶故事

```
你：我要做一個博客系統，幫我寫用戶故事

AI：## 博客系統用戶故事

### 內容管理
1. 作爲博主，我希望創建文章，以便分享我的知識
2. 作爲博主，我希望編輯文章，以便修正錯誤
3. 作爲博主，我希望刪除文章，以便移除過時內容
4. 作爲博主，我希望保存草稿，以便稍後繼續編輯

### 內容消費
1. 作爲讀者，我希望瀏覽文章列表，以便發現內容
2. 作爲讀者，我希望閱讀文章詳情，以便學習知識
3. 作爲讀者，我希望搜索文章，以便快速定位
...
```

### 實用建議

1. **每個功能都寫用戶故事**：養成習慣
2. **不要跳過"以便"**：這是理解需求的關鍵
3. **用 AI 生成初稿**：然後你來修改完善
4. **保持簡短**：一個故事描述一個功能點
