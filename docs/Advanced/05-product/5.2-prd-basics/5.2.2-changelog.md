# 5.2.2 到底改了啥——更新記錄格式

### 一句話破題

更新記錄讓你知道：**這份文檔和上次相比改了什麼**。

### 爲什麼需要更新記錄

- **回溯變更**：出問題時能追溯是哪次修改導致的
- **理解演進**：新成員能快速瞭解需求的演變過程
- **對比版本**：知道從哪個版本開始有了某個功能

### 更新記錄的標準格式

```markdown
## 更新記錄

| 版本 | 日期 | 變更內容 | 作者 |
|------|------|----------|------|
| v1.2 | 2024-01-20 | 新增評論功能 | 李四 |
| v1.1 | 2024-01-15 | 優化搜索邏輯，支持模糊匹配 | 張三 |
| v1.0 | 2024-01-10 | 初版發佈 | 張三 |
```

### 版本號規範

採用語義化版本（Semantic Versioning）的簡化版：

```
v主版本.次版本

主版本：重大改動、不兼容的變更
次版本：新增功能、向後兼容的變更
```

**示例**：
- `v1.0` → `v1.1`：新增了一個字段
- `v1.1` → `v2.0`：整個數據結構重構

### 變更內容的寫法

**好的寫法**：清晰、具體、可追溯

```
✅ 新增用戶頭像上傳功能，支持 jpg/png 格式
✅ 修改登錄接口，增加驗證碼驗證
✅ 刪除廢棄的 getOldUser API
```

**差的寫法**：模糊、籠統

```
❌ 修改了一些問題
❌ 優化
❌ 更新
```

### 與 Git 的關係

文檔的更新記錄和 Git 提交記錄可以配合使用：

```markdown
## 更新記錄

| 版本 | 日期 | 變更內容 | Commit |
|------|------|----------|--------|
| v1.2 | 2024-01-20 | 新增評論功能 | [abc123](./commit-link) |
```

但文檔更新記錄應該是**業務視角**的總結，而不是每個 Git commit 的複製。

### 簡化方案

對於個人項目，可以簡化爲：

```markdown
## 變更歷史

- 2024-01-20: 增加評論功能
- 2024-01-15: 優化搜索
- 2024-01-10: 初版
```

關鍵是**有記錄**，格式可以根據項目規模調整。

### 對 AI 協作的影響

當文檔有多個版本時，向 AI 說明版本信息：

```
請基於 v1.2 版本的 PRD 生成代碼。
注意：v1.2 相比 v1.1 新增了評論功能。

[文檔內容]
```
