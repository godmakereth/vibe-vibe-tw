# 5.2.4 別讓黑話搞暈團隊——術語表維護

### 一句話破題

術語表確保所有人（包括 AI）對**同一個詞有相同的理解**。

### 爲什麼術語容易混淆

同一個詞，不同人可能有不同理解：

| 術語 | 你以爲的意思 | 他以爲的意思 |
|------|--------------|--------------|
| 用戶 | 已註冊賬號 | 任何訪問者 |
| 文章 | 博客帖子 | 任何內容 |
| 管理員 | 超級管理員 | 普通運營人員 |

這種理解差異會導致：
- **代碼實現偏差**：AI 按錯誤的理解生成代碼
- **溝通效率低**：反覆解釋"我說的不是這個意思"
- **後期返工**：驗收時才發現理解不一致

### 術語表的標準格式

```markdown
## 術語表

| 術語 | 定義 | 備註 |
|------|------|------|
| 用戶（User） | 已完成註冊並通過郵箱驗證的賬號 | 與"訪客"區分 |
| 訪客（Visitor） | 未登錄狀態下的瀏覽者 | 可瀏覽公開內容 |
| 文章（Post） | 由用戶創建的博客內容 | 包含標題、正文、標籤 |
| 草稿（Draft） | 未發佈的文章 | 僅作者可見 |
| 管理員（Admin） | 擁有後臺管理權限的用戶 | 可管理所有用戶和內容 |
```

### 術語定義的原則

#### 1. 明確邊界

```
❌ 用戶：使用系統的人
（太模糊：訪客算不算用戶？）

✅ 用戶：已完成註冊並通過郵箱驗證的賬號
（清晰：有明確的判斷標準）
```

#### 2. 區分相似概念

```
✅ 用戶 vs 訪客
✅ 文章 vs 草稿
✅ 發佈 vs 保存
```

#### 3. 與代碼保持一致

術語表中的名稱應該與代碼中的命名一致：

```markdown
| 術語 | 代碼中的命名 |
|------|--------------|
| 用戶 | User |
| 文章 | Post |
| 評論 | Comment |
```

### 術語表的維護

1. **項目初期建立**：在寫第一份文檔時就創建術語表
2. **隨時補充**：發現新概念就添加
3. **統一引用**：所有文檔都引用同一份術語表
4. **定期審查**：確保定義仍然準確

### 對 AI 協作的影響

向 AI 提供術語表，能避免理解偏差：

```
在這個項目中，以下術語有特定含義：

- 用戶：已完成註冊並通過郵箱驗證的賬號
- 訪客：未登錄的瀏覽者
- 文章：用戶創建的博客內容，包含 title、content、tags

請基於以上定義，實現用戶發佈文章的功能。
```

### 簡化方案

對於小項目，可以直接在 PRD 開頭定義關鍵術語：

```markdown
## 術語說明

- **用戶**：已註冊賬號
- **文章**：博客帖子


## 功能描述
...
```

### 實用建議

1. **不要過度定義**：只定義容易混淆的術語，常識性概念不需要
2. **用示例輔助**：複雜概念可以舉例說明
3. **保持更新**：概念變化時及時更新定義
4. **中英文對照**：技術術語最好同時給出中英文
