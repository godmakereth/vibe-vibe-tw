# 5.2 爲什麼說先寫文檔再編碼——PRD 基礎

### 先寫文檔的價值

在 Vibe Coding 中，文檔不是"交差用的形式主義"，而是**你與 AI 溝通的核心界面**。

```mermaid
graph LR
    A["模糊的想法"] --> B["結構化文檔"]
    B --> C["AI 理解需求"]
    C --> D["生成代碼"]
    D --> E["對照文檔驗收"]
```

**先寫文檔的好處**：

1. **強迫你想清楚**：寫不出來說明還沒想明白
2. **減少溝通成本**：AI 一次就能理解，不用反覆解釋
3. **便於驗收**：有了文檔才知道"做對了沒有"
4. **方便迭代**：下次修改時，知道當初爲什麼這樣設計

### PRD 文檔的基本組成

一份合格的 PRD 應該包含以下元信息：

| 元素 | 作用 | 示例 |
|------|------|------|
| **文檔狀態** | 標識文檔當前階段 | 草稿 / 評審中 / 已發佈 |
| **更新記錄** | 追蹤變更歷史 | v1.1: 增加了搜索功能 |
| **相關文檔** | 關聯上下游資料 | 技術方案、API 文檔 |
| **術語表** | 統一概念定義 | "用戶"指已註冊賬號 |

### 文檔模板示例

```markdown
# [功能名稱] PRD

## 文檔信息
- **狀態**: 草稿
- **版本**: v0.1
- **作者**: [你的名字]
- **最後更新**: 2024-01-15

## 更新記錄
| 版本 | 日期 | 變更內容 | 作者 |
|------|------|----------|------|
| v0.1 | 2024-01-15 | 初稿 | xxx |

## 相關文檔
- [技術方案](./tech-spec.md)
- [API 文檔](./api.md)

## 術語表
| 術語 | 定義 |
|------|------|
| 用戶 | 已完成註冊的賬號 |
| 訪客 | 未登錄的瀏覽者 |

## 正文
[功能描述、需求說明...]
```

### 本節目標

學完本節，你將掌握：

1. **文檔狀態管理**：知道一份文檔處於什麼階段
2. **版本記錄規範**：讓每次修改都有跡可循
3. **文檔索引方法**：快速找到相關資料
4. **術語表維護**：避免溝通中的概念混亂

**記住**：寫文檔不是爲了交差，而是爲了讓 AI 更好地理解你的需求。一份好的文檔，就是一份好的 Prompt。
