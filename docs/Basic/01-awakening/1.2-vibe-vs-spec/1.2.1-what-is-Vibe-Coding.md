# 1.2.1 什麼是 Vibe Coding

## 一句話解釋

**Vibe Coding = 用自然語言告訴 AI 你想要什麼，讓 AI 幫你寫代碼。**

就像你跟一個超級能幹的助手說："幫我做個能記錄每天心情的小網頁"，然後它就做出來了。你不需要知道代碼怎麼寫，你只需要說清楚你想要什麼。

> 關於 Vibe Coding 的由來（Karpathy 的推文、Collins 年度詞彙），我們在 [1.1.1 節](/Basic/01-awakening/1.1-coder-to-commander/1.1.1-coder-to-commander.md) 已經詳細介紹過。這一節聚焦於：**Vibe Coding 到底怎麼用？**


## Vibe Coding 的四個核心特徵

### 1. 用自然語言表達需求

你不需要寫 `if...else...`、`for...loop...` 這些代碼。

你只需要說：
- "幫我做一個倒計時器"
- "我想要一個能記錄讀書筆記的網頁"
- "做一個背單詞的小程序"

AI 會理解你的意思，然後生成代碼。

### 2. 關注結果而非語法

傳統編程裏，你需要確保每一行代碼的語法都正確。

Vibe Coding 讓你從語法細節中解放出來，**把精力放在"結果對不對"上**。

你不需要檢查"分號有沒有漏"、"括號是否匹配"——AI 會處理這些。但你需要驗證：功能是不是你想要的？邏輯有沒有明顯問題？

這是一種責任轉移：語法交給 AI，結果審查交給你。

### 3. 快速迭代，跟着感覺走

不滿意？直接說：
- "按鈕換成藍色"
- "字體再大一點"
- "加一個分享功能"

幾秒鐘就能看到新結果。改到滿意爲止。

### 4. 接受"夠用就好"的起點

Vibe Coding 生成的代碼可能不是最優雅的，可能有冗餘，可能不符合"最佳實踐"。

**但對於起步階段，這完全沒問題。**

就像學騎自行車——先學會不摔倒，再談姿勢優美。先做出能用的東西，再逐步提高標準。

後面的學習會幫你理解什麼是"更好的代碼"，現在先專注於"做出來"。


## 一句話類比

Vibe Coding 就是"打車模式"——你只負責說清楚目的地，AI 負責把你送到。（更詳細的類比見 [1.1.3 節](/Basic/01-awakening/1.1-coder-to-commander/1.1.3-tools-guide.md)）


## 適用邊界

關於 Vibe Coding 適合什麼、不適合什麼，我們在 [1.1.3 節](/Basic/01-awakening/1.1-coder-to-commander/1.1.3-tools-guide.md) 已經討論過。

簡單回顧：
- ✅ 個人工具、快速驗證、學習實驗 → **適合 Vibe Coding**
- ❌ 高風險、長期維護、團隊協作 → **需要 Spec Coding**（下一節講）


## 常見誤解澄清

### Vibe Coding 是不是就是瞎搞？

不是。Vibe Coding 是一種有意識的選擇——在合適的場景下，用最快的方式達到目標。這不是"不負責任"，而是"把精力放在更重要的地方"。

### 出了問題怎麼辦？

把錯誤信息複製給 AI，讓 AI 幫忙修復。這就是 Vibe Coding 的核心循環：

**描述需求 → 生成代碼 → 測試 → 發現問題 → 描述問題 → AI 修復 → 重複**

![Vibe Coding 核心循環](/images/circle_vibe.png)

隨着使用次數增多，你會自然地開始理解代碼在做什麼。這不是必須的，但會讓你和 AI 的協作更高效。

### 這樣學不到編程吧？

Vibe Coding 是一個起點，不是終點。

很多人通過 Vibe Coding 做出第一個作品後產生了興趣，反過來開始學習代碼背後的原理。先有成就感，再談深入學習——這才符合人性。

本教程的進階版會帶你逐步理解更多技術細節，但那是後話。現在，先專注於做出東西。


::: info 小結
- Vibe Coding 是 2025 年最火的編程方式，由 Andrej Karpathy 提出
- 核心是：用自然語言描述需求，讓 AI 生成代碼
- 適合個人項目、快速驗證、學習實驗
- 不適合高風險、長期維護、團隊協作的場景
:::
