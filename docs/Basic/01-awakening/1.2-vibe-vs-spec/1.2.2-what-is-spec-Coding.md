# 1.2.2 什麼是 Spec Coding（規範驅動開發）

## 一句話解釋

**Spec Coding = 先寫清楚"要做什麼"，再讓 AI 按規範生成代碼。**

如果說 Vibe Coding 是"打車說目的地"，那 Spec Coding 就是"給司機一份詳細的路線圖"。


## 爲什麼需要 Spec Coding？

Vibe Coding 很爽，但當項目變複雜時，你會遇到問題：

- 功能越加越多，AI 開始"忘記"之前的需求
- 改一個地方，另一個地方莫名其妙壞了
- 你自己都記不清最初想做什麼了
- 和別人協作時，說不清楚項目要做成什麼樣

這時候，你需要一種更"規範"的方式——**Spec Coding**。


## 行業趨勢：Spec Coding 正在成爲主流

2025年，越來越多的工具開始支持 Spec Coding 方式：

- **Amazon Kiro**：AWS 推出的 AI IDE，核心賣點就是"Spec-Driven Development"。它要求你先寫 requirements.md、design.md、tasks.md，然後 AI 按規範生成代碼。

- **GitHub Spec Kit**：GitHub 推出的規範工具包，幫助團隊在 AI 編程時代建立"活文檔"（living documents）——隨項目演進的規範。

- **JetBrains Junie**：JetBrains 的 AI 編程助手，官方教程明確推薦"先寫需求文檔，再讓 AI 生成開發計劃"。

這說明什麼？**Spec Coding 不是"麻煩"，而是讓 AI 更好地幫你的方法**。

工具越來越智能，但"想清楚要做什麼"這件事，永遠是人類的工作。


## Spec Coding 的核心思路

**Spec** 是 Specification 的縮寫，意思是"規格說明書"。

Spec Coding 的核心是：**先想清楚、寫清楚，再動手**。

你需要先產出一些文檔，讓 AI（和你自己）都清楚：
- 要做什麼
- 怎麼做
- 分幾步做

然後 AI 按照這些文檔來生成代碼。


## Spec Coding 的三件套文檔

Spec Coding 的核心產物是三份文檔。別被"文檔"嚇到——它們可以很簡單，關鍵是**有總比沒有好**。

### 1. requirements.md（需求文檔）

**回答的問題：要做什麼？**

內容包括：
- 產品是什麼
- 用戶是誰
- 核心功能有哪些
- 不做什麼（同樣重要！）

**示例（簡化版）：**

```markdown
# 背單詞 App 需求文檔

## 產品概述
一個幫助用戶每天背 20 個單詞的網頁應用。

## 目標用戶
準備四六級考試的大學生。

## 核心功能（P0）
1. 每天展示 20 個單詞
2. 用戶可以標記"認識"或"不認識"
3. 查看今日學習進度

## 不包含（Out of Scope）
- 用戶註冊登錄
- 單詞發音
- 多設備同步
```

### 2. design.md（設計文檔）

**回答的問題：怎麼做？**

內容包括：
- 技術選型（用什麼工具/框架）
- 數據結構（信息怎麼組織）
- 頁面結構（有幾個頁面，每個頁面幹什麼）

**示例（簡化版）：**

```markdown
# 背單詞 App 設計文檔

## 技術選型
- 純前端實現，使用 HTML + CSS + JavaScript
- 數據存儲：LocalStorage

## 數據結構
- 單詞列表：[{ word: "apple", meaning: "蘋果", status: "unknown" }, ...]
- 學習進度：{ total: 20, learned: 5, date: "2025-01-15" }

## 頁面結構
1. 首頁：顯示今日進度，開始學習按鈕
2. 學習頁：卡片式展示單詞，認識/不認識按鈕
3. 結果頁：今日學習總結
```

### 3. tasks.md（任務清單）

**回答的問題：分幾步做？**

把大目標拆成小任務，每個任務都是一次可以完成的事。

**示例（簡化版）：**

```markdown
# 背單詞 App 任務清單

## 第一階段：基礎框架
- [ ] 創建項目基本文件結構
- [ ] 實現首頁 UI
- [ ] 準備 20 個測試單詞數據

## 第二階段：核心功能
- [ ] 實現單詞卡片展示
- [ ] 實現"認識/不認識"按鈕功能
- [ ] 實現進度統計

## 第三階段：數據持久化
- [ ] 用 LocalStorage 保存學習記錄
- [ ] 實現"繼續昨天的進度"功能
```


## Spec Coding vs Vibe Coding 對比

| 維度 | Vibe Coding | Spec Coding |
|------|-------------|-------------|
| **啓動速度** | 快，說幹就幹 | 慢，需要先寫文檔 |
| **適合項目規模** | 小（1-3 個功能） | 中大（5 個以上功能） |
| **代碼質量** | 夠用就行 | 相對規範 |
| **可維護性** | 低，改着改着可能亂 | 高，有文檔可追溯 |
| **協作友好度** | 低，別人看不懂 | 高，文檔是溝通工具 |
| **學習曲線** | 平，馬上能上手 | 陡一點，要學寫文檔 |


## 一個類比

還是用出行來類比：

| 場景 | 你會怎麼做？ |
|------|-------------|
| 去樓下便利店 | 直接走（Vibe Coding） |
| 第一次去一個陌生城市旅遊 | 先做攻略，列出要去的景點、路線（Spec Coding） |

**簡單任務，直接幹；複雜任務，先規劃。**


## 常見問題

### 寫文檔好麻煩，能不能讓 AI 幫忙？

完全可以。直接告訴 AI："我想做一個背單詞的應用，幫我寫一份簡單的需求文檔，包括產品概述、目標用戶、核心功能和不做什麼。" AI 會給出初稿，在此基礎上修改即可。

### 文檔要寫多詳細？

夠用就行。個人項目幾行字的清單就夠；團隊項目需要把每個功能的輸入輸出寫清楚；生產級項目則需要更專業的格式（超出本教程範圍）。

### Vibe 和 Spec 能混着用嗎？

不僅能，而且推薦這樣做。很多人的實際工作流是：先 Vibe Coding 快速搭原型 → 驗證通過後補文檔 → 後續迭代用 Spec Coding 方式。


::: info 小結
- Spec Coding 是"先規劃，再動手"的 AI 編程方式
- 核心產物是三件套文檔：需求、設計、任務清單
- 適合複雜項目、團隊協作、長期維護的場景
- 可以和 Vibe Coding 混合使用
:::
