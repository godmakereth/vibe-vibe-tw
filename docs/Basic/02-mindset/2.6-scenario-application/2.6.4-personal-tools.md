# 2.6.4 個人工具場景：警惕過度設計的陷阱

## 個人工具的特殊優勢

給自己做工具，有一個巨大的優勢：**你自己就是用戶，不需要猜測需求。**

你知道自己：
- 什麼時候會用這個工具
- 在什麼設備上用
- 能接受什麼程度的複雜度
- 什麼功能是真正需要的

這是做產品的人夢寐以求的「用戶洞察」——而你天然擁有。


## 個人工具的特殊陷阱

但這個優勢也會變成陷阱：**你很容易高估自己的需求。**

| 你想象中的需求 | 實際情況 |
|--------------|---------|
| 我需要分類功能，按工作/生活/學習分開 | 做了之後發現懶得分類，全堆在一起 |
| 我需要統計圖表，看看每月花了多少錢 | 做了之後發現從來不看 |
| 我需要多設備同步 | 做了之後發現只在一臺設備上用 |
| 我需要提醒功能 | 做了之後發現權限設置太麻煩，放棄了 |

**最後你會發現：做了10個功能，常用的只有2個。**


## 思維工具在個人工具中的應用

| 思維工具 | 在個人工具中要問的問題 |
|---------|---------------------|
| JTBD思維 | 我要完成什麼任務？現有的工具爲什麼不滿足我？ |
| 逆向思維 | 什麼會讓我做完就不用？（太複雜？打開慢？需要填很多東西？） |
| 減法思維 | 第一天必須有的功能是什麼？只做這一個 |
| 故事思維 | 我會在什麼場景下用？手機還是電腦？白天還是晚上？ |
| 靈魂三問 | 我自己的真實痛點是什麼？（不是「可能有用」，而是「真的很痛」） |


## 「過度設計」檢測清單

在開始之前，用這個清單檢測你的想法是否「過度設計」：

| 信號 | 問題 | 如果答案是「是」 |
|-----|------|----------------|
| 功能太多 | 你的功能列表超過5個了嗎？ | 砍到3個以內 |
| 解決假痛點 | 這個痛點你這周遇到過嗎？ | 如果沒有，可能不是真痛點 |
| 現有工具能用 | 現有的App/工具能解決80%的問題嗎？ | 如果能，考慮不做 |
| 想象的使用場景 | 你能具體描述「今天幾點我會用它」嗎？ | 如果說不清，可能不會用 |
| 完美主義 | 你是不是在想「要做就做好」？ | 先做一個能用的，再說「做好」 |

**如果有3個以上的「是」，建議重新審視這個項目。**


## 個人工具需求描述模板

```markdown
## 我要解決的問題
一句話描述痛點：[如「每天不知道時間花在哪了」]

這個痛點多久出現一次：[每天/每週/偶爾]
出現時我有什麼感受：[焦慮/煩躁/沮喪]

## 現有方案爲什麼不行
我現在怎麼解決這個問題：[如「用手機備忘錄記」]
爲什麼不滿意：[如「太麻煩，經常忘記記」]

## 使用場景
我會在什麼時候用：[如「晚上睡前」]
我會用什麼設備：[手機/電腦/都有]
我能接受的操作複雜度：[30秒內完成/1分鐘內完成]

## 第一版只需要的功能
只要這一個功能：[如「快速記錄今天做了什麼」]

不需要的功能（以後再說）：
- [如「分類功能」]
- [如「統計圖表」]
- [如「多設備同步」]

## 數據保存
可以保存在瀏覽器本地（最簡單）
```


## 實戰案例：極簡記賬

### 小美的情況

小美每個月發完工資，到月底就不知道錢花哪了。她想做一個記賬工具。

### 小美最初的想法

```
我想做一個個人財務管理系統：
- 收支記錄
- 分類統計（餐飲/交通/購物/娛樂...）
- 預算管理
- 月度報表
- 圖表可視化
- 多賬戶（微信/支付寶/銀行卡）
- 週期性支出（房租/話費）
- 數據導出
```

**問題**：這個列表太長了，做不完。

### 用思維工具重新審視

| 問題 | 小美的誠實回答 |
|-----|---------------|
| 你真正的痛點是什麼？ | 月底不知道錢花哪了，想知道「大頭」在哪 |
| 這個痛點多久出現一次？ | 每個月底 |
| 你會在什麼場景下記賬？ | 買完東西用手機順手記 |
| 你能接受多複雜的操作？ | 最好3秒內完成，太麻煩就懶得記了 |
| 現有App爲什麼不用？ | 太複雜，每次要選分類，煩 |

### 小美重新定義的第一版

```markdown
## 我要解決的問題
月底不知道錢花哪了，想知道大頭在哪。

## 使用場景
- 買完東西在手機上順手記一下
- 3秒內完成
- 月底看一眼「本月花了多少」

## 第一版只需要
1. 一個輸入框，輸入金額和簡單備註（如「午飯 35」）
2. 一個數字顯示「本月累計支出」
3. 能看到最近的記錄列表

就這樣。不需要分類、不需要圖表、不需要多賬戶。

## 技術要求
- 手機上打開
- 數據保存在瀏覽器本地
- 不需要登錄
```

### 這個版本爲什麼好

| 對比 | 原版 | 簡化版 |
|-----|------|-------|
| 開發時間 | 可能幾周 | 可能1小時 |
| 使用門檻 | 需要配置分類、賬戶 | 打開就能用 |
| 會不會堅持用 | 大概率放棄 | 更可能堅持 |
| 能不能解決核心痛點 | 能，但被其他功能淹沒 | 能，而且很直接 |


## 進階場景

### 場景1：極簡番茄鍾

```markdown
## 痛點
工作時容易走神，想用番茄鍾但現有App太花哨。

## 使用場景
電腦上開着，專注工作時用。

## 第一版只需要
1. 點一下開始25分鐘倒計時
2. 時間到了響一聲
3. 顯示今天完成了幾個番茄

不需要：統計報表、自定義時長、白噪音、任務關聯。
```

### 場景2：閱讀筆記收集器

```markdown
## 痛點
在微信讀書上劃了很多線，但散落在各本書裏，想集中看。

## 使用場景
讀完一本書後，把劃線內容整理到一起。

## 第一版只需要
1. 一個輸入框，粘貼劃線內容
2. 能標註是哪本書的
3. 能瀏覽所有筆記

不需要：分類標籤、搜索、導出、分享。
```


## 「做還是不做」決策流程

```
我有一個工具想法
    ↓
這個痛點這周出現過嗎？ → 沒有 → 可能是假痛點，觀察一下
    ↓ 是
現有工具能解決嗎？ → 能 → 用現有工具
    ↓ 不能
我能描述具體使用場景嗎？ → 不能 → 想法還不成熟，再想想
    ↓ 能
第一版只做1個核心功能 → 開始
```


## 個人工具自檢清單

- [ ] 我能用一句話說清楚「這個工具解決什麼問題」嗎？
- [ ] 這個痛點我這周遇到過嗎？
- [ ] 我能描述「今天什麼時候會用它」嗎？
- [ ] 第一版的功能列表是否只有1-2個？
- [ ] 我有沒有剋制住「既然做了就做全」的衝動？


## 核心要點

✓ 給自己做工具的優勢是「你就是用戶」，陷阱是「容易高估需求」

✓ 問自己：這個痛點我這周遇到過嗎？如果沒有，可能不值得做

✓ 第一版只做一個核心功能，能用就行，其他以後再說

✓ 「做完就不用」的工具，不如「簡陋但一直在用」的工具
