# 2.1.4 實踐練習：用JTBD重新審視你的想法

理論學完了，現在輪到你動手了。

這個練習會幫你把剛學到的JTBD思維應用到你自己的想法上。


## 練習一：填寫任務描述模板

拿出你現在想做的一個項目（或者用下面的示例），完成這個模板：

### 模板

```markdown
### 我的原始想法
我想做一個 _______________________

### JTBD任務描述

**用戶是誰？**
[描述具體的人，不是"所有人"]

**情境是什麼？**
[什麼時候、什麼情況下會用到？]

**要完成什麼任務？**
[用戶真正想達成的目標是什麼？]

**期望的結果/感受是什麼？**
[功能層面 + 情感層面]

**目前的替代方案？**
[用戶現在怎麼解決這個問題？]

**替代方案的痛點？**
[現有方案哪裏不好用？]

### 重新表述

當 [某類用戶] 在 [某個情境] 下，
他們想要 [完成某個任務]，
以便於 [獲得某種結果/感受]。

目前他們的方案是 [現有解決方式]，
但這個方案的問題是 [痛點]。
```

### 填寫示例

```markdown
### 我的原始想法
我想做一個讀書筆記工具

### JTBD任務描述

**用戶是誰？**
上班族，每天通勤時在手機上看電子書，每週能看1-2本

**情境是什麼？**
看書時遇到好句子想記下來，或者讀完後想回顧要點

**要完成什麼任務？**
保存書中的精華內容，日後能快速找到和回顧

**期望的結果/感受是什麼？**
功能：讀完的書有沉澱，不是看完就忘
情感：覺得時間沒白花，有成就感

**目前的替代方案？**
微信收藏、手機備忘錄、讀書App自帶的劃線功能

**替代方案的痛點？**
微信收藏找不到、備忘錄沒分類、App劃線導出麻煩

### 重新表述

當上班族在通勤讀書時看到好句子，
他們想要快速保存下來，並在日後能方便地回顧，
以便於讓讀書有沉澱、感到時間沒白花。

目前他們用微信收藏或備忘錄，
但內容零散、難以檢索、無法形成體系。
```


## 練習二：自檢清單

完成練習一後，用這個清單檢驗你的任務描述是否合格：

- [ ] **用戶具體嗎？** 不是"所有人"，而是有具體特徵的人
- [ ] **情境明確嗎？** 說清楚了"什麼時候、什麼情況下"
- [ ] **任務是動詞嗎？** 是"完成某事"而不是"擁有某功能"
- [ ] **包含情感層面嗎？** 不只是功能需求，還有心理需求
- [ ] **知道替代方案嗎？** 瞭解用戶現在怎麼解決
- [ ] **痛點真實嗎？** 替代方案的問題是你觀察到的，不是想象的

**如果有任何一項打不上勾，回去繼續完善你的描述。**


## 練習三：案例對比分析

看看下面兩個產品，思考它們成功/失敗的原因：

### 案例A：某習慣打卡App（失敗）

**產品功能：**
- 21天習慣養成計劃
- 每日打卡提醒
- 社交分享、排行榜
- 習慣統計、成就係統
- 付費解鎖高級功能

**結果：**
上線3個月後，日活不到1000，最終下架

**問題分析：**
這個App假設用戶需要"養成習慣"，但沒有問：
- 用戶真的想養成什麼習慣？
- 爲什麼他們之前沒養成？
- 一個App真的能解決意志力問題嗎？

**JTBD視角：**
用戶的任務不是"打卡"，而是"變成更好的自己"。
打卡只是手段，不是目的。
這個App把手段當成了目的。


### 案例B：一個簡單的計數器網頁（成功）

**產品功能：**
- 一個大按鈕
- 點一下加一
- 顯示今天的次數
- 沒了

**結果：**
一位健身教練用它記錄每天做了多少個俯臥撐。
沒有花哨功能，但每天都在用。

**成功原因：**
精準解決了一個具體任務——"記錄今天做了多少個"。
不多也不少。

**JTBD視角：**

```
當健身教練鍛鍊時，
他想要快速記錄每組做了多少個，
以便於追蹤進步、保持動力。

目前他用腦子記或者紙筆，
但容易忘記、事後補記不準確。
```


### 思考題

1. 案例A的功能比案例B多得多，爲什麼反而失敗了？
2. 如果你是案例A的產品經理，用JTBD視角重新定義問題，你會怎麼做？
3. 案例B成功的關鍵是什麼？
