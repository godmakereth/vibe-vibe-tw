# 2.1.2 初學者最常犯的三個錯誤

在開始學習正確的方法之前，我們先來看看大多數人是怎麼失敗的。

瞭解這些錯誤，能幫你少走很多彎路。


## 錯誤一：功能堆砌症

### 典型表現

```
"幫我做一個待辦清單App，要有：
任務分類、優先級標籤、截止日期提醒、重複任務、
子任務拆解、標籤系統、日曆視圖、統計報表、
多設備同步、協作共享、暗黑模式……"
```

### 爲什麼這是錯的

1. **你在列功能清單，而不是在解決問題**

   這個需求裏沒有提到：誰會用？在什麼場景下用？要解決什麼問題？

2. **你在假設用戶需要這些**

   這些功能是你想象出來的，還是真的有人需要？你驗證過嗎？

3. **複雜度會殺死項目**

   給AI一個20項功能的需求，AI會嘗試同時滿足所有要求。結果通常是一團亂麻的代碼，充滿bug，難以調試。

### 真實後果

小李嘗試讓AI做這個"功能齊全"的待辦清單，三個小時後：

- AI生成了超過2000行代碼
- 頁面加載後一片空白
- 控制檯報了十幾個錯誤
- 小李完全看不懂哪裏出了問題

**功能越多，失敗概率越高。這是鐵律。**


## 錯誤二：解決不存在的問題

### 典型表現

```
"幫我做一個數據可視化看板，展示：
用戶活躍度趨勢、留存率漏斗、轉化率分析、
用戶畫像分佈、行爲路徑追蹤……"
```

### 爲什麼這是錯的

問題在於：**你真的需要看這些數據嗎？看了之後要做什麼決策？**

很多人做數據分析項目時，會陷入"數據越多越好"的誤區。他們花大量時間搭建看板，最後發現：

- 沒人看這些圖表
- 看了也不知道該幹什麼
- 數據沒有驅動任何行動

### 真實後果

一位產品經理花了兩週時間，用AI做了一個"全面"的數據看板：

- 20多個圖表，覆蓋各種指標
- 可以按日、周、月切換
- 支持導出PDF報告

結果呢？

- 老闆只看了一眼銷售額趨勢圖
- 其他圖表從未被點擊過
- 兩週的時間可以做10個小功能了

**先問自己：我要回答什麼問題？我會根據答案做什麼決策？**


## 錯誤三：爲技術而技術

### 典型表現

```
"幫我寫一個Python腳本，用Pandas讀取Excel，
然後用正則表達式清洗數據，再用Matplotlib畫圖，
最後導出成PDF……"
```

### 爲什麼這是錯的

這個需求直接跳到了"怎麼做"，完全跳過了"爲什麼做"。

更好的問法是：

> "我每週要手動彙總5個部門的銷售數據，現在用的是複製粘貼。有沒有辦法讓這個過程自動化？"

### 兩種問法的區別

| 爲技術而技術 | 爲問題而技術 |
|------------|------------|
| 我要用Pandas處理Excel | 我要自動彙總5個Excel |
| 我要用正則表達式 | 數據格式有點亂需要清理 |
| 我要用Matplotlib畫圖 | 老闆想看趨勢變化 |
| 我要導出PDF | 最後要發郵件給團隊 |

**左邊那列是"怎麼做"，右邊那列是"做什麼"。**

AI可以幫你想"怎麼做"，但"做什麼"只有你自己能決定。

如果你把"怎麼做"也一起規定了，反而限制了AI發揮。也許有比Pandas更簡單的方案呢？


## 錯誤後果對比

| 錯誤類型 | 表面行爲 | 深層問題 | 典型後果 |
|---------|---------|---------|---------|
| 功能堆砌 | 一次性要求20個功能 | 沒想清楚核心問題 | 代碼混亂，bug滿天飛 |
| 解決不存在的問題 | 做一堆沒人看的功能 | 沒驗證真實需求 | 時間浪費，成果無用 |
| 爲技術而技術 | 指定具體技術方案 | 把手段當目的 | 限制了更好的解決方案 |


## 正確思維的起點

避開這三個錯誤的方法，歸結起來就是一句話：

**先想清楚"要解決什麼問題"，再考慮"怎麼解決"。**

這聽起來很簡單，但需要刻意練習。
