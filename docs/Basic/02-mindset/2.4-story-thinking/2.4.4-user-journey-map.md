# 2.4.4 用戶旅程地圖：把故事可視化

## 什麼是用戶旅程地圖

用戶旅程地圖是一個可視化工具，它把用戶與產品互動的完整過程畫出來。

你可以把它理解爲「用戶故事的分鏡腳本」——不只是講發生了什麼，還要畫出每一步用戶的行爲、想法和情緒。

爲什麼需要把故事「畫出來」？

- 文字描述容易遺漏細節，圖表更全面
- 情緒變化一目瞭然，能快速定位問題
- 方便與他人（或 AI）溝通，減少誤解


## 旅程地圖的五個維度

一份完整的用戶旅程地圖包含五個維度：

| 維度 | 記錄的內容 | 回答的問題 |
|-----|-----------|-----------|
| **階段** | 用戶與產品互動的主要階段 | 用戶的旅程分幾步？ |
| **行爲** | 用戶在每個階段做了什麼 | 用戶具體做了什麼操作？ |
| **想法** | 用戶在每個階段想什麼 | 用戶腦子裏在想什麼？ |
| **情緒** | 用戶在每個階段的情緒 | 用戶感到開心、困惑還是沮喪？ |
| **痛點/機會** | 問題和改進空間 | 哪裏可以做得更好？ |


## 案例：待辦清單的用戶旅程

讓我們爲小李畫一份完整的用戶旅程地圖：

### 旅程地圖表格

| 階段 | 用戶行爲 | 用戶想法 | 情緒 | 痛點/機會 |
|-----|---------|---------|------|----------|
| **發現** | 在應用商店搜索「待辦」 | "又一個待辦 App？希望別太複雜" | 😐 懷疑 | 第一印象很重要，簡潔的介紹文案 |
| **下載安裝** | 下載、打開 App | "看看長什麼樣" | 😐 中性 | 加載速度要快，別讓用戶等 |
| **首次使用** | 添加第一個任務 | "就這？挺簡單的" | 😊 驚喜 | 零門檻添加任務，創造正面印象 |
| **完成任務** | 點擊完成，任務劃掉 | "有點爽" | 😄 滿足 | 完成時的反饋要有成就感 |
| **日常使用** | 每天早上列待辦，做完劃掉 | "腦子輕鬆多了" | 😌 安心 | 形成習慣，成爲日常的一部分 |
| **遇到邊界** | 想要某個沒有的功能 | "要是能有 XX 就好了" | 😕 輕微失望 | 記錄需求，判斷是否真的需要 |
| **長期使用** | 已經用了幾個月 | "沒有它我會焦慮" | 😊 依賴 | 這是成功的標誌 |
| **推薦他人** | 向朋友安利 | "你也試試，特別簡單" | 😄 分享欲 | 口碑傳播的機會 |

### 情緒曲線

把情緒變化連成線，就形成了「情緒曲線」：

```
高 😄 ─────────────────────────────────────────────
      │                   ╱─╲              ╱─────
      │                  ╱   ╲            ╱
      │                 ╱     ╲          ╱
中 😐 │────╲           ╱       ╲        ╱
      │     ╲         ╱         ╲      ╱
      │      ╲       ╱           ╲    ╱
      │       ╲─────╱             ╲──╱
低 😞 ─────────────────────────────────────────────
      發現   下載   首次   完成   日常   遇到邊界   長期   推薦
```

從這個曲線可以看出：

- **情緒最低點**：發現階段（懷疑）和遇到邊界時（失望）
- **情緒最高點**：首次使用和完成任務時（驚喜和滿足）

這些情緒波動告訴你：
- 在「發現」階段要做好第一印象
- 「首次使用」的體驗要儘可能簡單
- 「完成任務」時要給足正反饋


## 簡化版：適合個人項目

上面的旅程地圖很完整，但做個人項目時不需要這麼複雜。

一個簡化版的旅程地圖只需要關注四個關鍵階段：

| 階段 | 你需要回答的問題 |
|-----|----------------|
| **發現** | 用戶怎麼知道這個工具的？他的第一反應是什麼？ |
| **首次使用** | 用戶第一次用時會做什麼？有沒有門檻？ |
| **核心流程** | 用戶最常重複做的事情是什麼？體驗流暢嗎？ |
| **長期價值** | 用戶爲什麼會持續使用？什麼會讓他推薦給別人？ |

簡化版模板：

```markdown
## 簡化版用戶旅程

### 發現階段
- 用戶如何發現這個工具：[來源]
- 用戶的第一反應：[想法]
- 關鍵挑戰：[如何讓用戶願意嘗試]

### 首次使用
- 用戶第一步會做什麼：[行爲]
- 可能遇到的困惑：[痛點]
- 期望的感受：[情緒]

### 核心流程
- 用戶最常做的操作：[行爲]
- 這個過程順暢嗎：[體驗評估]
- 如何讓體驗更好：[機會]

### 長期價值
- 用戶爲什麼會持續使用：[價值]
- 什麼會讓用戶推薦給別人：[觸發點]
```


## 非產品場景的旅程應用

旅程地圖不只適用於做產品。任何需要理解「用戶完整體驗」的場景都可以用。

### 場景一：數據分析——老闆看報告的旅程

| 階段 | 老闆的行爲 | 老闆的想法 | 痛點/機會 |
|-----|-----------|-----------|----------|
| **收到報告** | 看到郵件標題 | "又一份報告……" | 標題要直接說結論 |
| **打開報告** | 快速掃一眼 | "多少頁？關鍵是什麼？" | 第一頁要有摘要 |
| **找結論** | 翻到關鍵圖表 | "所以呢？我該做什麼？" | 圖表要有明確結論 |
| **做決策** | 根據數據決定下一步 | "這個分析靠譜嗎？" | 數據要有來源和信心度 |

**洞察**：老闆沒有時間看過程，只想要答案。報告的結構應該是「結論→證據→細節」，而不是「背景→分析→結論」。

### 場景二：自動化腳本——使用前後的對比

| 階段 | 使用腳本前 | 使用腳本後 |
|-----|-----------|-----------|
| **觸發** | 週五下午，該彙總數據了 | 週五下午，運行腳本 |
| **執行** | 打開 5 個 Excel，複製粘貼 2 小時 | 等待 30 秒 |
| **檢查** | 檢查有沒有漏數據、格式對不對 | 快速瀏覽結果 |
| **結束** | 身心疲憊，終於搞完了 | 輕鬆完成，開始週末 |
| **情緒** | 😩 疲憊、怕出錯 | 😌 輕鬆、有成就感 |

**洞察**：自動化腳本的價值不只是「省時間」，還有「省心力」和「減少出錯焦慮」。


## 關鍵洞察：痛點最高處就是最大機會

從旅程地圖中找到的最有價值的信息是：**情緒最低的地方**。

那通常就是你最應該解決的問題。

| 場景 | 情緒最低點 | 機會 |
|-----|-----------|------|
| 待辦清單 | 發現階段的懷疑 | 用極簡設計打消疑慮 |
| 數據報告 | 老闆找不到結論 | 第一頁就給答案 |
| 自動化腳本 | 手動執行時的疲憊 | 讓腳本一鍵運行 |
| 喫藥提醒 | 記不清喫沒喫過 | 明確顯示今天的狀態 |


## 本節要點

用戶旅程地圖幫你把故事變成可操作的分析工具：

- 旅程地圖有五個維度：階段、行爲、想法、情緒、痛點
- 情緒曲線讓你一眼看出問題在哪
- 個人項目可以用簡化版：發現→首次使用→核心流程→長期價值
- 情緒最低的地方就是最大的機會
