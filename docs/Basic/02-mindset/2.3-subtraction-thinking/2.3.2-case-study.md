# 2.3.2 真實案例對比

讓我們通過幾個真實案例，看看「功能堆砌」和「減法思維」的區別。


## 反面案例：14 功能待辦清單的失敗

### 項目背景

小李想做一個待辦清單工具，幫助自己管理工作事項。

他的第一版規劃包含以下功能：

1. 任務分類
2. 優先級標籤
3. 截止日期提醒
4. 重複任務
5. 子任務拆解
6. 標籤系統
7. 日曆視圖
8. 看板視圖
9. 統計報表
10. 多設備同步
11. 團隊協作
12. 評論功能
13. 暗黑模式
14. 桌面小組件

### 發生了什麼

- 第 1 周：搭建項目框架，實現了基礎的添加任務功能
- 第 2-4 周：開始做任務分類和標籤，遇到數據結構問題，反覆修改
- 第 5-8 周：日曆視圖和看板視圖開發，UI 調整花了大量時間
- 第 9-12 周：多設備同步需要後端，學習後端知識，進度緩慢
- 第 12 周後：心態崩潰，項目擱置

### 結果

- 斷斷續續做了 3 個月，功能還沒完成一半
- 每個功能都是半成品，充滿 bug
- 自己試用後發現：「這比我現在用的便籤紙還麻煩」
- 項目放棄

### 問題分析

| 問題 | 具體表現 |
|-----|---------|
| 沒有核心假設 | 不知道在驗證什麼，只是在「做功能」 |
| 功能太多 | 14 個功能分散了精力，沒有一個做好 |
| 沒有驗證節點 | 3 個月後才發現方向可能有問題 |
| 完美主義 | 每個功能都想做到完美，反而什麼都沒完成 |


## 正面案例：「只需一個按鈕」的習慣打卡

### 項目背景

另一個人想做一個習慣養成工具。

她的核心假設是：**人們堅持不了習慣，是因爲記錄太麻煩。如果記錄只需要點一下按鈕，堅持率會提高。**

### 她的 MVP

只有三個元素：

1. 一個習慣名稱（比如「喝水」）
2. 一個大按鈕（點擊 = 今天完成了）
3. 一個連續天數顯示

沒有統計圖表，沒有提醒功能，沒有社交分享，沒有成就係統。

### 發生了什麼

- 第 1 天：用 AI 生成了這個簡單頁面
- 第 2-7 天：自己試用，發現確實比之前用的 App 更容易堅持
- 第 2 周：發給 5 個朋友試用，收到反饋
- 第 3 周：根據反饋加了「多個習慣」功能
- 第 4 周：加了簡單的周統計

### 結果

- 1 個月內驗證了核心假設：「簡單確實能提高堅持率」
- 收集了真實用戶反饋，知道下一步該做什麼
- 項目持續迭代，現在有 50 多個朋友在用


## 更多場景的案例對比

### 數據分析場景

| 方面 | 失敗做法 | 成功做法 |
|-----|---------|---------|
| 目標 | 做一份「全面」的銷售分析報告 | 回答老闆的一個問題：「哪個渠道 ROI 最高？」 |
| 內容 | 50 個圖表，覆蓋所有指標 | 1 張對比圖 + 3 句結論 |
| 時間 | 花了 2 周，老闆沒時間看完 | 花了 2 小時，老闆當天就做了決策 |
| 價值 | 「辛苦了，放這兒吧」 | 「這個分析很有用，下次再做一個 XX」 |

### 自動化腳本場景

| 方面 | 失敗做法 | 成功做法 |
|-----|---------|---------|
| 目標 | 做一個「萬能」Excel 處理工具 | 自動化「每週五彙總 5 個部門數據」這一件事 |
| 功能 | 支持各種格式、各種操作 | 只做這一個特定任務 |
| 時間 | 做了 1 個月，還在加功能 | 2 小時搞定，每週省 3 小時 |
| 維護 | 代碼複雜，改一個地方壞另一個 | 代碼簡單，自己能維護 |

### 給家人做工具場景

| 方面 | 失敗做法 | 成功做法 |
|-----|---------|---------|
| 目標 | 給爸媽做一個「完美」的喫藥提醒 App | 讓爸媽不忘喫藥 |
| 設計 | 精美 UI、多種提醒方式、用藥記錄 | 大字、一個「喫了」按鈕 |
| 結果 | 爸媽覺得太複雜，不願意用 | 爸媽每天都在用 |
| 關鍵差異 | 按自己的審美設計 | 按爸媽的使用能力設計 |


## 失敗項目 vs 成功項目的特徵對比

| 特徵 | 失敗項目 | 成功項目 |
|-----|---------|---------|
| 核心假設 | 模糊或沒有 | 清晰明確 |
| 功能數量 | 越多越好 | 只做必須的 |
| 驗證週期 | 很長，甚至沒有 | 很短，快速迭代 |
| 成功標準 | 「功能都做完了」 | 「假設被驗證了」 |
| 用戶參與 | 做完再給用戶看 | 儘早讓用戶參與 |
| 對待反饋 | 防禦性：「他們不懂」 | 開放性：「他們的反饋很有價值」 |


## 本節要點

通過這些案例，你應該能看出區別：

- 失敗項目往往死於「功能太多」，而不是「功能太少」
- 成功項目的共同點是「聚焦」和「快速驗證」
- 減法思維的核心是：**先驗證假設，再考慮擴展**
