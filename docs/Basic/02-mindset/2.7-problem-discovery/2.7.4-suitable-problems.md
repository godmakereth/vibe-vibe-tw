# 2.7.4 什麼樣的問題適合用 Vibe Coding 解決

## 不是所有問題都適合

通過前面的觀察和記錄，你可能已經積累了一些待解決的問題。但在投入時間之前，需要做一個重要的判斷：

**這個問題，適合用 Vibe Coding 解決嗎？**

Vibe Coding 的優勢是快速、低門檻、容錯性高。但它也有明確的邊界。選對問題，事半功倍；選錯問題，可能浪費時間，甚至造成風險。


## 五維評估法

我們用五個維度來評估一個問題是否適合用 Vibe Coding 解決。每個維度滿分 5 分，總分 25 分。

### 維度一：重複性（這件事多久發生一次？）

| 頻率 | 分數 | 說明 |
|-----|------|------|
| 每天發生 | 5 分 | 高頻問題，解決後每天都能受益 |
| 每週發生 | 4 分 | 中頻問題，值得投入 |
| 每月發生 | 3 分 | 低頻但如果很痛也值得 |
| 每季度/每年 | 1 分 | 太低頻，專門做工具可能不划算 |
| 只發生一次 | 0 分 | 手動做完就行，不值得做工具 |

**判斷技巧**：問自己"如果這個問題今天解決了，下週我還會遇到嗎？"


### 維度二：規則性（這件事有固定的步驟嗎？）

| 規則性 | 分數 | 說明 |
|-------|------|------|
| 步驟完全固定 | 5 分 | 可以寫成清晰的流程圖 |
| 大部分固定，少數需要判斷 | 4 分 | 可以做主流程，特殊情況手動處理 |
| 一半固定一半靈活 | 3 分 | 需要更多人工介入 |
| 大部分需要判斷 | 1 分 | 難以自動化 |
| 完全靠經驗和直覺 | 0 分 | 不適合做成工具 |

**判斷技巧**：試着把這件事的步驟寫下來。如果寫不出來，說明規則性不夠。

**示例**：

| 任務 | 規則性 | 分數 |
|-----|-------|------|
| 從 5 個 Excel 彙總數據 | 列名固定，提取規則固定 | 5 分 |
| 判斷一封郵件該不該回復 | 需要理解內容和上下文 | 1 分 |
| 把會議錄音轉成文字 | 步驟固定（上傳→轉錄→輸出） | 5 分 |
| 決定要不要投資一隻股票 | 涉及大量主觀判斷 | 0 分 |


### 維度三：可驗證性（怎麼知道做對了？）

| 可驗證性 | 分數 | 說明 |
|---------|------|------|
| 結果非黑即白 | 5 分 | 要麼對要麼錯，一眼能看出來 |
| 有明確的檢查標準 | 4 分 | 可以對照清單驗證 |
| 需要人工審覈 | 3 分 | 做完後要花時間檢查 |
| 好壞難以判斷 | 1 分 | 很難說"做完了"還是"做對了" |
| 完全主觀 | 0 分 | 沒有標準答案 |

**判斷技巧**：問自己"如果 AI 幫我做完了，我怎麼知道它做得對不對？"

**示例**：

| 任務 | 可驗證性 | 分數 |
|-----|---------|------|
| 計算 5 個表格的數據總和 | 對照原數據可以驗算 | 5 分 |
| 寫一封得體的郵件 | 需要人判斷語氣是否合適 | 2 分 |
| 生成一張營銷海報 | 審美因人而異 | 1 分 |


### 維度四：敏感性（涉及的數據有多敏感？）

| 敏感性 | 分數 | 說明 |
|-------|------|------|
| 完全不敏感 | 5 分 | 公開信息，無隱私，無風險 |
| 輕微敏感 | 4 分 | 個人偏好、日程安排等 |
| 中等敏感 | 2 分 | 涉及工作內容，需要注意 |
| 高度敏感 | 1 分 | 涉及他人隱私或商業機密 |
| 極度敏感 | 0 分 | 密碼、支付信息、醫療記錄等 |

**判斷技巧**：問自己"如果這些數據被泄露了，會有什麼後果？"

**紅線**：以下類型的數據，不建議用 Vibe Coding 處理：
- 銀行賬戶、支付信息、密碼
- 身份證號、護照號等身份信息
- 醫療記錄、健康數據
- 他人的隱私信息（未經授權）


### 維度五：容錯性（做錯了會怎樣？）

| 容錯性 | 分數 | 說明 |
|-------|------|------|
| 做錯了沒任何影響 | 5 分 | 最壞情況就是重做 |
| 做錯了有點麻煩 | 4 分 | 需要花時間修復，但不嚴重 |
| 做錯了比較麻煩 | 3 分 | 需要他人幫忙或花較多時間修復 |
| 做錯了很嚴重 | 1 分 | 可能影響工作或他人 |
| 做錯了不可挽回 | 0 分 | 涉及資金、法律、安全 |

**判斷技巧**：問自己"如果工具出 bug 了，最壞的情況是什麼？"

**示例**：

| 任務 | 容錯性 | 分數 |
|-----|-------|------|
| 生成一個個人用的待辦清單 | 最壞就是數據丟失，重新記 | 5 分 |
| 生成一份給老闆看的週報 | 可能出錯要修改，但影響不大 | 4 分 |
| 自動發送郵件給客戶 | 發錯了可能影響公司形象 | 2 分 |
| 自動執行股票交易 | 做錯了可能損失真金白銀 | 0 分 |

## 紅線清單：這些問題不要碰

無論評分多高，以下類型的問題都**不建議**用 Vibe Coding 解決：

| 紅線類型 | 原因 | 示例 |
|---------|------|------|
| **涉及資金流動** | 安全風險極高，一旦出錯可能造成真實損失 | 自動轉賬、支付、理財操作 |
| **涉及用戶隱私** | 法律風險，數據泄露可能違法 | 收集他人信息、處理醫療數據 |
| **涉及監管合規** | 需要專業資質和知識 | 醫療診斷、法律諮詢、財務報表 |
| **需要長期維護** | 技術債務會累積，Vibe Coding 難以持續 | 多年運營的商業系統 |
| **高併發場景** | 需要專業的後端架構 | 同時幾千人使用的應用 |

**記住**：Vibe Coding 的定位是"快速驗證"和"個人工具"，不是替代專業開發。


## 灰色地帶：需要專業支持的情況

有些問題不在紅線內，但也不是完全適合自己用 Vibe Coding 解決。這時候你有兩個選擇：

### 選擇一：降級問題範圍

把問題縮小到安全範圍內。

**示例**：
- 原問題："做一個給客戶用的預約系統" → 涉及用戶數據
- 降級後："做一個只有我自己用的日程管理工具" → 只涉及自己的數據

### 選擇二：尋求專業支持

如果你確實需要做一個涉及敏感領域的產品，考慮：
- 找一個有經驗的開發者幫你審覈關鍵部分
- 使用成熟的第三方服務（如支付用 Stripe、認證用 Auth0）
- 把敏感部分外包，自己只做非敏感部分


## 邊界情況判斷

### "我的數據只給自己用，算不算敏感？"

如果只是自己的個人數據（日程、讀書筆記、待辦事項），且存儲在本地或個人賬戶裏，通常是安全的。

但要注意：如果你的"個人工具"以後可能給別人用，就要提前考慮數據安全。

### "我想做的東西有一點點涉及敏感信息，怎麼辦？"

儘量把敏感信息"剝離"出來：
- 不在工具裏存儲敏感信息，只存儲索引
- 敏感信息用鏈接指向原始位置，不復制到工具裏
- 不做"自動處理"敏感信息的功能，保留人工確認環節

### "我不確定我的問題適不適合，怎麼辦？"

兩個建議：
1. 先做一個"最安全"的版本：只解決核心問題，不涉及任何灰色地帶
2. 在 2.5 節學過的"靈魂三問"自檢清單裏再過一遍


## 核心要點

做好問題篩選，能幫你：

✓ 把有限的時間投入到最有價值的問題上

✓ 避免在不適合的問題上浪費精力

✓ 識別需要專業支持的場景，避免踩坑

**記住**：不是所有問題都要靠自己解決。知道什麼不做，和知道什麼要做一樣重要。
