# C. 常見誤區診斷清單

本附錄彙總第二章七個思維模型的常見誤區。每個誤區都配有「症狀描述」「問題診斷」「改進建議」，幫助你自檢和避坑。


## 如何使用本清單

1. **自檢**：對照每個誤區的「症狀」，看看自己是否踩坑
2. **診斷**：理解爲什麼這是問題
3. **改進**：按建議調整你的思路


## 一、JTBD 思維的常見誤區

### 誤區 1.1：只描述功能，不描述任務

**症狀**

你的需求描述是這樣的：
> 「我想做一個待辦清單 App，要有添加任務、刪除任務、分類標籤、截止日期……」

**問題診斷**

這是「功能清單」，不是「任務描述」。它告訴 AI 你想要什麼功能，但沒有說明用戶要完成什麼任務、爲什麼需要這些功能。

AI 可能會照單全做，但做出來的東西不一定能解決真正的問題。

**改進建議**

用 JTBD 句式重新描述：
> 「當職場新人在早上開始工作時，想要快速記錄今天要做的事，以便於不會遺漏重要任務。」

然後再推導功能：既然任務是「快速記錄」，那麼「添加任務」必須在 3 秒內完成。「分類標籤」可能反而會拖慢速度，可以不做。


### 誤區 1.2：只看功能任務，忽略情感和社會任務

**症狀**

你只考慮用戶「要做什麼」，沒有考慮用戶「想要什麼感受」和「想被如何看待」。

**問題診斷**

用戶選擇一個產品，往往不只是因爲功能，還因爲情感和社會因素。

- 一個待辦清單 App，功能任務是「記錄任務」
- 情感任務可能是「減少焦慮、感到掌控」
- 社會任務可能是「在同事面前顯得靠譜」

如果你只關注功能任務，可能會做出一個「功能正確但沒人想用」的產品。

**改進建議**

每次做 JTBD 分析時，都問自己三個問題：

| 層次 | 問題 |
|-----|------|
| 功能任務 | 用戶要完成什麼具體事情？ |
| 情感任務 | 用戶想要什麼感受？ |
| 社會任務 | 用戶想被別人如何看待？ |


### 誤區 1.3：把「我想做」當成「用戶需要」

**症狀**

你的出發點是「我想做一個 xxx」，而不是「我發現用戶需要 xxx」。

**問題診斷**

這是一個常見的心理偏差。我們往往會假設自己的想法就是用戶的需求，而不去驗證這個假設。

結果是：花了大量時間做出來的東西，只有自己覺得好。

**改進建議**

把「我想做」轉換成「用戶需要」：

| 原始想法 | 轉換後 |
|---------|-------|
| 我想做一個待辦清單 | 誰需要待辦清單？他們現在怎麼解決這個問題？ |
| 我覺得這個功能很酷 | 有沒有用戶真的需要這個功能？ |

最好的驗證方式：找到 3 個目標用戶，問問他們現在是怎麼做的。


## 二、逆向思維的常見誤區

### 誤區 2.1：只列風險，不想應對措施

**症狀**

你的 Pre-mortem 分析是這樣的：
> 「可能失敗的原因：功能太多做不完、用戶不買賬、技術實現不了……」

然後就沒了。

**問題診斷**

列出風險只是第一步。如果沒有對應的預防措施，這個分析就是空談。

**改進建議**

每個風險必須配一個預防措施。用這個格式：

```
風險：_______________
預防措施：_______________
```

如果想不出預防措施，說明這個風險需要更認真對待——要麼它真的很難避免（那可能不該做），要麼你還沒想清楚（那需要繼續思考）。


### 誤區 2.2：過度悲觀，被風險嚇住

**症狀**

你做完 Pre-mortem 分析後，發現有 10 個可能失敗的原因，然後決定不做了。

**問題診斷**

逆向思維不是爲了讓你放棄，而是爲了讓你「明知有坑還能避開」。

每個項目都有風險，關鍵是識別哪些是「致命風險」（必須避免），哪些是「可控風險」（可以接受）。

**改進建議**

用「可能性 × 嚴重性」矩陣評估每個風險：

| | 嚴重性高 | 嚴重性低 |
|---|---------|---------|
| **可能性高** | 必須解決 | 注意即可 |
| **可能性低** | 準備預案 | 可以忽略 |

只有「可能性高 + 嚴重性高」的風險是致命的。如果這類風險無法避免，可能確實不該做。但如果可以預防，那就做好預防措施，然後繼續。


### 誤區 2.3：Pre-mortem 做一次就完了

**症狀**

你在項目開始時做了 Pre-mortem，然後再也沒有回顧過。

**問題診斷**

項目進行過程中，會出現新的信息。當初沒預料到的風險可能會出現，當初擔心的風險可能已經化解。

**改進建議**

在項目關鍵節點（如 MVP 完成、第一次用戶反饋後）重新審視 Pre-mortem 清單：
- 有沒有新增的風險？
- 當初的預防措施有效嗎？
- 有沒有風險已經可以劃掉？


## 三、減法思維的常見誤區

### 誤區 3.1：把 MVP 當成「簡陋版」

**症狀**

你的 MVP 是「功能殘缺、體驗粗糙、湊合能用」的版本。

**問題診斷**

MVP 是「最小可驗證版本」，不是「最小可運行版本」。

關鍵區別：
- 錯誤理解：功能越少越好，能跑就行
- 正確理解：保留足以驗證核心假設的功能，這些功能要做好

如果你的 MVP 體驗太差，用戶可能因爲「不好用」而放棄，而不是因爲「不需要」而放棄。你就沒法判斷問題是出在產品還是需求。

**改進建議**

MVP 的原則是：功能要少，但核心功能要做到位。

| 維度 | MVP 應該做的 | MVP 不應該做的 |
|-----|------------|--------------|
| 功能數量 | 只做 P0（3 個以內） | 不做 P1、P2 |
| 功能深度 | P0 功能做到好用 | 不湊合、不敷衍 |
| 視覺設計 | 清晰可用 | 不需要精美 |


### 誤區 3.2：砍功能時心裏過不去

**症狀**

你知道應該做 MVP，但每個功能都捨不得砍：
> 「這個功能很重要啊……那個功能競品都有……這個功能我都想好怎麼做了……」

**問題診斷**

這是正常的心理。人類天生厭惡損失，砍掉已經想過的功能會讓你覺得「虧了」。

但如果不砍，你會花更多時間在不重要的功能上，拖延真正的驗證。

**改進建議**

用「不做清單」代替「功能清單」。不是「刪掉功能」，而是「把功能放到不做清單」。

心理上的區別：
- 「刪掉」= 這個功能沒了
- 「放到不做清單」= 這個功能我想過了，決定不做，原因是 xxx，將來 xxx 時候再考慮


### 誤區 3.3：不做清單太空泛

**症狀**

你的不做清單是這樣的：
> 「不做複雜的功能」「不做不必要的功能」

**問題診斷**

這種不做清單等於沒有。什麼是「複雜」？什麼是「不必要」？沒有清晰的邊界，遇到具體問題時還是會糾結。

**改進建議**

不做清單要具體到功能名稱，並寫明理由：

```
不做 多設備同步，因爲 需要後端開發，大大增加複雜度
不做 任務分類標籤，因爲 不是極簡體驗的核心
不做 截止日期提醒，因爲 先驗證「記錄」這個核心價值
```


## 四、故事思維的常見誤區

### 誤區 4.1：用戶畫像只有人口統計學特徵

**症狀**

你的用戶畫像是這樣的：
> 「25-35 歲，大城市，大學學歷，月收入 1-2 萬」

**問題診斷**

這些信息對產品設計幾乎沒有幫助。知道用戶是「25-35 歲」，並不能告訴你他需要什麼功能、會在什麼場景下使用、有什麼顧慮。

**改進建議**

用「三維畫像」代替人口統計學：

| 維度 | 應該包含的內容 |
|-----|--------------|
| 表面屬性 | 職業、每天處理多少事務、使用什麼設備 |
| 行爲習慣 | 什麼時候用、怎麼用、現在用什麼替代方案 |
| 深層動機 | 害怕什麼、追求什麼、想成爲什麼樣的人 |


### 誤區 4.2：用戶是「一羣人」而不是「一個人」

**症狀**

你說的用戶是「年輕人」「上班族」「學生」這樣的羣體。

**問題診斷**

羣體太抽象，無法指導具體的產品設計。「年輕人」裏有學生、有職場新人、有創業者，他們的需求完全不同。

**改進建議**

給你的用戶起個名字，把 TA 具體化到「能發微信問問」的程度：

| 抽象 | 具體 |
|-----|------|
| 年輕的上班族 | 小李，25 歲，互聯網公司運營，每天處理 10-15 件事務 |
| 經常忘事的人 | 小李，上週因爲忘了給客戶回郵件被領導批評 |


### 誤區 4.3：用戶旅程只考慮「正常路徑」

**症狀**

你的用戶旅程是「打開 App → 添加任務 → 完成任務 → 關閉」。

**問題診斷**

這只是理想情況。真實場景中，用戶會遇到各種意外：
- 添加任務時被打斷怎麼辦？
- 任務太多看不過來怎麼辦？
- 幾天沒用再打開時怎麼辦？

**改進建議**

在用戶旅程中加入「異常路徑」和「邊界情況」：

```
正常路徑：打開 → 添加 → 完成 → 關閉
異常路徑 1：打開 → 添加到一半被打斷 → 下次打開時能看到草稿嗎？
異常路徑 2：打開 → 看到 50 個未完成任務 → 會不會焦慮到關掉？
邊界情況：3 天沒用 → 再打開時顯示什麼？
```


## 五、靈魂三問的常見誤區

### 誤區 5.1：用戶描述太泛

**症狀**

你回答「用戶是誰」時，說的是：
> 「想提高效率的人」「有待辦需求的人」「年輕人」

**問題診斷**

這種描述等於沒有描述。「想提高效率的人」可能是學生、是職場人、是創業者，他們的需求差異巨大。

判斷標準：你能給這個用戶發微信嗎？如果不能，說明不夠具體。

**改進建議**

用這個檢驗標準：我能不能用一句話描述 TA 是誰、TA 在什麼場景下遇到什麼問題？

```
❌ 想提高效率的人
✅ 小李，25 歲職場新人，經常忘事被領導批評
```


### 誤區 5.2：痛點只有「想要」沒有「痛」

**症狀**

你描述的痛點是：
> 「用戶想要一個好用的待辦清單」「用戶想要更方便地記錄任務」

**問題診斷**

「想要 xxx」是需求，不是痛點。痛點應該帶有負面情緒：焦慮、煩躁、尷尬、害怕。

如果用戶只是「想要」而沒有「痛」，他可能不會真的去用你的解決方案——因爲現狀還過得去。

**改進建議**

檢驗痛點的三個標準：

| 標準 | 問題 |
|-----|------|
| 有負面情緒 | 用戶會因爲這個問題感到焦慮/煩躁/尷尬嗎？ |
| 正在發生 | 用戶這周遇到過這個問題嗎？ |
| 願意付出 | 如果有解決方案，用戶會立刻用嗎？ |

三個都滿足，纔是真痛點。


### 誤區 5.3：「爲什麼選我」只說「我更好」

**症狀**

你回答「爲什麼選我」時，說的是：
> 「因爲我的產品更好」「因爲我的設計更簡潔」

**問題診斷**

「更好」不是差異化。用戶爲什麼相信你「更好」？相比誰「更好」？好在哪裏？

**改進建議**

用這個句式回答：

> 「用戶現在用 [現有方案] 解決這個問題，但 [痛點]。我的方案是 [具體差異]，所以用戶應該選我。」

```
❌ 因爲我的產品更好
✅ 用戶現在用手機備忘錄記事，但經常忘記看。我的方案是「打開就是今日待辦」，3秒添加任務，所以更適合需要快速記錄的職場人。
```


## 六、場景應用的常見誤區

### 誤區 6.1：生搬硬套產品思維

**症狀**

你在做數據分析或自動化腳本時，也在糾結「用戶畫像」「MVP」「靈魂三問」。

**問題診斷**

思維模型是工具，不是教條。不同場景需要靈活調整：
- 做產品：需要完整的用戶畫像、MVP 規劃
- 做數據分析：核心是「給誰看、回答什麼問題」
- 做自動化腳本：核心是「什麼任務值得自動化」

**改進建議**

根據場景選擇重點：

| 場景 | 重點模型 | 次要模型 |
|-----|---------|---------|
| 產品/工具 | JTBD、減法、靈魂三問 | 全部 |
| 數據分析 | 靈魂三問（給誰看、回答什麼） | 逆向（什麼會讓分析出錯） |
| 自動化腳本 | 問題發現、逆向思維 | 減法（先自動化哪一步） |
| 給家人做 | 故事思維 | 減法（極簡設計） |


## 七、問題發現的常見誤區

### 誤區 7.1：煩惱日記只記不分析

**症狀**

你記了一週的煩惱日記，然後就放在那裏了。

**問題診斷**

記錄只是第一步。價值在於分析和篩選。

**改進建議**

第 8 天必須做分析：
1. 按頻率和痛苦程度打分
2. 用「問題篩選評分表」評估每個問題
3. 選出分數最高的 1-2 個問題，開始做 JTBD 分析


### 誤區 7.2：選了一個「不適合自動化」的問題

**症狀**

你選擇自動化的任務是：
- 涉及大量主觀判斷（如「幫我寫一份好的報告」）
- 涉及敏感數據（如「自動交易股票」）
- 錯了後果很嚴重（如「自動給客戶發合同」）

**問題診斷**

不是所有問題都適合用 Vibe Coding 解決。規則性低、敏感性高、容錯性低的任務，應該謹慎處理。

**改進建議**

用「問題篩選評分表」評估，總分低於 15 分的問題，要麼不做，要麼只做輔助（人工複覈）。


## 自檢清單總結

完成項目前，用這個清單快速自檢：

- [ ] 我用的是「任務描述」而非「功能清單」？
- [ ] 我考慮了功能、情感、社會三層任務？
- [ ] 每個風險都有對應的預防措施？
- [ ] MVP 的核心功能做到位了，而不是湊合？
- [ ] 不做清單是具體的功能名稱，而非空泛描述？
- [ ] 用戶畫像具體到「能發微信」的程度？
- [ ] 痛點帶有負面情緒，而非只是「想要」？
- [ ] 「爲什麼選我」說清楚了和現有方案的差異？

如果有任何一項打不了勾，回到對應章節重新思考。
