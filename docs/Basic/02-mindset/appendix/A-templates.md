# A. 思維模型速查表

本附錄彙總第二章七個思維模型的核心模板，每個模板都附有填寫示例和使用場景說明。


## 模板使用順序建議

```
問題發現 → JTBD → 逆向思維 → 減法思維 → 故事思維 → 靈魂三問 → 場景適配
    ↓         ↓         ↓           ↓           ↓           ↓
 找到問題   定義任務   預防失敗    聚焦MVP    理解用戶    最終自檢
```

不必每次都走完整流程。簡單項目可以跳過部分步驟，但**靈魂三問**是最後的底線檢查。


## 1. JTBD 任務描述模板

**適用場景**：當你有了一個想法，需要明確「用戶到底要完成什麼任務」時使用。

### 模板

```markdown
當 [用戶類型] 在 [具體場景] 時，
想要 [完成的任務]，
以便於 [獲得的結果或感受]。
```

### 填寫示例

```markdown
當 職場新人 在 早上到公司開始工作 時，
想要 快速記錄今天要做的事，
以便於 不會遺漏重要任務，可以安心投入工作。
```

### 進階：三層任務

| 任務層次 | 問題 | 示例 |
|---------|------|------|
| 功能任務 | 要完成什麼具體事情？ | 記錄和查看待辦事項 |
| 情感任務 | 想要什麼感受？ | 減少焦慮，感到安心 |
| 社會任務 | 想被別人如何看待？ | 在同事面前顯得靠譜 |


## 2. Pre-mortem 預演失敗模板

**適用場景**：在開始項目前，用逆向思維提前識別可能導致失敗的因素。

### 模板

```markdown
## Pre-mortem 分析

假設 [時間] 後項目失敗了，可能的原因是：

| 失敗原因 | 可能性(高/中/低) | 嚴重性(高/中/低) | 預防措施 |
|---------|----------------|----------------|---------|
| 1.      |                |                |         |
| 2.      |                |                |         |
| 3.      |                |                |         |

最需要警惕的風險（可能性高 + 嚴重性高）：
_______________

對應的具體行動：
_______________
```

### 填寫示例

```markdown
## Pre-mortem 分析

假設 3個月 後項目失敗了，可能的原因是：

| 失敗原因 | 可能性 | 嚴重性 | 預防措施 |
|---------|-------|-------|---------|
| 功能太多，做不完 | 高 | 高 | 第一版只做3個核心功能 |
| 用起來比便籤還麻煩 | 中 | 高 | 添加任務必須在3秒內完成 |
| 做完了自己都不用 | 中 | 高 | 先用一週紙質清單驗證習慣 |

最需要警惕的風險：功能太多，做不完

對應的具體行動：明確P0功能只有3個，其他全部放入「不做清單」
```


## 3. MVP 設計模板

**適用場景**：當你需要確定第一版要做什麼、不做什麼時使用。

### 模板

```markdown
## MVP 設計

**核心假設**：_______________
（用一句話描述你要驗證的最重要假設）

**驗證標準**：_______________
（什麼結果能證明假設成立？）

**P0 功能（必須有，3個以內）**：
1. _______________
2. _______________
3. _______________

**不做清單（明確不做的事情）**：
- 不做 _______________ ，因爲 _______________
- 不做 _______________ ，因爲 _______________
- 不做 _______________ ，因爲 _______________
```

### 填寫示例

```markdown
## MVP 設計

**核心假設**：一個極簡的每日待辦清單，比便籤紙和手機備忘錄更好用

**驗證標準**：自己連續使用7天，每天都用它記錄和完成任務

**P0 功能（必須有）**：
1. 添加任務
2. 完成任務（打勾）
3. 查看今日任務

**不做清單**：
- 不做 多設備同步 ，因爲 需要後端開發，大大增加複雜度
- 不做 任務分類標籤 ，因爲 不是極簡體驗的核心
- 不做 截止日期提醒 ，因爲 先驗證「記錄」這個核心價值
```


## 4. 用戶故事模板（三維畫像）

**適用場景**：當你需要清晰描述「用戶是誰」時使用。

### 模板

```markdown
## 用戶畫像

**用戶名字**：_______________（給TA起個名字）

**一句話描述**：_______________

| 維度 | 內容 |
|-----|------|
| 表面屬性 | 年齡、職業、每天處理多少事務 |
| 行爲習慣 | 日常使用什麼工具、什麼時候用、怎麼用 |
| 深層動機 | 害怕什麼、追求什麼、想成爲什麼樣的人 |

**使用場景**：
- 場景1：_______________
- 場景2：_______________
- 場景3：_______________
```

### 填寫示例

```markdown
## 用戶畫像

**用戶名字**：小李

**一句話描述**：25歲職場新人，怕遺漏任務被領導批評

| 維度 | 內容 |
|-----|------|
| 表面屬性 | 25歲，互聯網公司運營，每天處理10-15件事務 |
| 行爲習慣 | 早上到公司先看郵件，用手機備忘錄記事但經常忘記看 |
| 深層動機 | 害怕遺漏任務被批評，想成爲一個靠譜的人 |

**使用場景**：
- 場景1：早上到公司，打開電腦，快速記錄今天要做的事
- 場景2：工作中突然想起一件事，需要3秒內記下來
- 場景3：下班前，檢查今天的事情是否都完成了
```


## 5. 靈魂三問自檢模板

**適用場景**：在開始動手前，做最後一次自檢。三個綠燈才能開始。

### 模板

```markdown
## 靈魂三問自檢

**項目名稱**：_______________

| 問題 | 回答 | 狀態 |
|-----|------|------|
| 用戶是誰？ | _______________ | 🟢/🟡/🔴 |
| 痛點在哪？ | _______________ | 🟢/🟡/🔴 |
| 爲什麼選我？ | _______________ | 🟢/🟡/🔴 |

**狀態說明**：
- 🟢 綠燈：能用一句話清晰回答
- 🟡 黃燈：有想法但說不清楚
- 🔴 紅燈：完全回答不出來

**結論**：□ 可以開始（全綠） □ 需要完善（有黃） □ 先停下來（有紅）
```

### 填寫示例

```markdown
## 靈魂三問自檢

**項目名稱**：極簡待辦清單

| 問題 | 回答 | 狀態 |
|-----|------|------|
| 用戶是誰？ | 小李——25歲職場新人，怕遺漏任務 | 🟢 |
| 痛點在哪？ | 經常忘事被批評，現有工具要麼太複雜要麼容易忘記看 | 🟢 |
| 爲什麼選我？ | 打開就是今日待辦，3秒添加任務，比便籤和備忘錄更順手 | 🟢 |

**結論**：☑ 可以開始（全綠）
```


## 6. 煩惱日記模板

**適用場景**：當你不知道做什麼項目時，用這個方法發現值得解決的問題。

### 模板

```markdown
## 煩惱日記

**日期**：_______________

**煩惱瞬間**：_______________（一句話描述發生了什麼）

**當時在做什麼**：_______________

**爲什麼煩**：□ 太慢 □ 太麻煩 □ 容易出錯 □ 其他：___

**我最後怎麼解決的**：□ 忍着做完 □ 找人幫忙 □ 放棄了 □ 其他：___

**這件事多久發生一次**：□ 每天 □ 每週 □ 每月 □ 偶爾
```

### 填寫示例

```markdown
## 煩惱日記

**日期**：週三

**煩惱瞬間**：又要從5個Excel裏彙總數據做週報

**當時在做什麼**：準備週五的部門彙報

**爲什麼煩**：☑ 太麻煩（每次都要打開5個文件，複製粘貼，檢查有沒有錯）

**我最後怎麼解決的**：☑ 忍着做完（花了40分鐘）

**這件事多久發生一次**：☑ 每週
```


## 7. 問題篩選評分表

**適用場景**：當你有多個問題/想法時，用這個表格篩選出最適合 Vibe Coding 解決的問題。

### 模板

```markdown
## 問題篩選評分表

| 問題 | 重複性 | 規則性 | 可驗證 | 不敏感 | 容錯高 | 總分 |
|-----|-------|-------|-------|-------|-------|-----|
|     | /5    | /5    | /5    | /5    | /5    | /25 |

**評分標準**：
- 重複性：這個問題多久發生一次？（每天5分，每週4分，每月2分，偶爾1分）
- 規則性：能用清晰的規則描述嗎？（完全能5分，大部分能3分，說不清1分）
- 可驗證：自己用就能驗證效果嗎？（自己用5分，需要別人3分，很難驗證1分）
- 不敏感：不涉及隱私和資金嗎？（完全不涉及5分，略有涉及3分，高度敏感1分）
- 容錯高：錯了容易發現和修復嗎？（很容易5分，一般3分，錯了麻煩大1分）

**建議**：總分 ≥ 18 分的問題適合用 Vibe Coding 解決
```

### 填寫示例

```markdown
## 問題篩選評分表

| 問題 | 重複性 | 規則性 | 可驗證 | 不敏感 | 容錯高 | 總分 |
|-----|-------|-------|-------|-------|-------|-----|
| 待辦遺漏 | 5 | 4 | 5 | 5 | 4 | 23 ✓ |
| Excel彙總 | 4 | 5 | 5 | 4 | 4 | 22 ✓ |
| 投資決策 | 2 | 2 | 1 | 2 | 1 | 8 ✗ |

結論：「待辦遺漏」和「Excel彙總」適合做，「投資決策」不適合
```


## 8. 場景化需求模板

```markdown
3. _______________
```



## 快速查找指南

| 你的問題 | 使用哪個模板 |
|---------|-------------|
| 不知道做什麼項目 | 煩惱日記 + 問題篩選評分表 |
| 想清楚用戶要完成什麼 | JTBD 任務描述模板 |
| 擔心項目會失敗 | Pre-mortem 模板 |
| 不知道第一版做什麼 | MVP 設計模板 |
| 不清楚用戶是誰 | 用戶故事模板 |
| 不確定是否準備好開始 | 靈魂三問自檢模板 |
| 要做數據分析 | 數據分析需求模板 |
| 要做自動化腳本 | 自動化腳本需求模板 |
