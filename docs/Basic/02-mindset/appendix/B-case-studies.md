# B. 擴展案例庫

本附錄提供 6 個真實案例，展示如何在不同場景下應用第二章的思維模型。每個案例都包含：問題背景、思維模型應用過程、關鍵決策、最終結果。

## 案例 1：Dropbox —— 用視頻代替產品

**場景類型**：產品驗證  
**核心思維模型**：減法思維

### 背景

2007 年，Drew Houston 有一個想法：做一個能在多臺電腦之間自動同步文件的工具。

但他面臨一個難題：這個產品需要開發複雜的後端系統、處理各種邊界情況，開發週期可能需要一年以上。如果花一年時間做出來，結果沒人用怎麼辦？

### 思維模型應用：減法思維

Houston 問自己一個關鍵問題：**我真正要驗證的假設是什麼？**

| 常見做法 | Houston 的做法 |
|---------|---------------|
| 先做一個能用的產品 | 先驗證「有沒有人需要這個」 |
| 功能越完善越好 | 只需要讓人「理解並想要」 |
| 產品說話 | 視頻說話 |

他的核心假設是：**有足夠多的人遇到「多設備文件同步」這個痛點，並且願意爲解決方案付費/註冊。**

驗證這個假設，並不需要一個完整的產品。一個能讓人理解「這是什麼、能解決什麼問題」的視頻就夠了。

### 關鍵決策

Houston 做了一個 3 分鐘的演示視頻，展示 Dropbox 的使用場景：
- 在一臺電腦上修改文件
- 打開另一臺電腦，文件自動同步了
- 不需要 U 盤、不需要發郵件給自己

視頻發佈在 Hacker News 上，配合一個註冊等待列表頁面。

### 結果

- 一夜之間，等待列表從 5,000 人增長到 75,000 人
- Houston 得到了明確的驗證：這個問題值得解決，有大量人願意使用
- 基於這個驗證結果，他開始認真開發產品

### 啓示

**減法思維的核心不是「少做功能」，而是「想清楚要驗證什麼」。**

Dropbox 的 MVP 不是一個「功能很少的產品」，而是一個「能驗證核心假設的最小方案」。在這個案例中，驗證假設的最小方案是一個視頻，不是產品。


## 案例 2：Airbnb —— 從氣墊牀到獨角獸

**場景類型**：產品驗證  
**核心思維模型**：JTBD 思維

### 背景

2007 年，Brian Chesky 和 Joe Gebbia 付不起舊金山的房租。當時城裏正在舉辦一個設計大會，酒店全部訂滿。他們想到一個主意：在客廳放幾個氣墊牀，提供早餐，出租給參會者。

這就是 Airbnb 的起源——AirBed & Breakfast（氣墊牀和早餐）。

### 思維模型應用：JTBD 思維

最初，他們以爲自己在解決的問題是「便宜住宿」。

但通過和第一批房客交流，他們發現了更深層的任務：

| 表面需求 | 深層任務（JTBD） |
|---------|----------------|
| 我需要便宜的住宿 | 我想像當地人一樣體驗這個城市 |
| 我需要一張牀 | 我想要比酒店更有人情味的住宿體驗 |
| 我想省錢 | 我想把錢花在體驗上，而不是標準化的酒店房間 |

用 JTBD 模板描述：

> 當 旅行者 在 去一個新城市參加活動 時，  
> 想要 像當地人一樣住在這個城市，  
> 以便於 獲得獨特的、有人情味的旅行體驗。

### 關鍵決策

基於這個洞察，Airbnb 的定位從「便宜住宿」轉變爲「像當地人一樣生活」。

這影響了後續的產品設計：
- 鼓勵房東提供當地生活建議
- 展示房源的獨特個性，而非標準化配置
- 強調「體驗」而非「住宿」

### 結果

Airbnb 成爲估值超過 1000 億美元的公司。他們的成功不是因爲提供了更便宜的住宿，而是因爲理解了用戶真正要完成的任務。

### 啓示

**JTBD 思維幫你看到功能背後的任務。**

如果 Airbnb 只停留在「便宜住宿」這個表面需求，他們可能會陷入和廉價酒店的價格戰。正是因爲理解了「像當地人一樣生活」這個深層任務，他們開闢了一個全新的市場。


## 案例 3：電商渠道 ROI 分析

**場景類型**：數據分析  
**核心思維模型**：靈魂三問

### 背景

小王是一家電商公司的運營。老闆讓他「分析一下各渠道的投放效果」。

他打開 Excel，面對一堆數據，不知道從何下手：
- 要分析哪些指標？
- 要做什麼圖表？
- 報告應該多長？

### 思維模型應用：靈魂三問

小王決定先用「靈魂三問」理清思路：

**問題 1：用戶是誰？**

這份報告的「用戶」是老闆。老闆關心的是「錢花得值不值」，不是「數據有多詳細」。

**問題 2：痛點在哪？**

老闆的痛點是：每個月在 5 個渠道投了 50 萬廣告費，但不知道哪個渠道效果最好，下個月預算該怎麼分配。

**問題 3：爲什麼選我？**

老闆選擇讓小王做這個分析，是因爲小王能接觸到各渠道的數據。老闆需要的是「能支持決策的結論」，而不是「展示數據的報告」。

### 關鍵決策

基於靈魂三問，小王明確了分析方向：

| 之前的想法 | 調整後的方向 |
|-----------|-------------|
| 分析所有能分析的指標 | 只分析 ROI 相關指標 |
| 做各種漂亮的圖表 | 只做能回答「哪個渠道效果最好」的圖 |
| 報告越詳細越好 | 一頁 PPT 說清楚結論 |

他的核心交付變成了：

> 「基於 Q3 數據，渠道 A 的 ROI 是 3.2，渠道 C 是 1.1。建議下季度將渠道 C 的預算轉移 30% 到渠道 A。」

### 結果

老闆看完報告，5 分鐘內做出了預算調整決策。老闆評價：「終於有人能告訴我該怎麼做了，而不是扔給我一堆數據。」

### 啓示

**數據分析的價值不在於「分析了多少」，而在於「回答了什麼問題」。**

在做數據分析前，先問自己靈魂三問：
- 這份報告給誰看？
- 他需要回答什麼問題？
- 他看完後應該能做什麼決策？


## 案例 4：銷售週報自動化

**場景類型**：數據分析/自動化  
**核心思維模型**：逆向思維

### 背景

小張每週五要從 5 個部門的 Excel 文件中彙總數據，生成一份銷售週報。這個過程需要 2 小時，而且經常出錯。

他決定用 Vibe Coding 做一個自動化工具。

### 思維模型應用：逆向思維

在動手之前，小張做了一個 Pre-mortem 分析：

> 假設 1 個月後，這個自動化工具失敗了，可能是什麼原因？

| 失敗原因 | 可能性 | 預防措施 |
|---------|-------|---------|
| 部門改了 Excel 格式，腳本跑不了 | 高 | 加入格式檢測，發現異常時報警而非靜默失敗 |
| 彙總後的數據和手動彙總不一致 | 中 | 前幾周自動化和手動並行，對比驗證 |
| 忘了新增的部門沒有加入腳本 | 中 | 用文件夾掃描代替硬編碼文件名 |
| 腳本報錯了不知道怎麼修 | 中 | 讓 AI 寫詳細的錯誤提示和日誌 |

### 關鍵決策

基於 Pre-mortem 分析，小張在給 AI 的 Prompt 中加入了這些要求：

```
請幫我寫一個 Python 腳本，自動彙總多個 Excel 文件。

特別要求：
1. 自動掃描指定文件夾下的所有 .xlsx 文件
2. 檢測每個文件的格式是否符合預期，不符合時明確報錯
3. 彙總完成後，輸出一份校驗摘要（總行數、各部門數據條數）
4. 每一步操作都寫日誌，方便排查問題
```

### 結果

第一週，腳本在彙總財務部數據時報錯——因爲財務部的表頭多了一列。因爲有格式檢測和明確的錯誤提示，小張 5 分鐘就定位並修復了問題。

如果沒有 Pre-mortem 分析，這個問題可能會導致彙總數據錯誤，直到老闆發現才被追查。

### 啓示

**逆向思維不是悲觀主義，而是「提前踩坑」。**

對於自動化腳本來說，最危險的不是「跑不起來」，而是「跑起來了但結果不對」。Pre-mortem 幫你提前想到這些風險，並在設計時就加入防護措施。


## 案例 5：喫藥提醒給父母

**場景類型**：個人工具/給家人做  
**核心思維模型**：故事思維

### 背景

小李的父母都 60 多歲了，需要每天定時喫降壓藥。但他們經常忘記，或者喫完了忘記自己喫沒喫過。

小李想用 Vibe Coding 做一個喫藥提醒工具。

### 思維模型應用：故事思維

小李沒有直接開始設計功能，而是先用「三維畫像」理解父母：

| 維度 | 內容 |
|-----|------|
| 表面屬性 | 60+ 歲，視力不太好，不太會用智能手機 |
| 行爲習慣 | 早上起牀後先看電視，手機主要用來接電話和微信 |
| 深層動機 | 不想給子女添麻煩，但又怕忘記喫藥影響健康 |

然後，他用「用戶旅程」想象父母使用這個工具的場景：

```
早上 7:00
├── 場景：爸媽剛起牀，在客廳
├── 觸發：手機響了/震動了
├── 行動：看手機，看到喫藥提醒
├── 障礙：字太小看不清？不知道點哪裏確認？
└── 期望感受：「哦，該喫藥了」，點一下就完成
```

### 關鍵決策

基於故事思維的分析，小李確定了設計原則：

| 常見做法 | 小李的做法 |
|---------|-----------|
| 多種提醒方式可選 | 只用最簡單的方式：大字 + 震動 |
| 記錄喫藥歷史 | 只問「喫了嗎」，點「喫了」就完成 |
| 可以設置多種藥物 | 先只支持一種藥，降低複雜度 |
| 精美的 UI 設計 | 大按鈕、大字體、高對比度 |

給 AI 的 Prompt：

```
我要給 60 多歲的父母做一個喫藥提醒網頁。

用戶特點：
- 視力不太好，需要大字體（至少 24px）
- 不太會用手機，交互要極簡
- 不需要記錄歷史，只需要提醒和確認

功能要求：
- 顯示當前時間和「該喫藥了」提示
- 一個巨大的「我喫了」按鈕
- 點擊後顯示「好的，明天同一時間再提醒你」
- 頁面可以設爲手機瀏覽器主頁
```

### 結果

父母真的開始用這個工具了。媽媽說：「這個比你之前教我用的那些 App 簡單多了，我一看就懂。」

### 啓示

**給別人做工具時，「故事思維」幫你站在對方角度思考。**

如果小李直接問父母「你想要什麼功能」，他們可能說不清楚。但通過想象他們的一天、他們使用手機的場景、他們可能遇到的障礙，小李設計出了真正適合他們的工具。


## 案例 6：Excel 多表彙總

**場景類型**：自動化  
**核心思維模型**：問題發現

### 背景

小陳是公司的人事專員。每個月初，她需要從各部門收集考勤表，彙總成一份公司考勤總表。

這件事她做了兩年，每次都覺得煩，但從沒想過可以自動化。

### 思維模型應用：問題發現

某天，小陳開始記「煩惱日記」。一週後，她回顧記錄：

```markdown
## 煩惱日記彙總

| 日期 | 煩惱 | 頻率 | 痛苦程度 |
|-----|------|-----|---------|
| 週一 | 考勤彙總做了3小時 | 每月1次 | 9分 |
| 週二 | 有部門的表格格式又不對 | 每月1次 | 7分 |
| 週四 | 彙總完發現數據對不上，重新做 | 每月常有 | 10分 |
```

她用「問題篩選評分表」分析這個問題：

| 維度 | 分數 | 理由 |
|-----|------|------|
| 重複性 | 4 | 每月一次，固定發生 |
| 規則性 | 5 | 規則很清晰：從指定列提取數據，合併到一張表 |
| 可驗證 | 5 | 自己用就能驗證，對比手動結果 |
| 不敏感 | 4 | 是內部數據，但不涉及薪資等核心敏感信息 |
| 容錯高 | 4 | 彙總後會人工複覈，錯了能發現 |
| **總分** | **22** | 非常適合自動化 |

### 關鍵決策

小陳決定用 Vibe Coding 解決這個問題。她用「自動化腳本需求模板」整理需求：

```markdown
**我的重複性任務**：每月彙總各部門考勤表

**目前我怎麼做的**：
1. 打開各部門發來的 Excel 文件（20分鐘找齊文件）
2. 複製每個文件的 B-F 列數據（60分鐘）
3. 粘貼到彙總表，檢查格式（40分鐘）
4. 覈對總人數是否一致（20分鐘）

**我希望自動化的部分**：步驟 2 和步驟 3

**輸入**：一個文件夾，裏面是各部門的 .xlsx 文件

**輸出**：一個彙總後的 .xlsx 文件

**可能出錯的情況**：
- 有部門的表格列順序不對
- 有部門的表格多了或少了幾列
- 文件名格式不統一
```

### 結果

她用 AI 生成了一個 Python 腳本，現在每月的考勤彙總從 3 小時縮短到 10 分鐘（主要是等腳本跑完和人工複覈）。

更重要的是，因爲腳本會檢測格式異常，她再也不用擔心「彙總完才發現數據對不上」了。

### 啓示

**很多值得自動化的問題，我們每天都在「忍着」。**

煩惱日記的價值不在於記錄本身，而在於讓你「看見」那些習以爲常的痛點。當你把它們寫下來、打分、排序，你就會發現：原來我每個月都在一件事上浪費 3 小時，而這件事完全可以交給 AI。


## 案例總結

| 案例 | 核心啓示 |
|-----|---------|
| Dropbox | 驗證假設的最小方案，可能不是產品本身 |
| Airbnb | 看到功能背後的任務，才能找到真正的市場 |
| 電商 ROI 分析 | 數據分析的價值在於支持決策，不在於展示數據 |
| 銷售週報自動化 | 提前想失敗原因，在設計時就加入防護 |
| 喫藥提醒 | 給別人做工具，要站在對方角度想象使用場景 |
| Excel 彙總 | 煩惱日記幫你發現值得自動化的問題 |

這些案例的共同點是：**在動手之前，先用思維模型想清楚問題。**

這就是第二章「心法」的核心價值——它不會幫你寫代碼，但會幫你少走彎路。
