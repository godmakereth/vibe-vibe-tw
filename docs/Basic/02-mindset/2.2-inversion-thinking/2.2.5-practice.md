# 2.2.5 實戰練習：爲你的項目做 Pre-mortem

現在輪到你了。

無論你想做的是一個小工具、一份數據分析、一個自動化腳本，還是給家人做的小網頁，都可以用這個模板來做一次 Pre-mortem。


## Pre-mortem 完整模板

以下模板可以直接複製使用：

```markdown
# Pre-mortem 分析：[你的項目名稱]

## 第一步：失敗場景設定

現在是 [3個月/6個月] 後，我的 [項目名稱] 已經徹底失敗了。

具體表現是：
- [描述失敗的具體狀態，比如"我已經不再使用它"]
- [描述你回到了什麼狀態，比如"又回到了用Excel手動處理的日子"]


## 第二步：失敗原因列舉

這個項目失敗是因爲：

1. ________________________________
2. ________________________________
3. ________________________________
4. ________________________________
5. ________________________________
（至少寫5條，越多越好）


## 第三步：分類與評估

| 失敗原因 | 類型 | 可能性 | 嚴重性 | 優先級 |
|---------|------|-------|-------|-------|
| 原因1 | 需求/技術/場景/習慣 | 高/中/低 | 高/中/低 | ⚠️/⚡/✓ |
| 原因2 | | | | |
| 原因3 | | | | |
| ... | | | | |

優先級說明：
- ⚠️ 必須解決：高可能性 + 高嚴重性
- ⚡ 應該解決：高可能性 或 高嚴重性
- ✓ 可以接受：低可能性 + 低嚴重性


## 第四步：預防措施

| 失敗原因（⚠️和⚡的） | 預防措施 | 如何落實到需求 |
|---------------------|---------|---------------|
| | | |
| | | |
| | | |


## 第五步：更新後的需求描述

基於以上分析，我的項目需求更新爲：

**核心功能（必須有）：**
1. 
2. 
3. 

**保障性功能（防止失敗）：**
1. 
2. 

**明確不做的功能：**
1. 
2. 
```


## 填寫示例：待辦清單 App

```markdown
# Pre-mortem 分析：我的待辦清單 App

## 第一步：失敗場景設定

現在是 3個月 後，我的 待辦清單App 已經徹底失敗了。

具體表現是：
- 我已經兩個月沒打開過這個App
- 又回到了用微信給自己發消息記事的狀態


## 第二步：失敗原因列舉

這個項目失敗是因爲：

1. 功能太多，界面太複雜，每次打開都不知道該點哪裏
2. 添加任務太麻煩，要填太多字段，不如直接發微信快
3. 沒有提醒功能，總是忘記打開App查看任務
4. 有一次數據丟失後，我就不再信任它了
5. 和我的工作流不匹配，待辦事項來源太分散
6. 新鮮感過後，沒有動力繼續使用


## 第三步：分類與評估

| 失敗原因 | 類型 | 可能性 | 嚴重性 | 優先級 |
|---------|------|-------|-------|-------|
| 功能太多太複雜 | 需求 | 高 | 高 | ⚠️ |
| 添加任務太麻煩 | 場景 | 高 | 高 | ⚠️ |
| 沒有提醒功能 | 需求 | 高 | 中 | ⚡ |
| 數據丟失 | 技術 | 中 | 高 | ⚡ |
| 和工作流不匹配 | 場景 | 中 | 中 | ✓ |
| 新鮮感消退 | 習慣 | 高 | 中 | ⚡ |


## 第四步：預防措施

| 失敗原因 | 預防措施 | 落實到需求 |
|---------|---------|-----------|
| 功能太多太複雜 | 第一版只做3個功能 | 只做：添加、查看、完成 |
| 添加任務太麻煩 | 一鍵添加 | 只需輸入任務標題，其他都是可選 |
| 沒有提醒功能 | 每日定時提醒 | 每天早上9點推送今日任務 |
| 數據丟失 | 雲端存儲 | 使用雲數據庫，不存本地 |
| 新鮮感消退 | 設計使用激勵 | 完成任務時有小動畫反饋 |


## 第五步：更新後的需求描述

**核心功能（必須有）：**
1. 快速添加任務（只需輸入標題）
2. 查看任務列表
3. 標記任務完成

**保障性功能（防止失敗）：**
1. 每日提醒
2. 雲端數據同步

**明確不做的功能：**
1. 任務分類和標籤（第一版不做）
2. 子任務拆解（第一版不做）
3. 統計報表（第一版不做）
```


## 讓 AI 幫你做 Pre-mortem

如果你不知道從哪裏開始，可以用這個 Prompt 讓 AI 幫你做分析：

```
我想做一個 [簡要描述你的項目]。

請幫我做一個 Pre-mortem 分析：
1. 假設這個項目 3 個月後失敗了，請列出 10 個可能的失敗原因
2. 對每個原因評估可能性（高/中/低）和嚴重性（高/中/低）
3. 對於高可能性或高嚴重性的原因，給出具體的預防措施
4. 最後，基於這個分析，告訴我第一版應該做什麼、不應該做什麼

背景信息：
- 我的目標用戶是：[描述]
- 我想解決的核心問題是：[描述]
- 我計劃使用的工具是：[比如 Cursor、Bolt.new 等]
```

**使用示例：**

```
我想做一個幫助我媽媽記錄喫藥時間的小工具。

請幫我做一個 Pre-mortem 分析：
1. 假設這個項目 3 個月後失敗了，請列出 10 個可能的失敗原因
2. 對每個原因評估可能性（高/中/低）和嚴重性（高/中/低）
3. 對於高可能性或高嚴重性的原因，給出具體的預防措施
4. 最後，基於這個分析，告訴我第一版應該做什麼、不應該做什麼

背景信息：
- 我的目標用戶是：我60歲的媽媽，不太會用智能手機
- 我想解決的核心問題是：她經常忘記喫藥，或者忘記自己喫過沒有
- 我計劃使用的工具是：Bolt.new
```


## 不同場景的 Pre-mortem 問題清單

根據你想做的事情類型，重點關注不同的失敗風險：

### 如果你在做小工具/產品原型

- [ ] 我第一版要的功能是否超過 5 個？
- [ ] 核心功能的使用流程是否超過 3 步？
- [ ] 是否考慮了用戶的真實使用場景和時機？

### 如果你在做數據分析

- [ ] 我要回答的業務問題是否清晰？
- [ ] 數據源是否可靠？是否有缺失值、異常值？
- [ ] 分析結論是否能指導具體行動？

### 如果你在做自動化腳本

- [ ] 是否考慮了各種邊界情況（空值、特殊字符、超大文件）？
- [ ] 腳本失敗時是否有錯誤提示和恢復機制？
- [ ] 是否依賴特定的運行環境？換臺電腦還能跑嗎？

### 如果你在給家人做工具

- [ ] 他們的視力、手指靈活度是否與年輕人不同？
- [ ] 他們是否理解你認爲"常識"的交互方式（如滑動刪除）？
- [ ] 操作步驟是否足夠少（最好不超過 2 步）？


完成 Pre-mortem 後，你會發現自己的需求變得更加清晰和聚焦。

這正是下一節"減法思維"要講的內容——如何從一堆想法中，提煉出真正重要的核心。
