# 3.2.2 S.C.A.F.F. 框架（推薦框架）

## 經過本節學習，你將掌握

- S.C.A.F.F. 框架的五個要素及其含義
- 如何用這個框架描述一個技術開發任務
- 框架的精簡版，適合日常快速使用
- 獲得可直接複製的提示詞模板


## 框架結構

S.C.A.F.F. 是五個英文單詞的首字母縮寫：

| 字母 | 英文 | 中文 | 核心問題 |
|-----|------|------|---------|
| **S** | Situation | 情境 | 你在做什麼項目？當前狀態如何？ |
| **C** | Challenge | 挑戰 | 當前要解決的具體問題是什麼？ |
| **A** | Audience | 受衆 | 代碼/產出給誰用？誰來維護？ |
| **F** | Format | 格式 | 期望 AI 輸出什麼格式？ |
| **F** | Foundations | 基礎 | 必須遵守的約束和原則是什麼？ |

這個框架的設計邏輯是：**先給背景，再說問題，然後明確對象，最後約束邊界**。


## 五要素詳解

### S - Situation（情境）

回答：**你在做什麼？當前進展如何？**

填寫內容：
- 項目類型和目標
- 使用的技術棧
- 當前的開發階段
- 已有的代碼結構或設計決策

**填寫技巧**：想象你在向一個剛加入項目的同事介紹情況。

### C - Challenge（挑戰）

回答：**當前具體要解決什麼問題？**

填寫內容：
- 要實現的功能或要修復的問題
- 輸入是什麼，輸出是什麼
- 具體的業務邏輯

**填寫技巧**：一句話說清楚"做完這件事，什麼狀態變成什麼狀態"。

### A - Audience（受衆）

回答：**產出給誰用？**

填寫內容：
- 代碼的使用者是誰（用戶？其他開發者？）
- 代碼的維護者是誰（你自己？團隊？）
- 使用者/維護者的技術水平

**填寫技巧**：這決定了代碼複雜度和註釋詳細程度。

### F - Format（格式）

回答：**期望 AI 輸出什麼形式的結果？**

填寫內容：
- 完整代碼還是代碼片段
- 需要什麼文件格式
- 是否需要註釋
- 是否需要使用說明

**填寫技巧**：越具體越好。"給我代碼"不如"給我一個完整的 React 組件文件，包含 TypeScript 類型定義"。

### F - Foundations（基礎）

回答：**有什麼必須遵守的約束？**

填寫內容：
- 技術約束（必須/不能用某個庫）
- 性能約束（響應時間、內存限制）
- 風格約束（命名規範、代碼風格）
- 安全約束（敏感數據處理）

**填寫技巧**：用第二章的"不做清單"思維——明確告訴 AI 什麼不能做。


## 完整示例：登錄頁面

來看一個具體案例：你需要 AI 幫你實現一個登錄頁面。

### ❌ 差提示詞

```
幫我做一個登錄頁面
```

**問題**：AI 不知道用什麼技術、什麼風格、要哪些功能、有什麼限制。結果大概率是一個"通用模板"，和你的項目格格不入。

### ✅ 用 S.C.A.F.F. 框架改寫

```markdown
【Situation - 情境】
我正在開發一個個人博客系統的管理後臺。
技術棧：Next.js 16 + TypeScript + Tailwind CSS
當前進度：首頁和文章列表頁已完成，現在需要添加登錄功能。
設計風格：極簡風格，參考 Notion 的登錄頁面。

【Challenge - 挑戰】
實現管理員登錄頁面：
- 表單包含：郵箱輸入框、密碼輸入框、登錄按鈕
- 點擊登錄後，驗證格式（郵箱格式、密碼不爲空）
- 格式錯誤時，在對應輸入框下方顯示紅色提示
- 暫時不需要真正的後端驗證，先做前端界面和交互

【Audience - 受衆】
- 用戶：只有我一個人（博客管理員）
- 代碼維護者：我自己，React 初學者
- 請在關鍵邏輯處添加註釋，幫助我理解

【Format - 格式】
請提供：
1. 完整的登錄頁面組件代碼（單文件）
2. 包含 TypeScript 類型定義
3. 樣式使用 Tailwind CSS
4. 文件命名：LoginPage.tsx

【Foundations - 基礎約束】
- 不使用任何 UI 組件庫（如 shadcn/ui、Ant Design）
- 不使用第三方表單庫（如 React Hook Form）
- 狀態管理用 React 原生的 useState
- 響應式設計：移動端和桌面端都要適配
```

### 輸出差異對比

| 方面 | 差提示詞的可能輸出 | S.C.A.F.F. 的可能輸出 |
|-----|------------------|---------------------|
| 技術棧 | 可能是純 HTML 或 Vue | 確定是 Next.js + TS + Tailwind |
| 功能範圍 | 可能包含註冊、忘記密碼 | 只有登錄，符合當前需求 |
| 代碼複雜度 | 可能用了你不熟悉的庫 | 用 useState，初學者友好 |
| 註釋 | 可能沒有 | 關鍵邏輯有註釋 |
| 風格 | 隨機風格 | 極簡風格，參考 Notion |


## 待辦清單案例：添加任務功能

延續第二章小李的項目，用 S.C.A.F.F. 框架描述"添加任務"功能：

```markdown
【Situation - 情境】
我在做一個個人待辦清單網頁應用。
技術棧：React + TypeScript + Tailwind CSS
目標用戶：像我一樣容易忘事的職場新人
設計原則：極簡，打開就是今日待辦，添加任務 3 秒內完成
當前進度：項目剛初始化，還沒有任何功能代碼

【Challenge - 挑戰】
實現"添加任務"功能：
- 用戶在輸入框輸入任務內容
- 按回車鍵或點擊添加按鈕，任務添加到列表
- 添加成功後自動清空輸入框並聚焦
- 空內容不能添加，輸入框輕微抖動提示

【Audience - 受衆】
- 用戶：我自己，希望操作越簡單越好
- 維護者：我自己，React 初學者
- 請解釋爲什麼用這種實現方式

【Format - 格式】
請提供：
1. AddTask 組件代碼（處理輸入和添加）
2. TaskList 組件代碼（展示任務列表）
3. 簡單的組件使用說明

【Foundations - 基礎約束】
- 任務數據暫存在組件 state，不需要持久化
- 不用任何狀態管理庫（Redux、Zustand 等）
- 樣式用 Tailwind，保持簡潔
- 每個任務只需要：內容文本 + 完成狀態
```


## 精簡版：S.C.F. 三要素

日常快速使用時，可以只保留三個核心要素：

| 要素 | 必填原因 |
|-----|---------|
| **S** - Situation | 沒有背景，AI 只能猜 |
| **C** - Challenge | 這是你要解決的問題 |
| **F** - Foundations | 約束決定了輸出邊界 |

**精簡版模板**：

```markdown
【背景】
[項目類型] + [技術棧] + [當前狀態]

【任務】
[具體要做什麼]

【約束】
- [必須遵守的限制1]
- [必須遵守的限制2]
```

**精簡版示例**：

```markdown
【背景】
React + TS 的待辦清單項目，已完成基礎結構

【任務】
實現刪除任務功能：點擊任務右側的刪除圖標，該任務從列表中移除

【約束】
- 刪除前顯示確認提示
- 用 Tailwind CSS
- 刪除動畫：淡出效果
```

## 常見填寫誤區

| 誤區 | 問題 | 正確做法 |
|-----|------|---------|
| Situation 寫太長 | AI 會迷失在細節中 | 只寫與當前任務相關的背景 |
| Challenge 不具體 | "優化代碼"太模糊 | 說清楚優化什麼方面 |
| 忘記寫 Audience | AI 不知道代碼給誰看 | 初學者要求註釋，高手可以省略 |
| Format 只寫"給我代碼" | 輸出格式隨機 | 指定文件類型、是否要註釋 |
| Foundations 只寫"要做什麼" | AI 可能"加戲" | 同時寫清楚"不要什麼" |
