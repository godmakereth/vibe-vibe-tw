# 3.2.1 爲什麼需要框架

## 經過本節學習，你將掌握

- 理解框架的核心價值：減少思考負擔，提高輸出穩定性
- 明確框架不是限制，而是起點
- 知道什麼時候用框架，什麼時候靈活發揮
- 理解故事化 Prompt 與結構化框架的適用場景差異


## 一個真實的困擾

小李學完 3.1 節後，知道了要給 AI 提供上下文。但每次寫提示詞時，他還是會陷入這樣的狀態：

> "我知道要告訴 AI 項目背景...等等，還有什麼來着？技術棧、約束條件、輸出格式...每次都要從頭想一遍，好累。"

這不是小李的問題，而是**認知負擔**的問題。

人腦的工作記憶容量有限。當你同時要思考"說什麼"和"怎麼說"時，很容易顧此失彼——要麼遺漏關鍵信息，要麼表述混亂。

框架的價值，就是**把"怎麼說"變成固定格式，讓你專注於"說什麼"**。


## 框架的三個核心價值

### 1. 減少思考負擔

沒有框架時，你需要每次從零思考：
- 該說什麼？
- 先說什麼後說什麼？
- 有沒有遺漏？

有了框架後，你只需要**填空**：
- Situation（情境）→ 填入項目背景
- Challenge（挑戰）→ 填入當前問題
- ...以此類推

框架把"開放式問題"變成了"填空題"，認知負擔大幅降低。

### 2. 提高輸出穩定性

斯坦福大學人機交互研究院的研究表明：使用結構化框架的提示詞，任務完成準確率比隨機提問高出 30% 以上。

原因很簡單：框架確保了**關鍵信息不遺漏**。就像飛行員起飛前的檢查清單，不是因爲他們記不住，而是因爲清單能避免人爲疏忽。

### 3. 降低溝通成本

當你和 AI 建立了固定的"溝通協議"，AI 能更快理解你的意圖。

這就像和老同事合作：你們有默契的溝通方式，不需要每次都從頭解釋。框架就是你和 AI 之間的"默契"。


## 框架不是限制，而是起點

你可能會擔心：用框架會不會太死板？

答案是：**框架是起點，不是終點**。

| 階段 | 做法 |
|-----|------|
| 初學階段 | 嚴格按框架填寫，確保不遺漏 |
| 熟練階段 | 根據任務複雜度簡化或擴展 |
| 高手階段 | 框架內化成直覺，自然地組織信息 |

就像學開車：一開始要死記"打燈、看鏡、轉向"的順序，熟練後這些動作會變成本能。


## 什麼時候用框架，什麼時候自由發揮

| 場景 | 建議 | 原因 |
|-----|------|------|
| 複雜的技術任務 | 用完整框架 | 防止遺漏關鍵約束 |
| 簡單的快速提問 | 用精簡框架或自由發揮 | 節省時間 |
| 第一次做某類任務 | 用完整框架 | 幫助你理清思路 |
| 重複做過的任務 | 可以簡化 | 你已經知道關鍵要素 |
| AI 輸出不符合預期 | 回到框架檢查 | 定位遺漏了什麼信息 |

**一個實用建議**：當你不確定時，用框架；當框架感覺多餘時，簡化它。


## 故事化 Prompt vs 結構化框架

第二章介紹的故事化 Prompt 和本節的結構化框架，各有適用場景：

| 對比維度 | 故事化 Prompt | 結構化框架 |
|---------|--------------|-----------|
| **核心驅動** | 情感和痛點 | 邏輯和約束 |
| **表達方式** | 敘事性，像講故事 | 列表式，像填表格 |
| **適合任務** | 產品設計、用戶需求溝通 | 代碼實現、技術任務 |
| **AI 理解重點** | 用戶是誰、爲什麼需要 | 怎麼做、有什麼限制 |
| **典型場景** | "幫我設計一個待辦清單" | "幫我實現添加任務功能" |

**實際使用中，兩者可以組合**：

1. 先用故事化 Prompt 描述整體需求（產品層面）
2. 再用結構化框架拆解具體任務（技術層面）

例如：
- 第一輪對話：用故事化 Prompt 描述"我是誰、我的痛點、我想要什麼"
- 第二輪對話：用 S.C.A.F.F. 框架描述"當前要實現的具體功能"


## 本節要點

✓ **框架的價值**：減少思考負擔、提高輸出穩定性、降低溝通成本

✓ **框架不是限制**：初學時嚴格遵循，熟練後靈活調整

✓ **選擇原則**：複雜任務用完整框架，簡單任務可以簡化

✓ **與故事化 Prompt 的關係**：情感驅動 vs 邏輯驅動，可以組合使用
