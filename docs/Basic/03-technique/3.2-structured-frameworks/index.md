# 3.2 結構化提示詞框架：從"隨便說說"到"有章可循"

在 3.1 節，你學會了"上下文爲王"的核心原則，知道了該告訴 AI 什麼信息。但你可能還有一個困擾：

> "道理我都懂，但每次寫提示詞還是要從零開始想，有沒有更省力的方法？"

有的。這就是**結構化框架**存在的意義。


## 經過本節學習，你將掌握

- 理解框架的價值：爲什麼"套公式"反而能寫出更好的提示詞
- 掌握 S.C.A.F.F. 框架：適合技術開發任務的完整結構
- 掌握 R.G.C. 框架：適合快速提問的精簡結構
- 獲得萬能提示詞模板：不確定用哪個框架時的保底選擇
- 學會根據場景選擇合適的框架


## 與第二章的關係

第二章 2.4.5 介紹了**故事化 Prompt**（身份-現狀-痛點-期望），它特別適合需要表達情感和用戶痛點的場景。

本節的結構化框架與故事化 Prompt 是**互補關係**，而非替代關係：

| 框架類型 | 核心特點 | 適用場景 |
|---------|---------|---------|
| 故事化 Prompt | 情感驅動，強調用戶痛點 | 產品設計、需求溝通 |
| 結構化框架 | 邏輯驅動，強調技術約束 | 代碼實現、技術任務 |

你可以根據任務性質靈活選擇，甚至組合使用。


## 本節結構

```
3.2.1 爲什麼需要框架 → 理解框架的價值，消除"套公式很傻"的誤解
3.2.2 S.C.A.F.F. 框架 → 完整的五要素框架，適合複雜任務
3.2.3 R.G.C. 框架 → 精簡的三要素框架，適合快速提問
3.2.4 萬能提示詞模板 → 不確定用哪個時的保底選擇
3.2.5 框架選擇指南 → 什麼場景用什麼框架
```
