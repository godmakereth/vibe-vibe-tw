# 3.2.3 Role → Goal → Constraints 框架（簡化版）

## 經過本節學習，你將掌握

- R.G.C. 框架的三個要素及其含義
- 什麼場景適合用這個精簡框架
- S.C.A.F.F. 和 R.G.C. 的選擇標準
- 知道何時從 R.G.C. 升級到 S.C.A.F.F.


## 框架結構

R.G.C. 是三個英文單詞的首字母縮寫：

| 字母 | 英文 | 中文 | 核心問題 |
|-----|------|------|---------|
| **R** | Role | 角色 | 讓 AI 扮演什麼身份？ |
| **G** | Goal | 目標 | 要完成什麼任務？ |
| **C** | Constraints | 約束 | 有哪些限制條件？ |

這個框架的設計邏輯是：**定身份 → 說目標 → 劃邊界**。

相比 S.C.A.F.F.，R.G.C. 更輕量，適合快速提問和相對簡單的任務。


## 三要素詳解

### R - Role（角色）

讓 AI 進入特定的"專家模式"。

**爲什麼角色很重要**：同樣的問題，程序員視角和產品經理視角的回答會很不同。指定角色可以讓 AI 的輸出更符合你的期望。

**常用角色示例**：
- "你是一個資深的 React 開發者"
- "你是一個注重代碼可讀性的 Python 工程師"
- "你是一個擅長數據可視化的分析師"
- "你是一個對初學者友好的編程導師"

**填寫技巧**：角色描述可以包含專長、風格偏好、經驗水平。

### G - Goal（目標）

說清楚要完成什麼任務。

**填寫內容**：
- 具體的任務描述
- 期望的輸出結果
- 成功的標準

**填寫技巧**：用"動詞 + 名詞"的結構，如"實現一個函數"、"分析這段代碼"、"解釋這個概念"。

### C - Constraints（約束）

劃定任務的邊界。

**填寫內容**：
- 技術限制（語言、框架、庫）
- 格式要求（長度、風格）
- 內容限制（不要什麼）

**填寫技巧**：約束越明確，AI "加戲"的空間越小。


## 完整示例：代碼重構

### 任務場景

你有一段能用但很亂的代碼，想讓 AI 幫你重構。

### ❌ 差提示詞

```
幫我優化這段代碼：
[粘貼代碼]
```

**問題**："優化"可以有很多種理解——性能優化？可讀性優化？架構優化？AI 不知道你想要哪種。

### ✅ 用 R.G.C. 框架改寫

```markdown
【Role - 角色】
你是一個注重代碼可讀性和可維護性的 Python 工程師。

【Goal - 目標】
重構下面這段代碼，提高可讀性：
- 讓變量命名更清晰
- 把重複邏輯提取成函數
- 添加必要的註釋

```python
def f(l):
    r=[]
    for i in l:
        if i>0:
            r.append(i*2)
    return r
```

【Constraints - 約束】
- 保持功能完全不變
- 不使用列表推導式（我還沒學到那部分）
- 不需要添加類型註解
- 註釋用中文
```

### AI 的可能輸出

```python
def double_positive_numbers(numbers):
    """
    從列表中篩選正數，並將每個正數翻倍
    
    參數：numbers - 輸入的數字列表
    返回：包含翻倍後正數的新列表
    """
    result = []
    
    for number in numbers:
        # 只處理正數
        if number > 0:
            doubled = number * 2
            result.append(doubled)
    
    return result
```


## 更多 R.G.C. 示例

### 示例一：解釋概念

```markdown
【Role】你是一個對編程初學者友好的導師

【Goal】用通俗的語言解釋什麼是"回調函數"

【Constraints】
- 不要用其他技術術語
- 舉一個生活中的類比
- 控制在 200 字以內
```

### 示例二：生成測試數據

```markdown
【Role】你是一個熟悉電商業務的數據工程師

【Goal】生成 10 條模擬的訂單數據

【Constraints】
- 格式：JSON 數組
- 每條包含：訂單ID、用戶名、商品名、金額、下單時間
- 金額範圍：50-500 元
- 時間範圍：2024 年 1 月
```

### 示例三：代碼審查

```markdown
【Role】你是一個嚴格但友善的代碼審查員

【Goal】審查這段代碼，指出潛在問題：
[粘貼代碼]

【Constraints】
- 只關注安全性和性能問題
- 不評價代碼風格
- 每個問題給出修復建議
- 問題數量不超過 5 個
```


## S.C.A.F.F. vs R.G.C.：同一任務的兩種寫法

假設任務是：爲待辦清單添加"完成任務"功能。

### R.G.C. 寫法（快速版）

```markdown
【Role】你是一個 React 開發者

【Goal】實現點擊任務時將其標記爲已完成的功能

【Constraints】
- 用 TypeScript
- 完成的任務顯示刪除線
- 用 Tailwind CSS
```

### S.C.A.F.F. 寫法（完整版）

```markdown
【Situation】
待辦清單項目，React + TS + Tailwind，已有 AddTask 和 TaskList 組件

【Challenge】
實現"完成任務"功能：
- 點擊任務文本，切換完成/未完成狀態
- 完成的任務顯示刪除線和淺灰色
- 可以重複點擊切換狀態

【Audience】
我是 React 初學者，請解釋狀態管理的邏輯

【Format】
更新 TaskList 組件代碼，添加必要註釋

【Foundations】
- 用 useState 管理狀態
- 不用任何動畫庫
- 樣式變化要有 0.2s 過渡效果
```

### 選擇標準

| 場景 | 推薦框架 |
|-----|---------|
| 對項目背景熟悉，任務簡單明確 | R.G.C. |
| 新項目或複雜任務，需要詳細說明背景 | S.C.A.F.F. |
| 快速提問，只是想驗證一個想法 | R.G.C. |
| 正式開發，代碼要進入項目 | S.C.A.F.F. |
| AI 輸出不理想，需要增加約束 | 從 R.G.C. 升級到 S.C.A.F.F. |


## 何時從 R.G.C. 升級到 S.C.A.F.F.

當你用 R.G.C. 提問後，發現 AI 的輸出不符合預期，這通常是**信息不足**的信號。

**常見信號**：

| 問題表現 | 可能缺少的信息 | 升級方向 |
|---------|--------------|---------|
| 技術棧不對 | 項目背景 | 添加 Situation |
| 功能太簡單/太複雜 | 具體需求 | 補充 Challenge |
| 代碼風格不匹配 | 維護者信息 | 添加 Audience |
| 輸出格式混亂 | 格式要求 | 添加 Format |

**升級示例**：

```markdown
# 原來的 R.G.C.
【Role】React 開發者
【Goal】實現表單驗證
【Constraints】不用第三方庫

# AI 輸出了一個過於複雜的驗證方案

# 升級爲 S.C.A.F.F.
【Situation】個人博客項目，只有一個簡單的登錄表單
【Challenge】驗證郵箱格式和密碼非空，錯誤時顯示提示
【Audience】我是初學者，請用最簡單的方式實現
【Format】完整組件代碼 + 中文註釋
【Foundations】只用 useState，不用 useReducer 或其他複雜模式
```


## 可複製的 R.G.C. 模板

```markdown
【Role - 角色】
你是一個 [專長/風格描述] 的 [職業身份]

【Goal - 目標】
[動詞] [具體任務描述]

【Constraints - 約束】
- [限制條件1]
- [限制條件2]
- [限制條件3]
```


## 本節要點

✓ **R.G.C. 三要素**：Role（角色）→ Goal（目標）→ Constraints（約束）

✓ **適用場景**：快速提問、簡單任務、對項目已有清晰認知

✓ **與 S.C.A.F.F. 的關係**：R.G.C. 是精簡版，輸出不理想時可升級

✓ **升級信號**：當 AI 輸出不符合預期，檢查是否缺少背景、需求、受衆或格式信息
