# 3.4.3 適合初學者的 PRD 模板（精簡版）

本節給你一份可以直接複製使用的 PRD 模板，以及每個字段的填寫指南。

## 核心模板（可直接複製）

```markdown
# [項目名稱]

## 一句話描述
[用一句話說清楚這是什麼、給誰用、解決什麼問題]

## 目標用戶
[誰會用這個？保留與產品相關的特徵]

## 要解決的問題
[用戶遇到了什麼困擾？現有方案有什麼不足？]

## 核心功能（P0，必須有）
- [ ] 功能1：[簡要描述]
- [ ] 功能2：[簡要描述]
- [ ] 功能3：[簡要描述]

## 不做的功能
- 不要 [功能A]（原因）
- 不要 [功能B]（原因）
- 不要 [功能C]（原因）

## 用戶流程
1. 用戶 [動作] → 看到 [結果]
2. 用戶 [動作] → 看到 [結果]
3. 用戶 [動作] → 看到 [結果]

## 技術約束
- 技術棧：[使用什麼技術]
- 數據存儲：[存在哪裏]
- 界面語言：[中文/英文]
- 其他限制：[如有]

## 成功標準
[完成這些就算做完了：]
- [ ] 標準1
- [ ] 標準2
- [ ] 標準3
```

## 逐字段填寫指南

### 字段1：項目名稱

**目的**：給你的項目起一個簡短、有辨識度的名字。

**填寫技巧**：
- 2-6個字爲佳
- 能體現核心功能或特點
- 避免太抽象的名字

**示例**：
- ✅ 極簡待辦、每日打卡、銷售週報分析器
- ❌ 我的項目、系統、工具

### 字段2：一句話描述

**目的**：讓 AI 在 3 秒內理解你要做什麼。

**填寫公式**：
```
一個 [給誰用的] + [核心功能] + [產品形態]
```

**示例**：
- "一個給自己用的極簡待辦清單網頁"
- "一個幫銷售團隊分析渠道ROI的數據看板"
- "一個自動彙總每週 Excel 報表的腳本"

### 字段3：目標用戶

**目的**：告訴 AI 要爲誰設計。

**填寫技巧**：只保留與產品相關的特徵。

| 場景 | 填寫示例 |
|-----|---------|
| 給自己用 | "我自己（職場人士，每天處理5-10件事）" |
| 給特定人羣 | "銷售團隊成員（需要每週提交業績數據）" |
| 給家人用 | "我60歲的父母（智能手機操作不熟練）" |

### 字段4：要解決的問題

**目的**：讓 AI 理解你爲什麼要做這個。

**填寫技巧**：描述痛點，而不是解決方案。

❌ 錯誤：
```
需要一個待辦清單
```

✅ 正確：
```
怕遺漏重要事項。
便籤紙容易丟，手機備忘錄打開太麻煩，
需要一個"打開就能用"的工具。
```

### 字段5：核心功能（P0）

**目的**：明確告訴 AI 必須實現什麼。

**填寫技巧**：
- 只列必須有的功能（P0）
- 每個功能用一行，簡明扼要
- 建議 3-5 個，不超過 7 個

**示例**：
```markdown
## 核心功能（P0）
- [ ] 添加任務：輸入框 + 添加按鈕
- [ ] 查看任務：以列表形式顯示所有任務
- [ ] 完成任務：點擊勾選，顯示刪除線
- [ ] 刪除任務：點擊刪除按鈕移除任務
```

### 字段6：不做的功能

**目的**：防止 AI "自作主張"添加功能。

**填寫技巧**：
- 明確寫出不要什麼
- 最好說明原因（幫 AI 理解邊界）
- 把 P2 功能也放這裏（暫不做）

**示例**：
```markdown
## 不做的功能
- 不要登錄註冊（這是個人工具，不需要賬號系統）
- 不要雲同步（本地存儲夠用，避免複雜度）
- 不要分類標籤（保持極簡，第一版不需要）
- 不要定時提醒（這是 P2 功能，以後再加）
```

### 字段7：用戶流程

**目的**：讓 AI 理解用戶怎麼使用這個產品。

**填寫技巧**：
- 描述關鍵步驟，不需要太細
- 格式："用戶 [動作] → 看到 [結果]"
- 3-5 步爲佳

**示例**：
```markdown
## 用戶流程
1. 用戶打開網頁 → 看到任務列表（如果有歷史任務）
2. 用戶在輸入框輸入任務 → 點擊添加按鈕 → 任務出現在列表中
3. 用戶完成任務 → 點擊勾選 → 任務顯示刪除線
4. 用戶關閉網頁 → 下次打開 → 數據還在
```

### 字段8：技術約束

**目的**：告訴 AI 技術層面的限制條件。

**常見約束類型**：

| 約束類型 | 示例 |
|---------|------|
| 技術棧 | "純前端，用 HTML + CSS + JavaScript" |
| 數據存儲 | "存在瀏覽器 localStorage，不需要後端" |
| 界面語言 | "界面用中文" |
| 兼容性 | "需要在手機瀏覽器上能用" |
| 依賴限制 | "不要用任何框架，保持原生" |

**如果你不懂技術**，可以這樣寫：
```markdown
## 技術約束
- 我是零基礎，請用最簡單的技術
- 不需要數據庫，數據存在本地即可
- 界面用中文
```

### 字段9：成功標準

**目的**：定義什麼時候算"做完了"。

**填寫技巧**：用可驗證的行爲描述。

❌ 模糊：
```
能正常使用
```

✅ 具體：
```markdown
## 成功標準
- [ ] 打開網頁能看到輸入框和添加按鈕
- [ ] 輸入任務後點擊添加，任務出現在列表
- [ ] 點擊任務可以勾選完成
- [ ] 刷新頁面後，之前添加的任務還在
```

## 完整填寫示例：小李的待辦清單

```markdown
# 極簡待辦

## 一句話描述
一個給自己用的極簡待辦清單網頁，只有添加和勾選功能

## 目標用戶
我自己（職場人士，每天處理5-10件事務）

## 要解決的問題
怕遺漏重要事項。便籤紙容易丟，手機備忘錄打開太麻煩，
需要一個"打開瀏覽器就能用"的極簡工具。

## 核心功能（P0）
- [ ] 添加任務：輸入框 + 添加按鈕
- [ ] 查看任務：列表顯示所有待辦
- [ ] 完成任務：點擊勾選，顯示刪除線
- [ ] 刪除任務：移除不需要的任務

## 不做的功能
- 不要登錄註冊（個人工具不需要）
- 不要雲同步（本地存儲夠用）
- 不要分類標籤（保持極簡）
- 不要日曆視圖（不是日程管理工具）
- 不要定時提醒（以後再加）

## 用戶流程
1. 打開網頁 → 看到任務列表
2. 輸入任務 → 點擊添加 → 任務出現
3. 完成任務 → 點擊勾選 → 顯示刪除線
4. 關閉網頁 → 下次打開 → 數據還在

## 技術約束
- 純前端：HTML + CSS + JavaScript
- 數據存儲：瀏覽器 localStorage
- 界面語言：中文
- 樣式：簡潔美觀，白底黑字

## 成功標準
- [ ] 能添加任務並顯示在列表
- [ ] 能勾選任務標記完成
- [ ] 能刪除任務
- [ ] 刷新頁面數據不丟失
```

## 不同場景的模板變體

### 變體A：數據分析場景

```markdown
# [分析項目名稱]

## 一句話描述
[分析什麼數據、給誰看、回答什麼問題]

## 目標用戶
[誰會看這個分析結果]

## 要回答的問題
[這個分析要解決什麼業務問題]

## 數據來源
[數據從哪裏來，什麼格式]

## 分析維度
- 維度1：[說明]
- 維度2：[說明]
- 維度3：[說明]

## 輸出形式
[表格/圖表/報告，具體格式要求]

## 技術約束
[用 Python/Excel/其他工具]

## 成功標準
[老闆/同事看完後能做出什麼決策]
```

### 變體B：自動化腳本場景

```markdown
# [腳本名稱]

## 一句話描述
[自動完成什麼任務]

## 使用者
[誰會運行這個腳本]

## 要解決的問題
[現在手動做這件事有什麼痛點]

## 輸入
[腳本需要什麼輸入，格式是什麼]

## 輸出
[腳本產出什麼結果，保存在哪裏]

## 處理邏輯
1. 第一步：[做什麼]
2. 第二步：[做什麼]
3. 第三步：[做什麼]

## 技術約束
[用什麼語言，運行環境是什麼]

## 成功標準
[跑完腳本後，應該看到什麼結果]
```

## 讓 AI 更容易理解的技巧

### 技巧1：使用 Markdown 格式

AI 對 Markdown 格式有很好的理解。用 `#` 標題、`-` 列表、`**` 加粗，比純文本更清晰。

### 技巧2：列表優於段落

❌ 段落式：
```
核心功能包括添加任務、查看任務列表、勾選完成和刪除任務。
```

✅ 列表式：
```markdown
## 核心功能
- 添加任務
- 查看任務列表
- 勾選完成
- 刪除任務
```

### 技巧3：避免歧義表述

❌ 歧義：
```
界面要好看
```

✅ 具體：
```
界面風格：簡潔現代，白色背景，主色調藍色，無邊框圓角按鈕
```

### 技巧4：用"不要"明確邊界

AI 傾向於做得多而不是少。用"不要"明確告訴它邊界。

```markdown
## 不做的功能
- 不要添加任何我沒要求的功能
- 不要使用任何第三方庫
- 不要創建多個頁面，只需要一個單頁應用
```

## 本節核心要點

- ✅ PRD 模板有 8 個核心字段，每個都有明確用途
- ✅ "一句話描述"用公式：給誰用 + 核心功能 + 產品形態
- ✅ "不做的功能"和"核心功能"同樣重要
- ✅ "成功標準"要用可驗證的行爲描述
- ✅ 使用 Markdown 格式、列表、明確的"不要"讓 AI 更容易理解
