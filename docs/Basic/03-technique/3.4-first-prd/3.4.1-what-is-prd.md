# 3.4.1 什麼是 PRD，爲什麼 Vibe Coding 需要它

## 一個真實的困境

小李在第二章已經想清楚了：他要做一個極簡待辦清單，幫助職場人士管理每日任務。他打開 AI 工具，輸入：

> "幫我做一個待辦清單應用"

AI 回覆了一大段代碼。小李興奮地運行，結果發現：

- AI 做了一個帶登錄註冊的複雜系統（他只想給自己用）
- 有日曆視圖、標籤分類、團隊協作（他只要最簡單的添加和勾選）
- 界面是英文的（他想要中文）
- 用了他完全不懂的技術棧

小李花了兩個小時調試，最後放棄了。

**問題出在哪？**

不是 AI 不夠聰明，而是小李沒有把他想清楚的東西，用 AI 能理解的方式說出來。

## 什麼是 PRD

PRD（Product Requirements Document），直譯是"產品需求文檔"。

用大白話說：**PRD 是一份任務說明書，告訴執行者（AI 或開發者）要做什麼、爲誰做、做成什麼樣。**

一個類比：你去餐廳點菜，"隨便來點喫的"和"一份少油少鹽的西紅柿炒蛋，不要蔥"，哪個更可能得到你想要的結果？

PRD 就是那份詳細的點菜單。

## 傳統 PRD vs Vibe Coding PRD

在傳統軟件開發中，PRD 通常是一份幾十頁的正式文檔，包含市場分析、競品對比、技術架構等內容。

但在 Vibe Coding 中，**PRD 的目的完全不同**：

| 維度 | 傳統 PRD | Vibe Coding PRD |
|-----|---------|-----------------|
| **閱讀對象** | 產品經理、開發團隊、老闆 | AI |
| **核心目的** | 對齊團隊認知、作爲合同依據 | 讓 AI 準確理解你的意圖 |
| **篇幅** | 幾十頁，詳細全面 | 20-50 行，精簡清晰 |
| **格式** | Word/PPT，圖文並茂 | Markdown 純文本，結構化 |
| **更新頻率** | 立項時寫一次 | 每輪對話可能調整 |
| **專業要求** | 需要產品經理技能 | 只需要說清楚你要什麼 |

**核心洞見**：Vibe Coding 的 PRD 不是給人看的"報告"，而是給 AI 的"任務書"。

## PRD 解決的三個問題

### 問題一：AI 總是"自由發揮"

沒有 PRD 時，AI 會根據自己的"理解"填補你沒說的部分。它可能：
- 添加你不需要的功能
- 用你不熟悉的技術
- 做成和你想象完全不同的樣子

有了 PRD，你明確告訴 AI："只做這些，不做那些"，大大減少返工。

### 問題二：做着做着忘了初衷

開發過程中，你可能會被各種細節帶偏，忘記最初要解決什麼問題。

PRD 是一個"錨點"。每次迷茫時，回頭看 PRD："我當初要解決什麼問題？現在做的事情是在解決這個問題嗎？"

### 問題三：不知道什麼時候算"做完了"

沒有 PRD 時，你可能會陷入無限的"再加一個功能"循環。

PRD 裏的"核心功能"和"成功標準"告訴你：**做到這些就夠了，可以停下來驗證了。**

## 差提示詞 vs 好提示詞

讓我們看看小李的問題如何用 PRD 解決：

❌ **差提示詞**：
```
幫我做一個待辦清單應用
```

問題：
- AI 不知道用戶是誰（個人用還是團隊用？）
- AI 不知道核心功能是什麼（要日曆嗎？要提醒嗎？）
- AI 不知道技術約束（用什麼語言？要不要數據庫？）
- AI 不知道什麼算完成

✅ **好提示詞（PRD 形式）**：
```markdown
# 極簡待辦清單

## 一句話描述
一個只有添加和勾選功能的待辦清單網頁，給自己用

## 目標用戶
我自己（職場人士，每天處理 5-10 件事）

## 要解決的問題
怕遺漏重要事項，需要一個比便籤紙更容易堅持使用的工具

## 核心功能（只做這些）
- 添加任務（輸入框 + 按鈕）
- 查看任務列表
- 點擊勾選完成
- 刪除任務

## 不做的功能
- 不要登錄註冊
- 不要日曆視圖
- 不要分類標籤
- 不要團隊協作
- 不要雲端同步（本地存儲即可）

## 技術約束
- 純前端，用 HTML + CSS + JavaScript
- 數據存在瀏覽器 localStorage
- 界面用中文

## 成功標準
打開網頁能添加任務、勾選完成、刪除任務，刷新後數據還在
```

**這就是一份 Vibe Coding PRD。** 不需要幾十頁，只需要把關鍵信息寫清楚。

## 爲什麼"先寫後做"更高效

你可能會想：直接告訴 AI 不就行了，何必先寫文檔？

三個原因：

**1. 寫作是思考的過程**

當你把想法寫成文字，會發現很多模糊的地方。"做一個好用的待辦清單"——什麼叫"好用"？寫 PRD 的過程會逼你把模糊的想法變清晰。

**2. 減少來回修改**

根據 ChatPRD（一家專注 AI 輔助 PRD 的公司）的數據，使用結構化 PRD 的開發者，平均減少 60% 的返工次數。

**3. 可複用、可迭代**

PRD 寫好後，可以多次使用：
- 這次 AI 沒做好？調整 PRD 再試
- 換一個 AI 工具？直接複製 PRD
- 下次做類似項目？在這份 PRD 基礎上改

## 本節核心要點

- ✅ PRD 是給 AI 的"任務書"，不是給老闆看的"報告"
- ✅ Vibe Coding 的 PRD 只需要 20-50 行，核心是結構清晰、信息完整
- ✅ PRD 解決三個問題：防止 AI 自由發揮、保持初衷、定義完成標準
- ✅ "先寫後做"能減少返工，讓開發過程更高效
