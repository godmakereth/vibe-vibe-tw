# 3.4.2 從第二章成果到 PRD：整合你的思考

如果你認真完成了第二章的練習，恭喜你——你已經擁有了寫 PRD 所需的所有素材。

本節的任務是：**把這些思考成果"搬"到 PRD 的對應位置。**

## 第二章成果 → PRD 字段映射表

| 第二章工具 | 你已完成的思考 | 對應 PRD 字段 |
|-----------|--------------|--------------|
| 靈魂三問 2.5.2 | 用戶是誰 | **目標用戶** |
| 靈魂三問 2.5.3 | 痛點在哪 | **要解決的問題** |
| 靈魂三問 2.5.4 | 爲什麼選你 | **一句話描述**（差異化） |
| 減法思維 2.3 | P0 功能列表 | **核心功能** |
| 減法思維 2.3.5 | "不做清單" | **不做的功能** |
| 用戶旅程 2.4.4 | 關鍵步驟 | **用戶流程** |
| 故事思維 2.4 | 用戶畫像 | 融入**目標用戶**描述 |

**核心洞見**：PRD 不是從零開始寫，而是把第二章的思考"翻譯"成 AI 能執行的格式。

## 小李的映射示例

讓我們看看小李在第二章做了哪些思考，以及如何填入 PRD：

### 第二章：靈魂三問的答案

```
用戶是誰？
→ 我自己，28歲職場人士，每天處理5-10件事

痛點在哪？
→ 怕遺漏重要事項，便籤紙容易丟，手機備忘錄不夠直觀

爲什麼選我的方案？
→ 比便籤紙更不容易丟，比複雜待辦App更簡單容易堅持
```

### 第二章：減法思維的成果

```
P0（必須有）：添加任務、查看列表、標記完成
P1（應該有）：刪除任務、任務計數
P2（可以有）：分類標籤、優先級標記
不做清單：登錄註冊、雲同步、團隊協作、日曆視圖
```

### 第二章：用戶旅程的關鍵步驟

```
1. 打開網頁 → 看到任務列表
2. 輸入新任務 → 點擊添加
3. 完成任務 → 點擊勾選
4. 關閉網頁 → 下次打開數據還在
```

### 填入 PRD 後

```markdown
## 目標用戶
我自己（28歲職場人士，每天處理5-10件事）

## 要解決的問題
怕遺漏重要事項。便籤紙容易丟，手機備忘錄不夠直觀，
需要一個"打開就能用、關掉數據還在"的極簡工具。

## 一句話描述
一個比便籤紙更持久、比複雜App更簡單的待辦清單網頁

## 核心功能（P0）
- 添加任務
- 查看任務列表
- 標記任務完成

## 不做的功能
- 登錄註冊
- 雲端同步
- 團隊協作
- 日曆視圖
- 分類標籤（P2，暫不做）

## 用戶流程
1. 打開網頁 → 看到任務列表
2. 輸入任務 → 點擊添加按鈕
3. 完成任務 → 點擊勾選
4. 關閉網頁 → 下次打開數據還在
```

看到了嗎？PRD 的每一部分都有來源，不需要憑空編造。

## 填寫順序建議

不是所有字段都同等重要。推薦的填寫順序：

| 順序 | 字段 | 原因 |
|-----|------|------|
| 1 | 要解決的問題 | 這是核心，決定其他所有內容 |
| 2 | 目標用戶 | 問題是誰的問題？ |
| 3 | 核心功能 | 解決問題需要什麼功能？ |
| 4 | 不做的功能 | 同樣重要，防止範圍蔓延 |
| 5 | 用戶流程 | 用戶怎麼用這些功能？ |
| 6 | 技術約束 | 有什麼限制條件？ |
| 7 | 成功標準 | 怎麼算做完了？ |
| 8 | 一句話描述 | 最後總結提煉 |

**爲什麼"要解決的問題"排第一？**

因爲功能是爲問題服務的。如果你先寫功能，很容易陷入"我想要這個、那個"的功能堆砌。先寫問題，每個功能都要回答："這個功能是在解決什麼問題？"

## 常見錯誤

### 錯誤一：跳過問題，直接寫功能

❌ 錯誤示範：
```markdown
## 核心功能
- 添加任務
- 刪除任務
- 分類標籤
- 日曆視圖
- 提醒通知
```

問題：這些功能在解決什麼問題？爲什麼需要日曆視圖？

✅ 正確做法：
```markdown
## 要解決的問題
怕遺漏重要事項

## 核心功能
- 添加任務（記錄要做的事）
- 查看列表（知道還有什麼沒做）
- 標記完成（獲得成就感）
```

### 錯誤二：把"不做清單"當擺設

很多人寫 PRD 時，"不做清單"要麼空着，要麼隨便填幾個。

但這恰恰是 AI 最需要知道的信息。沒有明確的邊界，AI 會"貼心地"幫你加上各種功能。

✅ 好的"不做清單"：
```markdown
## 不做的功能
- 不要登錄註冊（這是個人工具）
- 不要雲同步（本地存儲夠用）
- 不要標籤分類（保持極簡）
- 不要優先級標記（第一版不需要）
- 不要移動端適配（先做桌面版）
```

### 錯誤三：用戶畫像太詳細

第二章的三維用戶畫像是爲了幫你深入理解用戶。但 PRD 裏不需要那麼詳細。

❌ 過於詳細：
```markdown
## 目標用戶
小李，28歲，在互聯網公司做運營，每天通勤1.5小時，
喜歡喝咖啡，週末會去健身房，有輕度焦慮症...
```

✅ 精簡版：
```markdown
## 目標用戶
職場人士，每天處理5-10件事務，需要簡單的任務追蹤工具
```

PRD 裏的用戶描述只需要保留**與產品相關的特徵**。

## 沒做第二章怎麼辦

如果你直接跳到這裏，沒有完成第二章的思考，也不用回頭從頭學。

你可以用這個快速版自檢清單：

- [ ] 我能用一句話說清楚"這是給誰用的"嗎？
- [ ] 我能說出用戶遇到的具體問題嗎？
- [ ] 我能列出 3 個必須有的核心功能嗎？
- [ ] 我能列出 3 個明確不做的功能嗎？

如果這四個問題你都能回答，就可以繼續。如果有任何一個答不上來，建議花 10 分鐘快速瀏覽第二章的 2.3（減法思維）和 2.5（靈魂三問）。

## 本節核心要點

- ✅ PRD 的內容來自第二章的思考成果，不是憑空編寫
- ✅ 先寫"要解決的問題"，再寫功能
- ✅ "不做清單"和"核心功能"同樣重要
- ✅ 用戶描述只保留與產品相關的特徵，不需要完整畫像
