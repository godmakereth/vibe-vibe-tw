# 3.4.5 PRD 實戰：從想法到文檔

本節通過一個完整案例，展示從"我想做一個..."到"PRD 寫完，可以開始開發"的全過程。

## 實戰流程概覽

```
1. 從模糊想法開始
      ↓
2. 快速回答靈魂三問
      ↓
3. 劃分功能優先級
      ↓
4. 填寫 PRD 模板
      ↓
5. 讓 AI 檢查 PRD
      ↓
6. 驗證 AI 的理解
      ↓
7. 開始第一輪開發
```

## 完整案例：小李的待辦清單

### 第一步：從模糊想法開始

小李的想法很簡單：

> "我每天有很多事要做，經常忘記。想做一個待辦清單。"

這個想法太模糊了。如果直接告訴 AI，會得到各種各樣的版本。

### 第二步：快速回答靈魂三問（3分鐘）

小李用第二章學的靈魂三問快速梳理：

| 問題 | 小李的回答 |
|-----|-----------|
| **用戶是誰？** | 我自己，職場人士，每天 5-10 件事 |
| **痛點在哪？** | 怕遺漏重要事項。便籤紙丟，手機備忘錄麻煩 |
| **爲什麼選你？** | 比便籤紙持久，比複雜App簡單 |

### 第三步：劃分功能優先級（2分鐘）

小李用第二章的減法思維，把想到的功能分三類：

**P0（必須有，沒有就不能用）**
- 添加任務
- 查看任務列表
- 標記完成

**P1（應該有，能讓產品更好用）**
- 刪除任務
- 任務計數

**P2（可以有，錦上添花）**
- 分類標籤
- 優先級標記
- 定時提醒

**不做清單**
- 登錄註冊
- 雲同步
- 團隊協作
- 日曆視圖

### 第四步：填寫 PRD 模板（5分鐘）

小李打開 3.4.3 的模板，開始填寫：

```markdown
# 極簡待辦

## 一句話描述
一個給自己用的極簡待辦清單網頁，只有添加和勾選功能

## 目標用戶
我自己（職場人士，每天處理5-10件事務）

## 要解決的問題
怕遺漏重要事項。
便籤紙容易丟，手機備忘錄打開太麻煩，
需要一個"打開瀏覽器就能用"的極簡工具。

## 核心功能（P0）
- [ ] 添加任務：輸入框 + 添加按鈕
- [ ] 查看任務：列表顯示所有待辦
- [ ] 完成任務：點擊勾選，顯示刪除線

## 不做的功能
- 不要登錄註冊（個人工具不需要）
- 不要雲同步（本地存儲夠用）
- 不要分類標籤（保持極簡）
- 不要日曆視圖（不是日程管理工具）
- 不要定時提醒（以後再加）
- 不要刪除功能（P1，下一輪做）

## 用戶流程
1. 打開網頁 → 看到任務列表（或空狀態提示）
2. 輸入任務 → 點擊添加 → 任務出現在列表
3. 完成任務 → 點擊勾選 → 顯示刪除線
4. 關閉網頁 → 下次打開 → 數據還在

## 技術約束
- 純前端：HTML + CSS + JavaScript
- 數據存儲：瀏覽器 localStorage
- 界面語言：中文
- 樣式：簡潔美觀，白底黑字

## 成功標準
- [ ] 能添加任務並顯示在列表
- [ ] 能勾選任務標記完成
- [ ] 刷新頁面數據不丟失
```

### 第五步：讓 AI 檢查 PRD

PRD 寫完後，小李請 AI 幫忙檢查：

```markdown
請幫我檢查這份 PRD 是否有遺漏或歧義。

[粘貼上面的 PRD]

請回答：
1. 這份 PRD 是否足夠清晰，你能準確理解我要什麼？
2. 有沒有哪些地方你不確定該怎麼做？
3. 有沒有我可能遺漏的邊界情況？
```

**AI 的反饋**：

> 1. 整體清晰，我能理解你要做一個極簡待辦清單。
> 2. 有幾個不確定的地方：
>    - 任務爲空時，添加按鈕是否禁用？
>    - 已完成的任務放在列表什麼位置（底部/原位置）？
>    - 空狀態提示具體顯示什麼文字？
> 3. 可能遺漏的邊界情況：
>    - 任務文字過長如何處理？
>    - 任務數量上限？

小李看了 AI 的反饋，補充了幾點：

```markdown
## 補充說明
- 任務爲空時，添加按鈕可點擊但不添加（不報錯）
- 已完成任務保持原位置，只是顯示刪除線
- 空狀態顯示"暫無任務，添加一個試試"
- 任務文字過長自動換行
- 第一版不設數量上限
```

### 第六步：驗證 AI 的理解

小李讓 AI 複述理解，確保雙方對齊：

```markdown
請用你自己的話描述一下，你理解我要做什麼產品？
包括：用戶是誰、解決什麼問題、有哪些功能、沒有哪些功能。
```

**AI 的複述**：

> 你要做一個極簡待辦清單網頁，給自己（職場人士）用。
> 
> 解決的問題：怕遺漏重要事項，需要一個比便籤紙更持久、比手機備忘錄更方便的工具。
> 
> 核心功能（P0）：
> - 添加任務（輸入框+按鈕）
> - 查看任務列表
> - 勾選完成（顯示刪除線）
> 
> 明確不做：
> - 登錄註冊、雲同步、分類標籤、日曆、提醒、刪除功能
> 
> 技術方案：純前端（HTML+CSS+JS），數據存localStorage，中文界面。

小李確認理解正確，可以開始開發了。

### 第七步：開始第一輪開發

小李按 3.4.4 的方法，發起第一輪對話：

```markdown
請根據這份 PRD 實現第一版，只做 P0 功能。

[粘貼完整 PRD]

特別強調：只做核心功能部分列出的 3 個功能，不要添加任何額外功能。
```

AI 返回代碼後，小李按驗證清單測試，確認無誤後進入 P1 開發。

## 你的練習：三種場景任選其一

現在輪到你了。請從以下三個場景中選一個，完成你的第一份 PRD：

### 場景 A：個人工具

> "我想做一個每日打卡工具，記錄自己每天喝了多少杯水。"

**提示**：
- 用戶是誰？（你自己）
- 核心功能是什麼？（記錄、查看）
- 不需要什麼？（統計圖表？社交分享？）

### 場景 B：數據分析

> "我想分析公司上個月的銷售數據，看看哪個渠道效果最好。"

**提示**：
- 分析給誰看？（老闆？自己？）
- 要回答什麼問題？（哪個渠道ROI最高？）
- 數據從哪來？（Excel？）
- 輸出什麼形式？（圖表？表格？）

### 場景 C：自動化腳本

> "我每週都要把10個Excel文件合併成一個，想讓這個過程自動化。"

**提示**：
- 輸入是什麼？（10個Excel，放在某個文件夾）
- 輸出是什麼？（一個合併後的Excel）
- 處理邏輯是什麼？（按什麼順序合併？）

## PRD 自檢清單

寫完 PRD 後，用這個清單檢查：

### 內容完整性
- [ ] 一句話描述清楚了"給誰用"和"做什麼"
- [ ] 目標用戶描述了與產品相關的特徵
- [ ] 要解決的問題描述了痛點，而不是解決方案
- [ ] 核心功能（P0）不超過 5 個
- [ ] "不做清單"明確列出了邊界
- [ ] 用戶流程覆蓋了關鍵步驟
- [ ] 技術約束說明了技術限制
- [ ] 成功標準是可驗證的

### AI 可讀性
- [ ] 使用了 Markdown 格式
- [ ] 用列表而不是段落
- [ ] 沒有模糊表述（如"好看"、"方便"）
- [ ] "不要"的邊界足夠明確

### 與第二章的銜接
- [ ] 靈魂三問的答案融入了"用戶"和"問題"部分
- [ ] P0/P1/P2 劃分來自減法思維
- [ ] 用戶流程簡化自用戶旅程地圖

## 本節核心要點

- ✅ 從模糊想法到 PRD，只需要 10 分鐘
- ✅ 靈魂三問 + 減法思維 = PRD 的核心內容
- ✅ 寫完後讓 AI 檢查，補充遺漏的邊界情況
- ✅ 讓 AI 複述理解，確保雙方對齊
- ✅ PRD 確認無誤後，再開始第一輪開發

## 本章總結

恭喜你完成了 3.4 節的學習。現在你已經掌握了：

1. **PRD 是什麼**：給 AI 的任務書，不是給老闆的報告
2. **內容從哪來**：整合第二章的靈魂三問、減法思維、用戶旅程
3. **怎麼寫**：用精簡版模板，8個核心字段
4. **怎麼用**：P0→P1→P2 分階段開發
5. **怎麼驗證**：讓 AI 檢查、複述、確認

你已經擁有了從"想法"到"AI開始執行"的完整方法論。
