# 3.7.2 第二章 + 第三章：完整工作流

## 從想法到代碼的八步流程

把第二章的「想清楚」和第三章的「說明白」串聯起來，形成一個完整的工作流：

```
┌─────────────────────────────────────────────────────────────────────┐
│                     從想法到代碼：八步流程                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  第二章：想清楚                                                      │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  Step 1: 問題發現                                                   │
│  └─▶ 識別值得解決的問題（煩惱日記、五維評分）                         │
│                                                                     │
│  Step 2: 任務定義                                                   │
│  └─▶ 用 JTBD 描述用戶要完成的任務                                    │
│                                                                     │
│  Step 3: 風險預判                                                   │
│  └─▶ 用逆向思維列出可能的失敗原因                                    │
│                                                                     │
│  Step 4: 範圍收斂                                                   │
│  └─▶ 用減法思維確定 MVP 範圍（P0/P1/P2）                             │
│                                                                     │
│  Step 5: 用戶理解                                                   │
│  └─▶ 用故事思維構建用戶畫像和旅程                                    │
│                                                                     │
│  Step 6: 最終自檢                                                   │
│  └─▶ 用靈魂三問確認準備就緒                                          │
│                                                                     │
│  第三章：說明白                                                      │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  Step 7: 編寫 PRD                                                   │
│  └─▶ 整合前六步成果，形成 AI 能理解的任務書                          │
│                                                                     │
│  Step 8: 迭代對話                                                   │
│  └─▶ 與 AI 對話，逐步逼近理想結果                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```


## 成果映射表：第二章輸出如何用於第三章

| 第二章輸出 | 對應 PRD 部分 | 如何使用 |
|-----------|-------------|---------|
| **煩惱日記** | 項目背景 | 說明爲什麼要做這個項目 |
| **JTBD 任務描述** | 用戶任務 | 描述用戶要完成什麼 |
| **Pre-mortem 清單** | 約束條件 | 轉化爲「不做清單」 |
| **MVP 優先級** | 功能範圍 | 明確 P0/P1/P2 功能 |
| **用戶畫像** | 目標用戶 | 描述用戶特徵和需求 |
| **用戶旅程** | 使用場景 | 描述關鍵觸點 |
| **靈魂三問答案** | 最終自檢 | 確認需求完整性 |


## 貫穿案例：小李的待辦清單完整演示

讓我們用小李的待辦清單項目，完整演示八步流程。

### Step 1-6：第二章的思考成果

小李在第二章完成了以下思考：

**問題發現**：
> 我經常遺漏重要任務，被領導批評了好幾次。用煩惱日記記錄了一週，五維評分 23 分，適合用 Vibe Coding 解決。

**JTBD 任務描述**：
> 當我每天開始工作時，我想快速記錄今天要做的事，這樣我就不會遺漏重要任務，可以安心投入工作。

**Pre-mortem 風險清單**：
| 失敗原因 | 預防措施 |
|---------|---------|
| 功能太多做不完 | 第一版只做 3 個核心功能 |
| 用起來比便籤還麻煩 | 添加任務必須 3 秒內完成 |
| 做完了自己都不用 | 先用一週紙質清單驗證習慣 |

**MVP 優先級**：
| 優先級 | 功能 |
|-------|------|
| P0 | 添加任務、完成任務、查看今日任務 |
| P1 | 刪除任務、任務排序 |
| 不做 | 分類標籤、截止日期、提醒通知、歷史統計 |

**用戶畫像**：
| 維度 | 內容 |
|-----|------|
| 表面屬性 | 25 歲，職場新人，每天處理 10-15 件事務 |
| 行爲習慣 | 早上到公司第一件事看郵件，用手機備忘錄但經常忘記看 |
| 深層動機 | 害怕遺漏任務被批評，想成爲靠譜的人 |

**靈魂三問**：
| 問題 | 回答 |
|-----|------|
| 用戶是誰？ | 像我一樣怕遺漏任務的職場新人 |
| 痛點在哪？ | 經常忘事被批評，感到焦慮 |
| 爲什麼選我？ | 比便籤好用，比複雜工具簡單 |


### Step 7：編寫 PRD（整合第二章成果）

現在，把第二章的思考成果整合進 PRD：

```markdown
# 個人待辦清單 PRD

## 1. 項目背景

### 問題來源
我是一名 25 歲的職場新人，每天需要處理 10-15 件大小事務。
經常因爲忘記任務被領導批評，用過手機備忘錄但經常忘記看。

### 核心任務（JTBD）
當我每天開始工作時，我想快速記錄今天要做的事，
這樣我就不會遺漏重要任務，可以安心投入工作。

### 爲什麼現有方案不夠好
- 便籤紙：容易丟，不能隨時帶着
- 手機備忘錄：需要專門打開，經常忘記看
- 專業工具（Notion、滴答清單）：功能太多，學習成本高


## 2. 目標用戶

### 用戶畫像
- **身份**：職場新人（1-3 年工作經驗）
- **特點**：事情多、容易忘、怕被批評
- **場景**：早上到公司打開電腦、突然想起一件事、下班前檢查

### 核心需求
- 打開就能看到今天的任務
- 添加任務要極快（3 秒內）
- 界面簡單，沒有學習成本


## 3. 功能範圍

### P0 功能（第一版必須有）
1. **添加任務**：輸入框 + 回車即可添加
2. **完成任務**：點擊任務左側圓圈打勾
3. **查看任務**：打開頁面就是今日任務列表

### P1 功能（第二版考慮）
- 刪除任務
- 任務排序（拖拽）

### 不做清單
- ❌ 任務分類或標籤
- ❌ 截止日期或提醒
- ❌ 歷史記錄或統計
- ❌ 用戶登錄
- ❌ 多設備同步


## 4. 技術約束

- 技術棧：React + TypeScript + Tailwind CSS
- 數據存儲：瀏覽器 localStorage（暫不需要後端）
- 響應式：手機和電腦都能用


## 5. 驗證標準

### MVP 成功標準
自己連續使用 7 天，每天都用它記錄和完成任務。

### 核心假設
一個極簡的每日待辦清單，比便籤紙和手機備忘錄更好用。
```


### Step 8：轉化爲第一個 Prompt

有了 PRD，我們可以開始與 AI 對話。第一個 Prompt 用於實現 P0 功能中的「添加任務」：

```markdown
【Situation - 情境】
我在做一個個人待辦清單網頁應用。
技術棧：React + TypeScript + Tailwind CSS
目標用戶：容易忘事的職場新人（就是我自己）
設計原則：
- 極簡，打開就是今日待辦
- 添加任務 3 秒內完成
- 沒有學習成本

當前進度：項目剛初始化，還沒有任何功能代碼

【Challenge - 挑戰】
實現「添加任務」功能：
- 用戶在輸入框輸入任務內容
- 按回車鍵或點擊添加按鈕，任務添加到列表
- 添加成功後自動清空輸入框並聚焦
- 空內容不能添加，輸入框輕微抖動提示

【Audience - 受衆】
- 用戶：我自己，希望操作越簡單越好
- 代碼維護者：我自己，React 初學者
- 請在關鍵邏輯處加註釋，解釋爲什麼這樣寫

【Format - 格式】
請提供：
1. AddTask 組件代碼（處理輸入和添加）
2. TaskList 組件代碼（展示任務列表）
3. 任務數據的 TypeScript 類型定義
4. 簡單的組件使用說明

【Foundations - 約束】
- 任務數據暫存在組件 state，不需要持久化
- 不用任何狀態管理庫（Redux、Zustand 等）
- 不用任何 UI 組件庫
- 樣式用 Tailwind，保持簡潔
- 每個任務只需要：id + 內容文本 + 完成狀態
```

## 常見問題：從第二章到第三章的過渡

### Q1：第二章的內容太多，PRD 要全部寫進去嗎？

**A**：不需要。PRD 只需要包含 AI 執行任務所需的信息。

| 第二章內容 | 是否寫入 PRD | 原因 |
|-----------|-------------|------|
| 問題來源和動機 | ✅ 寫入背景 | 幫助 AI 理解項目目的 |
| JTBD 任務描述 | ✅ 寫入用戶需求 | 明確用戶要完成什麼 |
| Pre-mortem 清單 | ✅ 轉化爲約束 | 變成「不做清單」 |
| 五維評分細節 | ❌ 不需要 | 這是你自己的決策過程 |
| 用戶畫像 | ✅ 精簡版寫入 | 幫助 AI 理解用戶特徵 |
| 用戶旅程全圖 | ❌ 精簡爲場景 | 只保留關鍵觸點 |

### Q2：每次對話都要寫完整的 PRD 嗎？

**A**：不需要。PRD 是項目級別的文檔，單次對話只需要提供當前任務所需的上下文。

- **項目開始時**：寫完整的 PRD，讓 AI 理解整體背景
- **後續對話**：引用 PRD 中的相關部分，聚焦當前任務

### Q3：PRD 和 Prompt 有什麼區別？

**A**：

| 維度 | PRD | Prompt |
|-----|-----|--------|
| **粒度** | 整個項目 | 單個任務 |
| **用途** | 記錄思考、統一理解 | 讓 AI 執行具體任務 |
| **更新頻率** | 項目開始時寫，後續微調 | 每次對話都要寫 |
| **包含內容** | 背景、用戶、功能、約束 | 當前任務 + 相關上下文 |


## 本節要點

✓ **八步流程**：問題發現 → 任務定義 → 風險預判 → 範圍收斂 → 用戶理解 → 最終自檢 → 編寫 PRD → 迭代對話

✓ **成果映射**：第二章的每個輸出都能對應到 PRD 的某個部分

✓ **PRD 是橋樑**：它連接了「想清楚」和「說明白」，是從思考到執行的轉換器

✓ **不需要完美**：PRD 是活文檔，可以在開發過程中不斷完善
