# 3.7.1 本章核心要點回顧

## 六大核心技能總覽

本章教會你六項核心技能。記住這張表，你就記住了本章的精華：

| 技能 | 核心問題 | 一句話答案 | 對應章節 |
|-----|---------|-----------|---------|
| **上下文思維** | AI 需要知道什麼？ | 項目背景 + 任務細節 + 個人需求 | 3.1 |
| **結構化表達** | 如何組織信息？ | 用 S.C.A.F.F. 或 R.G.C. 框架 | 3.2 |
| **技巧選擇** | 什麼情況用什麼技巧？ | Zero-shot → Few-shot → CoT → ToT | 3.3 |
| **PRD 整合** | 如何把想法變成任務書？ | 整合第二章成果，填入 PRD 模板 | 3.4 |
| **迭代對話** | AI 沒做對怎麼辦？ | 精確反饋 + 逐步逼近 | 3.5 |
| **問題處理** | AI 出錯了怎麼辦？ | 診斷根因 + 精確修正 + 驗證幻覺 | 3.6 |


## 從「差提示詞」到「好提示詞」的蛻變

學完本章，你的提示詞應該發生這樣的變化：

### 蛻變前：模糊、不完整

```
幫我做一個待辦清單
```

問題清單：
- ❌ 沒有項目背景
- ❌ 沒有技術棧
- ❌ 沒有功能範圍
- ❌ 沒有約束條件
- ❌ 沒有輸出格式

### 蛻變後：清晰、結構化

```markdown
【Situation - 情境】
我在做一個個人待辦清單網頁應用。
技術棧：React + TypeScript + Tailwind CSS
目標用戶：容易忘事的職場新人（就是我自己）
設計原則：極簡，打開就是今日待辦，添加任務 3 秒內完成

【Challenge - 挑戰】
實現「添加任務」功能：
- 用戶在輸入框輸入任務內容
- 按回車鍵或點擊添加按鈕，任務添加到列表
- 添加成功後自動清空輸入框
- 空內容不能添加（顯示簡單提示）

【Audience - 受衆】
代碼維護者是我自己，React 初學者，請在關鍵邏輯處加註釋

【Format - 格式】
給我完整的 AddTask 組件代碼，使用 TypeScript

【Foundations - 約束】
- 不用任何外部狀態管理庫
- 不用任何 UI 組件庫
- 樣式用 Tailwind CSS
```

檢查清單：
- ✅ 有項目背景
- ✅ 有技術棧
- ✅ 有功能範圍
- ✅ 有約束條件
- ✅ 有輸出格式

## 五個技巧的使用場景速查

| 技巧 | 使用頻率 | 典型場景 | 關鍵詞 |
|-----|---------|---------|-------|
| **Zero-shot** | 50% | 代碼轉換、內容生成、代碼解釋 | 「直接問」 |
| **Few-shot** | 25% | API 文檔生成、分類任務、風格模仿 | 「給示例」 |
| **Chain of Thought** | 15% | 複雜度分析、Bug 排查、邏輯審查 | 「分步驟」 |
| **Tree of Thoughts** | 2% | 技術選型、架構決策 | 「比方案」 |
| **Self-Critique** | 8% | 核心代碼、公開文檔 | 「自檢查」 |

**實踐建議**：
1. 從 Zero-shot 開始嘗試
2. 如果輸出不滿意，再升級到其他技巧
3. 重要輸出記得加 Self-Critique


## 心法與技法的完整閉環

第二章（心法）和第三章（技法）解決的是不同問題，但缺一不可：

```
┌─────────────────────────────────────────────────────────────┐
│                        完整閉環                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   第二章：心法                    第三章：技法               │
│   ──────────────                 ──────────────             │
│                                                             │
│   問題發現 ─────────────────────▶ 寫入 PRD 背景             │
│   JTBD 思維 ────────────────────▶ 描述用戶任務              │
│   逆向思維 ─────────────────────▶ 列出約束條件              │
│   減法思維 ─────────────────────▶ P0/P1/P2 優先級           │
│   故事思維 ─────────────────────▶ 用戶畫像 + 場景           │
│   靈魂三問 ─────────────────────▶ 最終自檢                  │
│                                                             │
│   輸出：清晰的需求                輸出：AI 能執行的 Prompt    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 維度 | 心法（第2章） | 技法（第3章） |
|-----|-------------|-------------|
| **核心問題** | 做什麼？爲什麼做？ | 怎麼讓 AI 理解我要做什麼？ |
| **輸入** | 模糊的想法 | 清晰的需求 |
| **輸出** | 清晰的需求 | AI 能執行的指令 |
| **關鍵能力** | 產品思維、用戶洞察 | Prompt 工程、需求表達 |


## 本章關鍵公式

記住這三個公式，你就掌握了本章的精髓：

### 公式一：AI 輸出質量公式

> **AI 輸出質量 = 上下文質量 × AI 能力**

AI 的能力是固定的，你能控制的只有上下文質量。

### 公式二：好提示詞公式

> **好提示詞 = 背景 + 任務 + 約束 + 格式 + 示例（可選）**

缺少任何一個要素，AI 的輸出都可能偏離預期。

### 公式三：迭代修正公式

> **精確修正 = 指出問題 + 說明原因 + 給出期望**

模糊的「不對」不如精確的「這裏應該是那樣」。


## 四種代碼幻覺速查

| 類型 | 危害程度 | 表現 | 驗證方法 |
|-----|---------|------|---------|
| **包幻覺** | 🔴 高 | 推薦不存在的 npm/pip 包 | `npm search <包名>` 或訪問官網 |
| **API 幻覺** | 🟡 中 | 編造不存在的函數或方法 | 查閱官方文檔 |
| **邏輯幻覺** | 🟡 中 | 代碼邏輯與描述不符 | 用邊界情況測試 |
| **版本幻覺** | 🟡 中 | 使用已廢棄的語法或 API | 檢查版本兼容性 |

**驗證習慣**：對 AI 推薦的包和 API，先驗證再使用。
