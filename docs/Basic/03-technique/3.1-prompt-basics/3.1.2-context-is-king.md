# 3.1.2 Context is King：上下文爲王

## 經過本節學習，你將掌握

- 什麼是「上下文」：背景信息、約束條件、期望結果
- 三層上下文模型：項目上下文、任務上下文、個人上下文
- 上下文窗口的概念：AI 能「記住」多少內容
- 同一需求，有無上下文的輸出差異（真實案例對比）


## 一個公式

先記住這個公式：

> **AI 輸出的質量 = 你提供的上下文質量 × AI 的能力**

AI 的能力是固定的（由模型決定），你能控制的只有上下文質量。

這就是爲什麼同樣一句話，不同人問 AI 會得到完全不同的結果——差異不在於「怎麼問」，而在於「問之前告訴了 AI 什麼」。


## 什麼是上下文

「上下文」聽起來有點抽象，讓我們拆解一下。

上下文包含三類信息：

| 類型 | 說明 | 例子 |
|-----|------|-----|
| **背景信息** | AI 需要知道的前提條件 | 我在做一個待辦清單應用，用 React 開發 |
| **約束條件** | AI 需要遵守的限制 | 代碼要簡潔，不要用外部庫 |
| **期望結果** | 你想要什麼樣的輸出 | 給我一個可以直接複製使用的代碼片段 |

缺少任何一類，AI 的輸出都可能偏離你的期望。


## 三層上下文模型

爲了更系統地思考「該告訴 AI 什麼」，我們把上下文分爲三個層次：

```
┌─────────────────────────────────────┐
│         項目上下文                   │
│   你在做什麼項目？用什麼技術？        │
├─────────────────────────────────────┤
│         任務上下文                   │
│   當前要解決什麼具體問題？            │
├─────────────────────────────────────┤
│         個人上下文                   │
│   你的技術水平？你的偏好風格？        │
└─────────────────────────────────────┘
```

### 第一層：項目上下文

項目上下文回答的問題是：**你在做什麼？**

包含的信息：
- 項目類型（網頁應用、數據分析腳本、自動化工具等）
- 技術棧（如果涉及代碼）
- 項目背景和目標
- 目標用戶是誰（第二章的用戶畫像派上用場了）
- 已有的代碼規範或風格偏好

**示例**：
```
我正在開發一個個人待辦清單網頁應用。
- 技術棧：React + TypeScript
- 目標用戶：像我一樣容易忘事的職場新人
- 風格要求：代碼簡潔，組件化設計
```

### 第二層：任務上下文

任務上下文回答的問題是：**你現在要做什麼？**

包含的信息：
- 當前具體要完成的任務
- 任務的輸入和輸出
- 具體的約束條件
- 期望的輸出格式

**示例**：
```
當前任務：實現「添加任務」功能
- 輸入：用戶在輸入框輸入任務內容，點擊添加按鈕
- 輸出：新任務出現在任務列表中
- 約束：添加後自動清空輸入框，空內容不能添加
- 格式：給我完整的 React 組件代碼
```

### 第三層：個人上下文

個人上下文回答的問題是：**你是誰？**

包含的信息：
- 你的技術水平
- 你的學習目標
- 你的溝通偏好

**示例**：
```
我是編程初學者，第一次用 React。
請在關鍵代碼處加上註釋，解釋爲什麼這樣寫。
如果有更好的實現方式，請告訴我，但先給我一個能跑的版本。
```


## 案例對比：有無上下文的差異

讓我們用小李的待辦清單項目，看看上下文如何影響 AI 的輸出。

### ❌ 差提示詞：缺少上下文

```
幫我做一個待辦清單
```

**AI 可能的回應**：

AI 會開始猜測：
- 用什麼技術？HTML/CSS/JS？React？Vue？還是 Python？
- 做到什麼程度？簡單的還是複雜的？
- 要什麼功能？只是列表還是要分類、提醒、統計？

結果往往是：AI 給你一個「通用模板」，可能和你想要的相差甚遠。

### ✅ 好提示詞：包含三層上下文

```
【項目背景】
我正在開發一個個人待辦清單網頁應用，幫助像我一樣容易忘事的職場新人管理每日任務。
技術棧：React + TypeScript + Tailwind CSS
設計原則：極簡風格，打開就是今日待辦，添加任務要在 3 秒內完成

【當前任務】
請幫我實現「添加任務」組件：
- 用戶在輸入框輸入任務內容
- 點擊添加按鈕或按回車鍵，任務添加到列表
- 添加成功後自動清空輸入框
- 空內容不能添加（顯示簡單提示）

【輸出要求】
- 給我完整的 React 組件代碼
- 使用 TypeScript，定義好類型
- 樣式用 Tailwind CSS
- 在關鍵邏輯處加上註釋

【關於我】
我是 React 初學者，如果有初學者容易踩的坑，請提醒我。
```

**AI 的回應**會精準得多：
- 使用你指定的技術棧
- 實現你要求的具體功能
- 遵循你的設計原則
- 代碼風格符合你的水平


## 上下文窗口：AI 能「記住」多少

你可能會問：我能告訴 AI 多少信息？

這涉及到一個概念：**上下文窗口**。

把 AI 的「記憶」想象成一張餐桌。餐桌有固定大小，你放上去的盤子（信息）不能超過桌面。

| 模型 | 上下文窗口大小 | 相當於 |
|-----|--------------|-------|
| Claude Opus 4.5 | 200K tokens | 約 15 萬字 |
| Gemini 3 Pro | 1M tokens | 約 75 萬字 |
| 普通對話場景 | 通常夠用 | 無需擔心 |

**對於初學者來說**：不用太擔心這個限制。日常的項目描述和任務說明，遠遠不會超出上下文窗口。

但有一個重要提醒：**AI 不會跨對話記憶**。

如果你開啓一個新對話，之前告訴 AI 的所有信息都會「清零」。這就是爲什麼很多 AI 工具提供了「項目規則」或「系統提示詞」功能——讓某些上下文持久化，不用每次重複輸入。

（這部分會在 3.5 節「迭代對話的藝術」中詳細講解）


## 上下文檢查清單

在向 AI 提問之前，用這個清單快速檢查：

### 項目上下文 ✓
- [ ] 說明了項目類型和目標
- [ ] 說明了技術棧（如果涉及代碼）
- [ ] 說明了目標用戶是誰
- [ ] 說明了設計原則或風格偏好

### 任務上下文 ✓
- [ ] 清晰描述了當前要完成的任務
- [ ] 說明了輸入和輸出
- [ ] 列出了約束條件和邊界情況
- [ ] 指定了期望的輸出格式

### 個人上下文 ✓
- [ ] 說明了自己的技術水平
- [ ] 告訴 AI 你需要什麼樣的解釋
- [ ] 說明了特殊偏好（如果有）

不需要每次都填滿所有項。根據任務複雜度，選擇必要的信息提供即可。


## 第二章成果的應用

還記得第二章做的那些思考嗎？現在它們派上用場了：

| 第二章成果 | 對應的上下文層 | 如何使用 |
|-----------|--------------|---------|
| 靈魂三問 | 項目上下文 | 「用戶是誰」「痛點在哪」直接寫入項目背景 |
| 用戶畫像 | 項目上下文 | 描述目標用戶的特徵和需求 |
| MVP 範圍 | 任務上下文 | 明確當前階段要實現什麼、不實現什麼 |
| 不做清單 | 任務上下文 | 作爲約束條件，防止 AI「加戲」 |

小李在第二章完成的思考，可以直接轉化爲上下文：

```
【項目背景 - 來自靈魂三問】
用戶：像我一樣容易忘事的職場新人
痛點：經常遺漏重要任務，被領導批評，感到焦慮
差異化：比便籤紙好用（打開就看到），比複雜工具簡單（3 秒添加）

【範圍約束 - 來自減法思維】
第一版只做：添加任務、完成任務、查看今日任務
不做：分類標籤、截止日期、提醒通知、歷史統計
```


## 本節要點

✓ **公式**：AI 輸出質量 = 上下文質量 × AI 能力。你能控制的是上下文。

✓ **三層上下文模型**：
  - 項目上下文：你在做什麼項目
  - 任務上下文：當前要解決什麼問題
  - 個人上下文：你是誰，需要什麼

✓ **上下文窗口**：AI 的「記憶容量」有限，但日常使用通常夠用。

✓ **第二章的思考成果**可以直接轉化爲高質量的上下文。
