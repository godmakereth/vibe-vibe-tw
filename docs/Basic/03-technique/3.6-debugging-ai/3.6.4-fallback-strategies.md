# 3.6.4 兜底策略：當 AI 真的搞不定

## 經過本節學習，你將掌握

- 識別 AI 的能力邊界
- 三種兜底方案及其適用場景
- 「混合開發」模式的操作方法
- 知道何時該尋求外部幫助

## 認識 AI 的能力邊界

AI 很強大，但並非無所不能。瞭解它的邊界，能幫你在關鍵時刻做出正確判斷，避免在死衚衕裏浪費時間。

### AI 擅長的事

| 領域 | 具體任務 |
|-----|---------|
| **標準化代碼** | 實現常見功能、轉換數據格式、生成模板代碼 |
| **文檔和解釋** | 解釋代碼邏輯、生成註釋、編寫文檔 |
| **模式識別** | 識別代碼問題、找到類似方案、重構建議 |
| **快速原型** | 生成 UI 組件、搭建項目骨架、實現 MVP |

### AI 不擅長的事

| 領域 | 原因 | 示例 |
|-----|------|------|
| **複雜環境配置** | 涉及本地路徑、系統權限、網絡配置等 AI 無法訪問的信息 | Docker 網絡配置、環境變量設置 |
| **需要即時驗證的任務** | AI 無法真正運行代碼驗證結果 | 調試性能問題、排查內存泄漏 |
| **高度項目特定的邏輯** | 依賴大量項目上下文，超出對話窗口容量 | 複雜業務規則、遺留系統集成 |
| **安全敏感操作** | 涉及密鑰、憑證等不應該暴露的信息 | 配置 API 密鑰、處理用戶密碼 |
| **即時數據操作** | 無法訪問真實數據庫、外部 API | 數據庫遷移、線上問題排查 |

### 識別「AI 做不到」的信號

- 🚨 你已經提供了所有能想到的上下文，AI 還是給不出正確答案
- 🚨 AI 反覆給出「看起來對但實際不能用」的方案
- 🚨 問題涉及你無法通過文字描述的內容（如視覺效果、本地環境）
- 🚨 修改了 5 輪以上，問題還沒解決

## 三種兜底方案

### 方案一：換一個 AI 工具

不同的 AI 模型有不同的訓練數據和擅長領域。如果一個工具反覆卡住，換一個試試往往有效。

**選擇建議**：

| 任務類型 | 可嘗試的替代方案 |
|---------|----------------|
| 前端/React 相關 | 不同的 AI 可能有不同的框架偏好 |
| 算法邏輯問題 | 嘗試偏理論型的 AI 模型 |
| 代碼解釋和學習 | 選擇擅長教學的 AI |
| 快速原型 | 選擇代碼生成速度快的工具 |

**操作要點**：
- 換工具時，重新整理一遍需求描述
- 不要把前一個 AI 的錯誤代碼帶過去（避免干擾）
- 只描述問題本身和期望結果

### 方案二：搜索引擎找現成方案

很多問題已經被其他人遇到並解決過了。用搜索引擎找現成方案，可能比讓 AI 從頭生成更可靠。

**高效搜索技巧**：

```
[錯誤信息關鍵詞] site:stackoverflow.com
```

```
[技術棧] [功能描述] github
```

```
[庫名] [具體問題] 官方文檔
```

**示例**：
```
TypeError map undefined react site:stackoverflow.com
```

**然後讓 AI 幫你理解和適配**：
```markdown
我在 Stack Overflow 上找到了這個解決方案：

[粘貼找到的代碼]

請幫我：
1. 解釋這段代碼的原理
2. 根據我的項目情況（[描述你的技術棧]）進行適當修改
```

### 方案三：用 AI 理解概念，自己動手實現

當 AI 生成的代碼不可靠時，可以退一步——讓它幫你理解原理，然後你自己寫代碼。

**操作流程**：

1. **讓 AI 解釋原理**：
   ```markdown
   不要寫代碼。請解釋 [概念/功能] 的工作原理，包括：
   - 核心思路是什麼
   - 有哪些關鍵步驟
   - 有哪些常見的坑需要注意
   ```

2. **讓 AI 提供僞代碼**：
   ```markdown
   請用僞代碼（不是真正的代碼）描述實現步驟，
   讓我理解整體邏輯。
   ```

3. **你自己編寫代碼**，遇到具體問題再問 AI

**這種方式的好處**：
- 避免 AI 幻覺導致的錯誤
- 你對代碼有完整的理解
- 後續維護更容易

## 混合開發模式

「混合開發」是一種務實的策略：讓 AI 做它擅長的部分，你來補充它做不好的部分。

### 模式一：AI 做骨架，你做細節

```markdown
請生成一個 [功能] 的代碼框架，包括：
- 基本的文件結構
- 主要函數的簽名（參數和返回值）
- 每個函數的功能註釋

不需要實現具體邏輯，我自己來寫。
```

**適用場景**：
- 你熟悉業務邏輯，但懶得搭架子
- AI 對細節理解可能出錯，但骨架是通用的

### 模式二：AI 做初版，你來審查修改

**工作流**：
1. 讓 AI 生成完整代碼
2. 你來審查，標記有問題的地方
3. 對於簡單問題，讓 AI 修正
4. 對於複雜問題，你自己改

**審查要點清單**：
- [ ] 代碼邏輯是否正確？
- [ ] 邊界情況是否處理？
- [ ] 引用的庫和 API 是否存在？
- [ ] 是否有安全隱患？
- [ ] 是否符合項目規範？

### 模式三：分工協作

把任務按特性分成兩類：

| AI 負責 | 你負責 |
|--------|-------|
| 生成標準 CRUD 代碼 | 複雜業務邏輯 |
| 編寫單元測試模板 | 關鍵算法實現 |
| 生成 UI 組件骨架 | 交互細節和動畫 |
| 編寫文檔註釋 | 架構設計決策 |

## 何時尋求外部幫助

有些問題確實超出了「Vibe Coding + AI」能解決的範圍。識別這些情況，及時尋求幫助，是更高效的選擇。

### 應該尋求幫助的信號

- 問題涉及你完全不理解的領域（如安全、性能優化、DevOps）
- 卡住超過 2 小時，沒有任何進展
- 問題可能影響到數據安全或系統穩定性
- 需要做出可能影響長遠的架構決策

### 去哪裏尋求幫助

| 渠道 | 適合的問題 |
|-----|-----------|
| Stack Overflow | 技術細節問題，已有成熟答案的問題 |
| GitHub Issues | 特定庫或框架的問題 |
| 社區論壇/Discord | 需要討論的開放性問題 |
| 付費諮詢/顧問 | 複雜的架構問題、安全問題 |

### 如何有效地提問

無論是問 AI 還是問人，好的問題能大大提高獲得幫助的概率：

```markdown
**我想做什麼**：[一句話描述目標]

**我遇到的問題**：[具體問題描述]

**我已經嘗試過**：
1. [嘗試過的方法 1]
2. [嘗試過的方法 2]

**相關代碼/錯誤信息**：
[附上最小可復現的代碼或完整錯誤信息]

**我的環境**：[技術棧版本等信息]
```

## 本節要點

✓ **認識邊界**：環境配置、即時驗證、項目特定邏輯、安全操作是 AI 的弱項

✓ **三種兜底**：換 AI 工具、搜索現成方案、用 AI 學原理後自己實現

✓ **混合開發**：AI 做骨架/初版，你來補充細節和審查

✓ **尋求幫助**：卡住超過 2 小時、涉及安全架構時，及時尋求外部幫助
