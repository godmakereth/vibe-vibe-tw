# 3.3.6 技巧選擇指南：什麼情況用什麼技巧

## 經過本節學習，你將掌握

- 快速判斷任務應該用哪種技巧
- 技巧組合使用的策略
- 一張可以隨時查閱的速查表

## 核心決策流程

遇到一個任務，按以下流程選擇技巧：

```
開始
  ↓
這個任務 AI 應該「直接會做」嗎？
  ├─ 是 → Zero-shot（直接問）
  └─ 否 ↓
      
需要特定的輸出格式嗎？
  ├─ 是 → Few-shot（給示例）
  └─ 否 ↓
      
需要多步推理/分析嗎？
  ├─ 是 → Chain of Thought（分步思考）
  └─ 否 ↓
      
需要對比多個方案嗎？
  ├─ 是 → Tree of Thoughts（多路徑對比）
  └─ 否 ↓
      
輸出質量很重要嗎？
  ├─ 是 → 加上 Self-Critique（自查修正）
  └─ 否 → Zero-shot 就夠了
```

## 完整決策表格

| 任務類型 | 推薦技巧 | 原因 | 示例任務 |
|---------|---------|------|---------|
| 代碼格式轉換 | Zero-shot | AI 已掌握轉換規則 | Python 2 → Python 3 |
| 內容生成 | Zero-shot | 創意任務不需要示例 | 寫產品描述 |
| 代碼解釋 | Zero-shot | AI 理解代碼能力強 | 解釋這段代碼做什麼 |
| 格式化輸出 | Few-shot | 用示例確保格式一致 | 生成 API 文檔 |
| 分類任務 | Few-shot | 用示例明確分類標準 | 用戶反饋分類 |
| 風格模仿 | Few-shot | 用示例學習風格 | 按特定風格寫文案 |
| 複雜度分析 | Chain of Thought | 需要分步推理 | 分析時間複雜度 |
| Bug 排查 | Chain of Thought | 需要逐步推理 | 分析爲什麼頁面不更新 |
| 邏輯審查 | Chain of Thought | 需要考慮多個分支 | 審查表單驗證邏輯 |
| 技術選型 | Tree of Thoughts | 需要對比多個方案 | 選擇數據庫 |
| 架構決策 | Tree of Thoughts | 需要權衡多種方案 | 設計 API 風格 |
| 重要代碼 | + Self-Critique | 需要保證質量 | 核心業務邏輯 |
| 公開輸出 | + Self-Critique | 需要減少錯誤 | 對外 API 文檔 |

## 組合使用策略

單一技巧有時不夠，可以組合使用：

### 組合一：Few-shot + Self-Critique

**場景**：需要格式化輸出，且質量要求高

```
任務：生成 API 接口文檔

示例1：
接口：GET /users
文檔：
- 描述：獲取用戶列表
- 參數：page（分頁）, limit（每頁數量）
- 返回：用戶數組

示例2：
接口：POST /users
文檔：
- 描述：創建新用戶
- 參數：name（必填）, email（必填）
- 返回：新創建的用戶對象

請爲以下接口生成文檔：
接口：DELETE /users/:id

生成後，請自查：
- 是否遵循了示例格式？
- 是否遺漏了必要信息？
如有問題請修正。
```

### 組合二：Chain of Thought + Self-Critique

**場景**：複雜分析任務，需要確保推理正確

```
請分析這個排序算法的時間複雜度。

分析步驟：
1. 識別循環結構
2. 分析每層循環的執行次數
3. 計算總的時間複雜度

分析完成後，請自查：
- 每一步的推理是否正確？
- 最終結論是否合理？
如有錯誤請修正。
```

### 組合三：Tree of Thoughts + Chain of Thought

**場景**：複雜決策，每個方案都需要深入分析

```
請爲我的應用選擇狀態管理方案。

分析三種方案：
1. React Context
2. Redux Toolkit
3. Zustand

對每種方案，請分步分析：
- 第一步：理解其核心原理
- 第二步：分析在我的場景下的優勢
- 第三步：分析潛在的問題
- 第四步：給出適用場景

最後對比推薦。
```

## 日常使用頻率參考

根據實際開發經驗，各技巧的使用頻率大致如下：

| 技巧 | 日常使用頻率 | 典型場景 |
|-----|------------|---------|
| Zero-shot | 50% | 大部分簡單任務 |
| Few-shot | 25% | 格式化輸出 |
| Chain of Thought | 15% | 複雜邏輯分析 |
| Self-Critique | 8% | 重要輸出質量保證 |
| Tree of Thoughts | 2% | 技術選型決策 |

**實踐建議**：
1. 從 Zero-shot 開始嘗試
2. 如果輸出不滿意，再升級到其他技巧
3. 重要輸出記得加 Self-Critique

## 快速選擇清單

### 我要做代碼轉換/格式轉換
→ **Zero-shot**

### 我需要統一的輸出格式
→ **Few-shot**（給 2-3 個示例）

### 我需要分析複雜邏輯/排查 Bug
→ **Chain of Thought**（分步思考）

### 我需要做技術選型/對比方案
→ **Tree of Thoughts**（多方案對比）

### 這個輸出很重要，不能出錯
→ 加上 **Self-Critique**（自查修正）
