# 3.3.2 Few-shot Prompting：用示例教會 AI

## 經過本節學習，你將掌握

- Few-shot 的定義和工作原理
- 爲什麼示例比長篇解釋更有效
- 如何選擇高質量的示例
- 示例數量的最佳實踐
- Few-shot 的適用場景和侷限性

## 什麼是 Few-shot Prompting

Few-shot（少樣本）的意思是：**給 AI 幾個示例，讓它學習模式後處理新任務**。

```
任務：將用戶需求轉換爲技術任務

示例1：
需求："用戶可以添加待辦事項"
任務：
- 創建輸入框組件
- 實現添加按鈕點擊事件
- 更新任務列表狀態

示例2：
需求："用戶可以刪除待辦事項"
任務：
- 爲每個任務添加刪除按鈕
- 實現刪除確認邏輯
- 從列表中移除對應任務

現在請處理：
需求："用戶可以標記任務爲已完成"
```

AI 看到兩個示例後，就「學會」了轉換規則，能按相同格式處理新需求。

## 核心洞見：爲什麼示例比解釋更有效

> "展示一次，勝過解釋一百遍。"

這不是誇張。人類學習新技能時，看示範比聽講解更有效。AI 也一樣。

| 方式 | 效果 | 原因 |
|-----|------|------|
| 長篇解釋規則 | 容易遺漏細節，格式不穩定 | AI 需要「理解」你的意圖 |
| 提供示例 | 格式一致，細節完整 | AI 直接「模仿」模式 |

**實際測試表明**：對於格式化輸出任務，Few-shot 的準確率比 Zero-shot 高出 20-40%。

## 示例數量：多少個最合適？

| 數量 | 適用場景 | 權衡 |
|-----|---------|------|
| **1 個**（One-shot） | 格式明確，規則簡單 | 最省 token，但模式可能不夠清晰 |
| **2-3 個** | 大多數場景 | 平衡效果和成本，推薦默認選擇 |
| **4-5 個** | 複雜模式、邊界情況多 | 效果更好，但 token 消耗增加 |
| **更多** | 很少需要 | 收益遞減，考慮是否需要微調 |

**經驗法則**：從 2 個示例開始，如果輸出不穩定，再增加到 3 個。

## 如何選擇高質量示例

### 原則一：代表性

示例要覆蓋**典型情況**，不要全是特例。

```
❌ 差示例選擇（全是邊界情況）：
- 空字符串處理
- 超長文本處理
- 特殊字符處理

✅ 好示例選擇（先典型，後邊界）：
- 普通長度的文本（典型）
- 包含中英文混合的文本（常見變體）
- 空字符串處理（邊界情況）
```

### 原則二：多樣性

示例要覆蓋**不同類型**，避免 AI 過度擬合單一模式。

```
任務：分類用戶反饋

❌ 差示例（全是負面）：
- "太難用了" → 負面
- "垃圾軟件" → 負面

✅ 好示例（覆蓋各類）：
- "太難用了" → 負面
- "還不錯，但希望能加個夜間模式" → 中性
- "用了三個月，效率提升很多" → 正面
```

### 原則三：格式一致

所有示例的**輸入輸出格式必須統一**。

```
❌ 格式不一致：
示例1：輸入 "..." → 輸出：任務列表
示例2：輸入："..." 輸出 - 任務1 - 任務2

✅ 格式一致：
示例1：
輸入："..."
輸出：
- 任務1
- 任務2

示例2：
輸入："..."
輸出：
- 任務1
- 任務2
```

## 差提示詞 vs 好提示詞

### ❌ 差提示詞

```
幫我把用戶故事轉成開發任務，格式要清晰，包含前端和後端的工作。

用戶故事：作爲用戶，我希望能夠搜索任務
```

**問題**：沒有示例，AI 不知道你期望的「清晰格式」是什麼樣。

### ✅ 好提示詞

```
任務：將用戶故事轉換爲開發任務清單

示例：
用戶故事：作爲用戶，我希望能夠按日期篩選任務
開發任務：
【前端】
- 添加日期選擇器組件
- 實現篩選狀態管理
- 更新任務列表渲染邏輯
【後端】
- 添加日期篩選 API 參數
- 實現數據庫查詢條件

現在請處理：
用戶故事：作爲用戶，我希望能夠搜索任務
```

### 輸出對比

好提示詞的輸出會嚴格遵循示例格式：
```
開發任務：
【前端】
- 添加搜索輸入框組件
- 實現搜索關鍵詞狀態管理
- 實現防抖搜索邏輯
- 更新任務列表渲染，高亮匹配文字
【後端】
- 添加搜索關鍵詞 API 參數
- 實現模糊匹配數據庫查詢
```

## 實戰示例：待辦清單項目

小李需要 AI 幫他生成多個組件的 TypeScript 類型定義。他希望格式統一。

### Few-shot 提示詞

```
任務：爲 React 組件生成 TypeScript Props 類型定義

示例1：
組件描述：一個顯示任務標題的組件，標題必填，可選顯示完成狀態
輸出：
interface TaskTitleProps {
  title: string;           // 任務標題，必填
  isCompleted?: boolean;   // 完成狀態，可選
}

示例2：
組件描述：一個按鈕組件，顯示文字必填，點擊事件必填，可選禁用狀態
輸出：
interface ButtonProps {
  label: string;           // 按鈕文字，必填
  onClick: () => void;     // 點擊事件，必填
  disabled?: boolean;      // 禁用狀態，可選
}

現在請處理：
組件描述：一個任務卡片組件，包含標題必填、描述可選、截止日期可選、優先級必填（高/中/低）
```

### AI 輸出

```typescript
interface TaskCardProps {
  title: string;                        // 任務標題，必填
  description?: string;                 // 任務描述，可選
  dueDate?: Date;                       // 截止日期，可選
  priority: 'high' | 'medium' | 'low';  // 優先級，必填
}
```

注意 AI 自動學會了：
- 註釋風格（`// 說明，必填/可選`）
- 可選屬性用 `?`
- 枚舉類型用聯合類型

## 適用場景

| 場景 | 示例 | 爲什麼需要 Few-shot |
|-----|------|-------------------|
| 格式化輸出 | 生成 API 文檔 | 確保格式一致 |
| 分類任務 | 情感分析、標籤分類 | 明確分類標準 |
| 轉換任務 | 需求 → 任務、代碼 → 文檔 | 學習轉換規則 |
| 風格模仿 | 按特定風格寫文案 | 學習語言風格 |

## Few-shot 的侷限性

| 侷限 | 表現 | 解決方案 |
|-----|------|---------|
| Token 消耗高 | 示例佔用大量上下文 | 精簡示例，只保留關鍵信息 |
| 示例選擇難 | 選錯示例導致效果差 | 遵循代表性、多樣性、一致性原則 |
| 過度模仿 | AI 照搬示例細節 | 用不同的示例內容，只保留格式相同 |

## 可複製模板

### 通用 Few-shot 模板

```markdown
任務：[任務描述]

示例1：
輸入：[示例輸入1]
輸出：
[示例輸出1]

示例2：
輸入：[示例輸入2]
輸出：
[示例輸出2]

現在請處理：
輸入：[實際輸入]
```

### 分類任務模板

```markdown
任務：將用戶反饋分類爲 [類別列表]

示例：
反饋："[示例反饋1]"
分類：[類別]

反饋："[示例反饋2]"
分類：[類別]

反饋："[示例反饋3]"
分類：[類別]

請分類：
反饋："[待分類內容]"
```

### 轉換任務模板

```markdown
任務：將 [源格式] 轉換爲 [目標格式]

轉換規則示例：

源：[示例源內容]
目標：
[示例目標內容]

源：[示例源內容]
目標：
[示例目標內容]

請轉換：
源：[實際內容]
```

## 進階預告：動態 Few-shot

在進階版中，你會學到「動態 Few-shot」——根據用戶輸入**自動選擇最相關的示例**，進一步提升效果並節省 token。

## 本節要點

✓ **Few-shot 的本質**：用示例教會 AI 模式，而非用文字解釋規則

✓ **示例數量**：2-3 個是最佳平衡點

✓ **示例選擇三原則**：代表性、多樣性、格式一致性

✓ **適用場景**：格式化輸出、分類、轉換、風格模仿

✓ **與 Zero-shot 的選擇**：如果 Zero-shot 格式不穩定，換用 Few-shot
