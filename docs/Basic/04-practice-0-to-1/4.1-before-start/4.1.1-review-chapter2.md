# 4.1.1 回顧第二章成果：小李的待辦清單

> **本小節目標**：用 5 分鐘把第二章的思考成果"打包"成可以交給 AI 的任務

## 從"思考"到"執行"

還記得第二章的小李嗎？他是一個職場新人，經常忘事，想做一個待辦清單來管理自己的任務。

在第二章，我們幫小李做了很多思考工作：用戶畫像、靈魂三問、MVP 定義、風險預判......

現在，這些思考要**真正變成代碼**了。

但在開始寫 Prompt 之前，我們需要把這些思考成果"打包"成一個簡潔的"開發任務卡"。

## 小李案例的一頁紙總結

如果你跟着第二章做了思考，現在把它整理成這樣的格式：

### 靈魂三問

| 問題 | 答案 |
|------|------|
| **用戶是誰** | 小李，職場新人，25歲，經常忘事 |
| **痛點在哪** | 腦子裏裝太多事，容易遺漏重要任務，事後懊悔 |
| **爲什麼選這個方案** | 簡單、快速、隨時能用，不需要學習成本 |

### MVP 功能範圍

| 優先級 | 功能 | 說明 | 本章實現 |
|--------|------|------|---------|
| **P0** | 添加任務 | 輸入任務名稱，點擊添加 | ✅ |
| **P0** | 顯示列表 | 看到所有待辦任務 | ✅ |
| **P0** | 標記完成 | 點擊任務，標記爲已完成 | ✅ |
| **P0** | 刪除任務 | 刪除不需要的任務 | ✅ |
| **P1** | 數據保存 | 刷新頁面後數據還在 | ✅ |
| P2 | 分類標籤 | 按類型給任務分組 | ❌ 留給你擴展 |
| P2 | 截止日期 | 設置任務的截止時間 | ❌ 留給你擴展 |
| P2 | 優先級 | 區分任務的緊急程度 | ❌ 留給你擴展 |

### 不做清單

第一版明確**不做**的事情：
- ❌ 多用戶/賬號系統
- ❌ 數據同步/雲存儲
- ❌ 移動端專屬適配
- ❌ 提醒/通知功能

## 爲什麼現在不需要完美的 PRD

你可能會想："第二章不是教我們寫 PRD 嗎？我是不是應該把 PRD 寫得更完整？"

不需要。

**第一版的目標是驗證想法，不是做出完美產品。**

以小李的待辦清單爲例：
- 核心假設是"一個簡單的待辦清單能幫小李減少遺忘"
- 驗證這個假設，只需要：能添加、能看到、能完成、能刪除
- 其他功能都是在假設驗證成功後才需要考慮的

所以，你現在手裏的"靈魂三問 + MVP 功能列表"就夠用了。

::: info 💡 提示
如果你在第二章寫了更詳細的 PRD，那更好。但如果你只有上面這些內容，完全足夠開始了。
:::

## 把思考成果轉化爲"開發任務卡"

現在，把你的思考成果整理成這張"項目啓動卡"：

```markdown
# 項目啓動卡

## 項目名稱
個人待辦清單

## 一句話描述
幫助小李管理日常任務，減少遺忘的簡單網頁工具

## 目標用戶
職場新人，經常忘事，需要一個隨時能用的任務管理工具

## 第一版核心功能（按優先級）
1. 添加任務（輸入任務名稱，點擊添加按鈕）
2. 顯示任務列表（看到所有待辦任務）
3. 標記完成（點擊任務，標記爲已完成，視覺上有區分）
4. 刪除任務（刪除不需要的任務）
5. 數據保存（刷新頁面後數據還在）

## 明確不做
- 賬號系統
- 雲同步
- 提醒通知
- 移動端專屬適配

## 技術約束
- 純前端實現（HTML + CSS + JavaScript）
- 使用瀏覽器本地存儲（localStorage）保存數據
- 不需要後端服務器
```

這張卡片包含了你需要告訴 AI 的所有核心信息。在 4.2 節，我們會用第三章學的 S.C.A.F.F. 框架，把這些信息組織成 Prompt。

## 如果你有自己的項目想法

到目前爲止，我們一直在用小李的待辦清單作爲案例。如果你想做自己的項目，完全可以。

**但建議第一次先跟着小李案例走完整個流程**，原因是：

1. 待辦清單涵蓋了最常用的操作（增刪改查）
2. 遇到問題時，你可以對照教程排查
3. 完成後你會獲得完整的經驗，做自己的項目時更有信心

做完小李的項目後，你可以立刻用同樣的方法做自己的項目。

## 本小節要點

✓ **你已經有了"需求文檔"**：第二章的思考成果就是你的需求文檔，現在只是把它整理成可執行的格式。

✓ **不需要完美**：靈魂三問 + MVP 功能列表就夠開始了。

✓ **項目啓動卡是你的指南針**：接下來的開發過程中，隨時可以回來看這張卡，確認自己在做對的事情。
