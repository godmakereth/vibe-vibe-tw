# 4.5.4 Debug 心法總結

經過本節學習，你將掌握：
- 面對報錯的正確心態
- 高效 debug 的五條心法
- 一份可複用的 Debug 檢查清單

## 心法一：別慌，報錯是正常的

報錯不是失敗的信號，而是代碼在告訴你「這裏需要調整」。

| 錯誤認知 | 正確認知 |
|---------|---------|
| 報錯說明我做錯了 | 報錯是調試過程的一部分 |
| 專業程序員不會遇到報錯 | 專業程序員每天都在 debug |
| 代碼報錯了，我的項目完蛋了 | 90% 的報錯都能在幾分鐘內解決 |

**記住**：每一次成功的 debug，都是你能力的提升。專業程序員和初學者的區別，不是「會不會遇到報錯」，而是「遇到報錯時知不知道怎麼處理」。

## 心法二：先讀後改，理解報錯信息

在動手修改之前，先讀懂報錯信息在說什麼。

**報錯信息的三個關鍵部分**：

```
Uncaught TypeError: Cannot read properties of undefined (reading 'push')
        ↑ 錯誤類型           ↑ 錯誤描述
    at addTask (index.html:45:15)
                    ↑ 錯誤位置（文件名:行號:列號）
```

**讀懂報錯的三個問題**：

1. **是什麼類型的錯誤？** TypeError / ReferenceError / SyntaxError
2. **具體出了什麼問題？** 讀錯誤描述，用自己的話複述
3. **錯誤發生在哪裏？** 根據行號定位代碼

**避免**：看到報錯就慌，不讀信息就開始改代碼。這往往會越改越亂。

## 心法三：隔離問題，一次只改一處

當代碼有多個問題時，要逐個擊破，而不是一次性全改。

**正確做法**：

```
問題 1：點擊添加沒反應
    ↓ 解決
問題 2：刷新後數據丟失
    ↓ 解決
問題 3：樣式顯示不對
    ↓ 解決
```

**錯誤做法**：

```
同時告訴 AI："添加不行，刷新丟數據，樣式也不對，幫我全部修好"
    ↓
AI 返回大量改動
    ↓
不確定哪些改動解決了哪些問題
    ↓
新問題出現，不知道是哪個改動導致的
```

**隔離問題的好處**：
- 每次改動都能確認效果
- 出現新問題時容易定位原因
- 避免「修一個問題引入三個新問題」

## 心法四：善用撤銷，改錯了就退回

改代碼就像走迷宮，走錯了要能退回來重新選路。

**實用技巧**：

| 場景 | 操作 |
|------|------|
| 剛改錯了代碼 | `Ctrl+Z` / `Cmd+Z` 撤銷 |
| 改了很多想全部放棄 | 不保存，直接刷新頁面 |
| 想保留多個版本 | 每完成一個功能就備份一次文件 |

**備份策略**：

```
todo-v1.html  ← 完成靜態頁面
todo-v2.html  ← 完成添加功能
todo-v3.html  ← 完成刪除功能
todo-v4.html  ← 完成 localStorage
```

如果 v4 改崩了，隨時可以回到 v3 重來。

## 心法五：知道何時換思路

有時候，與其死磕一個問題，不如換個方法。

**判斷標準**：

| 情況 | 建議 |
|------|------|
| 同一個問題，AI 反覆給出類似的錯誤方案 | 換一個 Prompt 表述方式 |
| 改動越來越多，但問題沒解決 | 回退到上一個能用的版本 |
| 代碼已經「縫縫補補」面目全非 | 讓 AI 重新生成這部分功能 |
| 花了 30 分鐘還沒解決 | 休息一下，換個思路或尋求幫助 |

**「重來」不是失敗**：

有時候從頭開始比修補更快。你已經知道要實現什麼、踩過哪些坑，第二次寫會比第一次順利得多。

## Debug 檢查清單

遇到問題時，按這個清單逐項檢查：

### 初步檢查

- [ ] 打開控制檯，查看是否有紅色報錯
- [ ] 閱讀報錯信息，識別錯誤類型
- [ ] 根據行號定位到具體代碼位置

### 信息收集

- [ ] 記錄錯誤現象（發生了什麼）
- [ ] 記錄期望行爲（應該發生什麼）
- [ ] 複製完整的報錯信息
- [ ] 定位相關代碼（報錯行附近 10-20 行）

### 問題解決

- [ ] 先診斷原因，再請求修復
- [ ] 一次只解決一個問題
- [ ] 替換代碼前做好備份
- [ ] 修改後立即驗證效果

### 驗證確認

- [ ] 控制檯沒有新的報錯
- [ ] 功能按預期工作
- [ ] 沒有影響其他功能
- [ ] 刷新頁面後仍然正常

## 本節要點

- 報錯是正常的，專業程序員每天都在 debug
- 先讀懂報錯信息，再動手修改
- 一次只解決一個問題，避免越改越亂
- 善用備份和撤銷，給自己留後路
- 知道何時該換思路或重新開始

恭喜你完成了 Debug 實戰章節。現在你不僅知道如何開發功能，也知道如何解決開發中遇到的問題。

這正是 Vibe Coding 的核心技能：**讓 AI 幫你寫代碼，也讓 AI 幫你修代碼**。
