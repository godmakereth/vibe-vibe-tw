# 5.1.6 版本管理最佳實踐

學會了工具的基本操作，接下來是一些讓版本管理更有效的技巧。

## Commit 信息怎麼寫

每次 Commit 都需要寫一句描述。好的描述能讓你快速找到需要的版本。

### 好的寫法

```
✅ "添加任務刪除功能"
✅ "修復刷新後數據丟失的問題"
✅ "調整按鈕顏色爲藍色"
✅ "第四章完成版"
✅ "準備添加深色模式前的備份"
```

特點：
- 說清楚**做了什麼**
- 一眼就能看懂

### 不好的寫法

```
❌ "修改了一些東西"
❌ "update"
❌ "fix"
❌ "..."
❌ "1"
```

問題：
- 看不出做了什麼
- 以後找版本時完全沒幫助

### 簡單原則

**用一句話描述"這次改動做了什麼"**。

不需要寫得很正式，只要自己以後能看懂就行。

## 多久 Commit 一次

**原則：完成一個小功能就 Commit**。

| 做法 | 優缺點 |
|------|-------|
| 每改一行就 Commit | ❌ 太頻繁，歷史太亂 |
| 做完一個功能就 Commit | ✅ 推薦，粒度合適 |
| 寫了一大堆再 Commit | ❌ 出問題難回退，找不到問題點 |

### 推薦的 Commit 時機

- 添加了一個新功能
- 修復了一個問題
- 調整了樣式/佈局
- 重構了代碼結構
- 準備下班/休息

不要等到"完美"了再 Commit。半成品也可以 Commit，反正隨時可以繼續改。

## 什麼文件應該/不應該提交

有些文件應該納入版本管理，有些不應該。

### 應該提交的文件

- 你寫的代碼（HTML、CSS、JavaScript 等）
- 項目配置文件
- 文檔和說明

### 不應該提交的文件

| 類型 | 例子 | 爲什麼 |
|------|------|-------|
| 依賴包文件夾 | `node_modules/` | 太大，可以重新安裝 |
| 環境配置 | `.env` | 可能包含密碼等敏感信息 |
| 系統文件 | `.DS_Store`（Mac） | 與項目無關 |
| 編譯產物 | `dist/`、`build/` | 可以重新生成 |

### 使用 .gitignore 文件

在項目根目錄創建一個名爲 `.gitignore` 的文件，寫入要忽略的文件/文件夾：

```
# 依賴
node_modules/

# 環境配置
.env
.env.local

# 系統文件
.DS_Store
Thumbs.db

# 編譯產物
dist/
build/
```

GitHub Desktop 會自動忽略這些文件，不會把它們納入版本管理。

::: tip 提示
如果你用的是純 HTML/CSS/JS 的簡單項目（比如待辦清單），暫時不需要擔心 `.gitignore`。這個文件在項目變複雜後纔會用到。
:::

## 三級方案對比與選擇

| 方案 | 學習成本 | 可靠性 | 雲端備份 | 適合場景 |
|------|---------|-------|---------|---------|
| AI IDE 自帶歷史 | 零 | 中 | 否（網頁版除外） | 日常小改動 |
| 手動複製文件夾 | 零 | 高 | 否 | 重要節點備份 |
| GitHub Desktop | 低 | 高 | 是 | 長期項目、想學專業技能 |

### 推薦組合

**最省心的方案**：
1. 日常依賴 AI IDE 自帶歷史
2. 完成重要功能時手動備份一個文件夾
3. 項目完成後考慮上傳到 GitHub

**想要專業一點**：
1. 從一開始就用 GitHub Desktop
2. 每完成一個功能就 Commit
3. 每天結束時 Push 到雲端
