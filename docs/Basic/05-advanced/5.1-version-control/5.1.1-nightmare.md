# 5.1.1 一個真實的噩夢場景

## 這種經歷，每個學習者都會遇到

想象一下這個場景：

1. 你花了 2 小時，終於做好了待辦清單
2. 頁面很漂亮，功能也正常，你很有成就感
3. 想再加個"深色模式"功能，讓它更炫酷
4. 改着改着，頁面突然白屏了
5. 你不記得改了哪裏，Ctrl+Z 也按不回去
6. 2 小時的成果，就這樣沒了...

這不是誇張。在社區裏，幾乎每週都能看到類似的求助帖："我的代碼改崩了，怎麼恢復？"

## 更常見的幾種情況

| 場景 | 發生了什麼 | 後果 |
|------|-----------|------|
| 改一個功能，壞了另一個 | 添加刪除功能時，不小心影響了添加功能 | 花時間排查，可能越改越亂 |
| 想回到"昨天那個版本" | 昨天的代碼能用，今天改了一堆，全壞了 | 找不到昨天的版本 |
| 試了三種方案，想回到第一種 | 第一種方案其實最好，但已經被覆蓋了 | 要麼重寫，要麼將就 |
| AI 改了不該改的地方 | 讓 AI 改一個文件，它順手改了其他文件 | 不知道哪些被改了 |

## 爲什麼會發生這種事

Vibe Coding 的工作方式是：你告訴 AI 想要什麼，AI 直接修改代碼。

這個過程非常高效，但有一個風險：**改動是即時的、覆蓋式的**。

傳統程序員有一套複雜的工具來管理代碼歷史，但對於剛入門的你，這些工具學習成本太高。

好消息是：你不需要學那些複雜的東西。

## 解決方案：給代碼裝上"時光機"

**版本管理**，本質上就是給你的代碼創建"存檔點"。

就像玩遊戲時的存檔功能：
- 打 Boss 之前存一個檔
- 打輸了可以讀檔重來
- 試新技能之前存一個檔
- 不好用可以回退

代碼也是一樣：
- 功能做好了存一個檔
- 準備大改之前存一個檔
- 改崩了可以回退

::: tip 核心觀點
版本管理不是高級技能，是基本生存技能。

你不需要成爲 Git 專家。只需要知道：**怎麼存檔、怎麼讀檔**，就夠了。
:::

## 接下來學什麼

我們會介紹三種方案，從簡單到進階：

1. **AI IDE 自帶的歷史功能**：最簡單，立即可用
2. **手動複製文件夾**：最直觀，人人都會
3. **GitHub Desktop**：更專業，爲進階版打基礎

先從最簡單的開始。
