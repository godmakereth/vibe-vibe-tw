# 5.3.2 絕對禁區：永遠不要告訴 AI 的東西

經過本節學習，你將掌握：
- 哪些信息絕對不能發送給 AI
- 爲什麼這些信息會帶來風險
- 如果已經發送了該怎麼補救
- 如何安全地讓 AI 幫你處理敏感場景

## 絕對禁令清單

**永遠不要**在與 AI 對話時發送以下內容：

| 類型 | 示例 | 爲什麼危險 |
|------|------|-----------|
| API 密鑰 | `sk-abc123...`、`AKIA...` | 可被用於消費你的賬戶，或訪問你的雲服務 |
| 數據庫連接字符串 | `postgres://user:pass@host/db` | 攻擊者可直接訪問你的數據庫 |
| 密碼 | 任何真實密碼 | 身份被盜用，賬戶被入侵 |
| 私鑰/證書 | `-----BEGIN RSA PRIVATE KEY-----` | 可用於僞裝成你的身份，入侵你的系統 |
| 個人身份信息 | 身份證號、銀行卡號、社保號 | 隱私泄露，可能被用於詐騙 |
| 公司/學校機密 | 內部文檔、客戶數據、商業祕密 | 數據泄露，可能違反保密協議 |
| Token/Session | JWT token、登錄憑證 | 可用於冒充你的登錄狀態 |

## 爲什麼 AI 對話不安全

你可能會想：我用的是正規的 AI 服務，應該沒問題吧？

事實上，風險來自多個方面：

1. **訓練數據**：部分 AI 服務可能將對話用於模型訓練，你的信息可能被"記住"
2. **日誌記錄**：對話內容可能被記錄在服務器上
3. **數據泄露**：即使是大公司也可能發生數據泄露事件
4. **員工訪問**：某些情況下，服務提供商的員工可能有權限查看對話

::: danger 核心原則
**假設你發送給 AI 的任何內容都可能被公開。**

如果你不願意把某條信息貼在公告欄上，就不要發送給 AI。
:::

## 常見危險場景

### 場景一：讓 AI 檢查密鑰

```
❌ 錯誤做法：
"幫我檢查這個 OpenAI API Key 格式對不對：sk-proj-abc123xyz..."

✅ 正確做法：
"OpenAI API Key 的格式是什麼樣的？"
```

### 場景二：讓 AI 幫忙調試連接問題

```
❌ 錯誤做法：
"我的數據庫連接失敗了，連接字符串是 postgres://admin:MyP@ssw0rd@db.example.com/mydb"

✅ 正確做法：
"我的 PostgreSQL 連接失敗了，錯誤信息是 'connection refused'，可能是什麼原因？"
```

### 場景三：讓 AI 幫忙寫配置文件

```
❌ 錯誤做法：
"幫我寫一個 .env 文件，我的 Stripe 密鑰是 sk_live_xxx..."

✅ 正確做法：
"幫我寫一個 .env 文件的模板，包含 Stripe 配置項，用佔位符表示"
```

## 如果已經發送了怎麼辦

如果你不小心發送了敏感信息，立即採取以下步驟：

| 步驟 | 行動 | 說明 |
|------|------|------|
| 1 | **立即更換** | 更換泄露的密鑰/密碼，不要等到"有空再說" |
| 2 | **檢查使用記錄** | 查看相關服務的使用日誌，是否有異常活動 |
| 3 | **設置告警** | 如果服務支持，開啓異常使用告警 |
| 4 | **清除對話** | 如果 AI 服務支持，刪除包含敏感信息的對話 |
| 5 | **記錄教訓** | 下次更加小心 |

## 安全替代方案

那麼，如何讓 AI 幫忙處理涉及敏感信息的場景呢？

**使用佔位符**：

```javascript
// 讓 AI 幫你寫這樣的代碼：
const apiKey = process.env.API_KEY;  // 從環境變量讀取
const dbUrl = process.env.DATABASE_URL;  // 不要硬編碼

// 而不是：
const apiKey = "sk-真實密鑰";  // 絕對不要這樣
```

**只描述問題，不暴露數據**：

```
✅ "我的 API 調用返回 401 錯誤，可能是什麼原因？"
✅ "如何驗證一個 JWT token 是否過期？"
✅ "數據庫連接超時通常怎麼排查？"
```

::: tip 記住
AI 不需要看到你的真實密鑰就能幫你解決問題。

就像你問醫生"頭疼怎麼辦"，不需要把你的病歷號告訴 AI。
:::
