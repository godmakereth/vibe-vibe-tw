# 5.3.1 爲什麼這一節很重要

## 真實案例：10 分鐘損失數千美元

這是 2024-2025 年間反覆發生的場景：

1. 開發者用 AI 生成代碼
2. AI 把 API 密鑰直接寫進了代碼裏
3. 代碼被上傳到 GitHub（即使是私有倉庫也不安全）
4. **幾分鐘後，賬戶被盜刷**

2024 年 11 月，區塊鏈項目 Pump Science 就因爲 GitHub 密鑰泄露，導致攻擊者創建了欺詐性代幣，造成用戶損失。

## 大規模憑證盜竊事件

2024 年 10 月，安全研究人員發現了代號爲 **EMERALDWHALE** 的攻擊行動：

- 攻擊者掃描了約 **5 億個 IP 地址**
- 從暴露的 Git 配置文件中竊取了超過 **15,000 個雲服務憑證**
- 涉及 **10,000+ 個私有代碼倉庫**
- 被盜憑證被用於釣魚攻擊和垃圾郵件

這些憑證包括 AWS 密鑰、數據庫連接字符串、API Token——都是開發者"不小心"提交到代碼庫裏的。

## 爲什麼 AI 編程時代風險更高

還記得第三章提到的 AI 幻覺嗎？在安全領域，AI 的"不靠譜"可能直接導致：

| AI 行爲 | 安全後果 |
|---------|---------|
| 把密鑰當作"示例"寫進代碼 | 密鑰泄露 |
| 推薦不存在的軟件包 | 可能安裝惡意代碼 |
| 跳過輸入驗證 | 留下安全漏洞 |
| 使用過時的加密方式 | 數據容易被破解 |

根據 Veracode 2025 報告，當 AI 在"安全寫法"和"不安全寫法"之間做選擇時，**45% 的情況下會選擇不安全的方式**。
